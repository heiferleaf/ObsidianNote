# 第五章 运输层

> 本章主要介绍运输层协议的特点、进程之间的通信和端口等概念，UDP、TCP。
> 主要部分在于TCP协议和可靠传输的工作原理，包括停止等待协议、ARQ协议


## 5.1 运输层协议概述

### 5.1.1 进程之间的通信

**运输层向上面的应用层提供通信服务**，他是面向通信的最高层，也是用户功能的最底层。

既然我们现在知道网络层可以交换分组，转发数据，那为什么还需要运输层呢？
- 答案很简单，因为网络层的地址表标识IP是主机的IP地址，而在实际的通信过程，都是主机进程之间的通信，比如qq和qq。
而这种不同主机的进程之间的通信，**就叫做端到端的通信**，我们之前说的点对点的通信，更多是指两个主机之间的通信。
进而，我们发现运输层需要的一个重要功能--复用和分用
	复用 ： 发送方的应用进程可以使用同一个运输层协议来传输数据（当然还要首部）
	分用 ： 接受方可以从运输层中找到应该交付给哪一个进程的信息。
当然了，**运输层还需要为收到的报文进行差错检测**，因为网络层的首部校验字段不会校验数据部分。

既然是为应用程序服务的，就要看应用程序的需要来设计功能，运输层有两种不同协议，**面向连接的 TCP 和 面向无连接的 UDP**
运输层为下层的通信进行了封装，所以看上去进程和进程之间有一条端到端的逻辑通信信道。当使用TCP协议的时候，尽管下层网络不可靠，这条信道还是**全双工的可靠信道**，当使用 UDP，这条信道还是**不可靠的**

### 5.1.2 运输层的两个主要协议

1. 用户数据报协议 UDP
	不需要事前建立连接，实现简单
2. 传输控制协议 TCP
	面向连接的服务，提供可靠运输服务，需要确认、流量控制、计时器(持续计时器-流量控制探测零窗口，保活计时器-TCP连接建立后，服务器在客服很长没有消息后计时，关闭TCP连接时，客户进入CLOSED前的2MSL的计时器)等功能
![[一些应用程序使用的运输层协议.png]]

### 5.1.3 运输层的端口

前面提到复用分用功能，这就需要一个标识符作为分用的准则。运输层使用端口作为提供给应用层的SAT（服务访问点），每一个端口有一个端口号。
	1. 端口号16bit ： 0~65535
	2. 端口号还可以分类
		1. 熟知端口号：0~1023，是所有计算机进程中都一样的端口号
			![[常用的熟知端口号.png]]
		2. 登记端口号：1024~49151，这一类端口需要在 IANA（一个服务器进程）中进行登记，防止重复
		3. 客户端使用的端口号：49512~65535，这类端口号只会在客户进程动态分配，通信结束之后就收回了


## 5.2 用户数据报协议UDP

### 5.2.1 UDP概述

UDP需要在IP数据包的服务之上实现复用分用、差错检测，其特点为：
	1. 无连接，所以没有建立连接和释放连接的延迟
	2. 最大努力
	3. 面向报文：对于应用程序传输下来的报文，不做分割，加上首部信息之后就直接交给IP数据报，每一次传输完整的报文
	4. 没有拥塞控制，网络阻塞不会让主机的发送速率降低
	5. UDP支持一对一，一对多、多对一、多对多的交互通信
	6. UDP首部开销小：只有8字节

## 5.3 传输控制协议 TCP 概述

### 5.3.1 TCP的主要特点

1. 面向连接的运输层协议
2. 每一条TCP连接只能有两个端点，点对点的连接
3. 可靠交付
4. 全双工的通信，允许通信双方什么时候都能发数据
5. 面向字节流

### 5.3.2 TCP的连接

TCP 把连接作为最基本的抽象，每一条 TCP 连接有两个端点，把端点叫做嵌套字或者插口$$嵌套字 socket=\{IP地址：端口号\}$$

## 5.4 可靠传输的工作原理

理想的传输应该是
	1. 传输信道不出错
	2. 发送方不管以多快的速度发数据，接受方总是能来得及处理
这就要求TCP协议，对于传输出错的数据能要求重传，对于接受数据处理不过来的时候，可以让发送方发慢点

### 5.4.1 停止-等待协议

1. 无差错情况
	A发送数据后暂停等待，B接受数据后发送确认，A收到确认后发送下一个数据，循环
2. 出现差错
	A保持一个超时计时器，防止确认信息丢失，超时重传
	这个计时器的长短很讲究，短了会有很多无必要的重传，长了效率会很低
3. 确认丢失和确认迟到
	在确认丢失之后，A重传，B收到了重复的数据，这是，B需要
		1. 丢弃重复数据
		2. 向A发送确认
	确认没丢失，但是迟到了，A又发数据，B又确认，确认重复了，这时A
		丢弃确认，啥都不做

这种可靠协议也可以叫做 ARQ 自动重传请求

### 5.4.2 连续 ARQ 协议

就是后续介绍的 GBN 方式，是滑动窗口的一种实现机制，还可以选择重传

## 5.5 TCP 报文段的首部格式

TCP 是面向字节流的，也就是说应用层发送的数据在TCP看来就是一串字节，需要保证发送和接受的字节流是一样的，不过TCP本身是以数据块为单位发送的，发送的数据块和接受到的数据块不一定要对应。

可以通过TCP的首部信息了解TCP的功能：
	TCP首部有长度20的固定字节，后面可以增加的选项是4n字节，首部最大60字节
	![[TCP报文段的首部格式.png]]
> 序号指的是本报文段传输的第一个字节的字节序号
> 确认号是期望对方下一个报文段的第一个数据字节的序号
> $\boxed{若确认号=N，则表明：到序号N-1为止的所有数据都已经正确收到}$
> $\boxed{窗口字段指出了现在允许对方发送的数据量，单位是字节}$
> 校验和 和 UDP一样，需要考虑12字节伪首部，内容格式也基本一样，只要把第四个字段值改为6，对应TCP协议号，并且把UDP长度改为TCP长度

> 拓展选项：窗口扩大、事件戳选项、选择确认 等

1. 窗口扩大：3字节，有一个字节标识移位值 S,作用是让窗口大小左移S，从而变大，最大移位量是14，对应最大窗口$2^{(14+16)} -1$
2. 时间戳选项：10字节，包括时间戳值字段和时间戳回送回答字段，发送的时候，把当前时间写到时间戳值，接受方发送确认时把值复制到时间戳回送回答字段，发送方接收确认的时候，根据这个值就能算出RTT

> [!NOTE] 防止序号绕回作用
> 时间戳选项还有防止序号绕回的作用，在快速链路的网络中，32位的序号很快就会开启新的一轮，这时候事件戳可以表明一个报文是新一轮的还是老一轮的
3. 选择确认：首先固定需要两个字节，一个标识是SACK，一个字节标识有几个块的边界信息。一个块有两个边界信息，一个边界信息32位-8字节，而首部选项长度最大40字节，(40 - 2) / 8 = 4...，所以最多放4个块的确认信息
>块的边界信息，左边界是块的第一个字节序号，右边界是块的最后一个字节序号+1

## 5.6 TCP可靠传输的实现

大体和之前的滑动窗口机制都是一样的，谈论几点：
1. 我们之前说的ACK，在这里就是TCP的报文，通过确认号来进行窗口的推进
2. TCP是全双工的点对点信道，每一个主机既有发送窗口，又有接受窗口
3. 主机需要有发送缓存和接受缓存
	1. 发送缓存：（1）存放应用程序要让发送方发送的数据；（2）存放发送窗口中还没被确认的数据
	2. 接收缓存：（1）存放接收窗口中没有按序到达的数据；（2）存放按序到达，但是没有被应用程序读取的数据
4. 发送窗口一般根据通信对方的接受窗口大小设置自己的大小，但并不要求一样大，可以根据网络拥塞状况变化
5. 对于接受方来说，累积确认可以减小发送确认的次数，但是过长的累积会让接受方超时
6. 对于乱序到达的数据，可以丢弃，也可以缓存

### 5.6.2 超时重传时间的选择

![[QQ_1762162205197.png]]
很显然，超时重传时间 RTO 需要略大于$RTT_S$,一般计算方式为$$RTO = RTT_S + 4 \times RTT_D$$![[QQ_1762162354565.png]]
$RTT_D的\beta一般取0.25$

*以下这小段是在没有事件戳这个字段的时候，采用的方法，不过现在也依然采用*
这样子还存在问题：如果一个报文段发送之后，超时重发，之后接收到确认，无法辨认是对第一次发送的确认还是对重传的确认，如果辨认错误，会导致RTO错误变大或者变小。
	**解决方案**：karn算法，只要报文段重传，就不更新 $RTT_S,RTO$
	但是这样还有问题：当网络不好，延迟确实增大了，导致重传，但是算法又不更新RTO，那就一直重传
		**解决方案**:每一次发生重传，把 RTO 增大一些，一般是增大两倍


## 5.7 TCP 的流量控制

### 5.7.1 利用滑动窗口实现流量控制

在TCP报文中，有一个字段是窗口大小，它可以表明接受方目前的窗口大小，在ACK报文中携带这个信息给发送方，从而控制发送发的速度，进而实现了流量控制。

当接受方窗口大小为0的时候，发送方就不能在发送数据了，过了一段时间，接受方缓存数据交给了CPU处理，窗口大小又不为0了，于是发送方通知这个信息，但是通知一旦丢失，就出现发送方以为接受方接受不了，接受方等待发送方发送的死锁现象。
为了解决这个问题，TCP 为每一对连接设置一个**持续计时器**，连接的一方收到对方的零窗口通知，会启动持续计时器，计时器到期，发送一个零窗口探测报文段（1字节的数据），对方收到后给出窗口值，为0重新计时。通过这种被动通知改为主动探测的方式，解决问题。

### 5.7.2 TCP 的传输效率

进程把发送数据传送到发送缓存中，接下来的发送由TCP控制，而发送的实际比较讲究
	1. 当缓存数据量达到 MSS 字节时发送（MSS是TCP维护的一个变量，等于最大报文段长度）
	2. 当进程指明要求发送数据，执行push操作发送
	3. 设置一个计时器，到期了发送（但还是不能超过MSS）

上述的三种都是示例，实际中广泛使用的是Nagle算法：
	1. 发送进程发送数据到缓存时，先把第一个数据字节发送出去，后续的字节缓存，当收到确认报文后，把目前缓存到的字节发送。重复这个收到确认后发送的步骤
	2. 同时，当到达数据到达窗口大小的一般或者MSS，就马上发送一个报文段

而对于接方来说，存在**糊涂窗口综合征**，简单说就是腾出1字节的窗口大小就让发送方继续发，这样传输效率很低
解决方法有：接受方等待一段时间，等到空闲大小有MSS或者缓存一半，这时候在通知发送方

## 5.8 TCP 的拥塞控制

### 5.8.1 概念介绍

[[读书笔记4#^dfaec9|在路由的结构那一章节我们了解到线速的概念]]，当路由器的处理速度小于发送到输入端口的速度，分组会丢失。这其实就是一种拥塞现象。
>[!NOTE] 拥塞现象
>当网络中对某一资源的需求超过资源能提供部分，网络性能会变坏$$\sum{对资源需求} > 可用资源$$
拥塞控制是一个**全局性的问题**，防止过多数据注入到网络，导致结点或者链路过载
而流量控制是点对点的通信量的控制，**是一个端到端的问题**

![[拥塞控制起的作用.png]]

### 5.8.2 TCP 拥塞控制的方法

> 为了方便讨论，假定
> 1. 数据单方向传输，对方只传送确认报文
> 2. 接受方总是有足够大的缓存空间，因此发送窗口大小只由网络拥塞程度决定

1. **慢开始和拥塞避免**
2. **快重传和快恢复**
	当无法接收到ACK，发送方超时重传时，认为拥塞了，一般不考虑因比特差错而丢弃分组，现在的网络质量一般很好
	![[TCP拥塞控制的流程图.png]]

在讨论的开始，我们规定接受方缓存大小足够大，但是实际中肯定不行，所以还需要考虑接受方的窗口大小，综合后$$发送方窗口上限值=Min[rwnd,cwnd]$$
### 5.8.3 主动队列管理AQM

TCP 的拥塞管理其实和网络层还有很密切的联系。在路由器对分组的处理中，如果分组比较长，处理时间长，会导致发送超时，拥塞窗口降为1，而当输入队列满，会导致很多TCP连接的分组丢失重传，很多的TCP的拥塞窗口同时变小，然后又一起指数增长。这种情况都是不利的。

AQM技术其实就是对路由器中的分组进行管理，不需要等到队列满才丢弃分组，而是在早期的时候，按照某种策略，对分组进行管理。

## 5.9 TCP 的运输连接管理

现在，我们都知道 TCP 是面向连接的协议，运输的连接是服务传送 TCP 报文的，所以连接的建立和释放就很重要。可以把TCP的通信分为三个过程：**连接建立、数据传输、连接释放**

### 5.9.1 TCP 的连接建立

TCP 的连接建立需要解决三个问题
	1. 要让双方知道对方的存在
	2. 双方要协议一些参数（比如窗口最大值，是否使用窗口扩大选项和时间戳选项等）
	3. 对运输的资源进行分配（比如缓存大小）
而建立的方式是**客户服务器方式**，发起连接的是客户，被动等待连接建立的是服务器。

通常，我们把 TCP 建立连接的过程叫做握手，握手需要在客户和服务器之间交换三个 TCP 报文。![[TCP 建立连接的三次握手.png]]
>[!NOTE] 客户和服务器连接建立的阶段
>对于客户：
>	在CLOSED状态准备一个TCB传输控制块，发送同步请求报文，进入SYNSENT状态，当收到服务器发送的确认报文后，自己发送一个确认报文，进入ESTABLISHED状态
>对于服务器：
>	在CLOSED状态准备一个TCB，进入LISTEN状态，收到客户的同步请求报文，发送一个确认报文后，进入SYNRCVD状态，在收到A的连接确认后，进入ESTABLISHED状态
>其中SYN报文规定要消耗1个序号，并且不能传送数据，所以能看到客户两次发的序号，是从x到x+1
>其中ACK报文规定不携带数据的时候不消耗序号，在考虑SYN消耗的一个序号，所以A的连接确认报文的ack=y+1
>其实B对于同步报文的确认消息，可以分成两个报文发送，先发ACK，再发SYN，这样就是4次握手，不过效果一样
>A最后发送确认连接的报文，是为了防止已经失效的连接请求突然到达了B

### 5.9.2 TCP 的连接释放

连接的释放通过**四次握手**
![[TCP的连接释放.png]]
>[!NOTE] 连接释放阶段
>对于FIN=1的报文，即时不携带数据，也要消耗一个序号
>对于MSL的长度，RFC的建议是2分钟，所以2MSL就是4分钟
>设置2MSL的目的：（1）如果最后向B发送的ACK丢失，B会重传FIN，A重新ACK，保证B能CLOSED（2）保证本次连接内产生的报文都在网络中消失，不会出现连接建立时候那种已经失效的报文
>通过2MSL的设置，B会先进入CLOSED，而除此之外，还有一个计时器叫做保活计时器，保证客户端还活着，当客户长时间不传消息，服务器进行探测，如果多次连续探测没有响应，服务器觉得客户挂掉了，就关闭连接

# 补充

## TCP滑动窗口与链路层滑动窗口的对比

### 1. 接收缓存机制
```plaintext
链路层：
- GBN接收窗口=1，不缓存失序帧，直接丢弃
- SR支持固定大小接收窗口，可缓存失序帧

TCP：
- 支持动态大小接收窗口(rwnd)
- 可缓存失序数据
- 缓存满可能导致死锁（因为占满接收窗口）
```

### 2. 重传策略
```plaintext
链路层(GBN)：
- 收到NAK或ACK超时，重传最大确认号之后的所有帧
- 接收方丢弃所有失序帧

TCP：
- 支持多种重传机制（超时重传、快速重传）
- 重传ACK号后的数据时，接收方可能已缓存部分数据
- 使用SACK时可以只重传丢失部分
```

### 3. 确认机制
```plaintext
链路层：
- 每收到一帧立即发送确认
- 独立确认每个帧

TCP：
- 支持累积确认，可延迟发送ACK
- 接收方维护最大连续确认序号
- 不能跳过序号空隙确认更大序号的数据
```

### 4. 选择确认（SR/SACK）
```plaintext
链路层SR：
- 接收方记录每个成功接收的帧
- 发送方单独重传NAK的帧

TCP SACK：
- 接收方告知已缓存的不连续数据段
- 发送方可以只重传丢失数据
- 仍然保持基本的累积确认机制
```
