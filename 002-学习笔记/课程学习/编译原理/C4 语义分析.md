> 词法分析关注于把程序的源代码作为输入，分析出怎么样的一个词是合法的，属于何种模式。 regex - NFA -DFA
> 语法分析关注把词法分析获取到的词序列，判断是否是合规的语法（任意数量排列组合） CFG
> 这两个步骤，都关注于的是输出 yes/no 的判断
> 语义分析，更多关注于程序的语义，比如变量的值、类型、等等，检查完这一步之后，才能执行中间代码的生成

# 语法制导定义 SDD

在 $CFG$ 的基础上，每一个文法符号有自己的**属性**，每一个产生式有属性的计算规则（**语义规则**）。在语义分析的语境中，把 $CFG$ 称为基础文法。综上三者：基础文法、属性、语义规则。放在一起叫做 $SDD$

$SDD$ 的属性，进一步分为
- **综合属性**：产生式左侧符号的属性，由产生式右侧符号属性或者自己的其他属性计算而来
- **继承属性**：产生式右侧符号的属性，由自己的兄弟结点的属性、产生式左侧符号的属性计算而来

先不考虑计算机如何完成属性的计算过程，对于人来说，如何确定属性的计算流程：**输入串---分析树---属性依赖图---属性计算次序**
> 分析树一般指带注释的语法分析树，在输入串构建的语法树上，给每一个结点绑定它的属性，属性A如果依赖输入B计算，就画一条B指向A的有向边，最后得到属性的依赖图

通过确定属性依赖图的拓扑排序，来确定有效的计算次序，把存在拓扑排序的$SDD$，称作**良定义的**

分析树的表示有时候过于繁琐，一个属性向上传递几层，才能被用作计算。可以通过语法树，只保留叶子属性结点的构造和多属性运算出来的结点。

除了这种通过分析树确定属性计算次序，还有两种方式
- 基于规则：静态分析的方法，设计编译器的时候，根据文法的特点，确定规则的计算顺序
- 忽视规则：$SDT$ 就是这种思想，不关心属性之间的依赖关系，通过在产生式中嵌入动作，从而在语法分析的同时完成语义分析。可能会因为依赖关系出错。


# $S-SDD \,\, L-SDD$

对于一个 $SDD$ ，无法确定所有的输入串构成的语法树，都有属性计算的拓扑排序。但是存在特殊的$SDD$，保证有拓扑排序。

$S-SDD$：S属性，SDD中只有综合属性
$L-SDD$：L属性，SDD中的属性，包含
- 综合属性
- 继承属性，限制要求：
	1. 依赖产生式左侧属性的继承属性
	2. 依赖属性所属符号左侧的符号的属性
	3. 所属符号的其他属性，保证不形成循环

# SDT

通过在 $SDD$ 的基础文法的产生式中，嵌入程序规则（**语义动作**），指定语义规则的执行时机：这种具体的实现语义分析的技术，叫做**语法制导的翻译**

通过 $SDT$ ，可以实现$LR$分析的 $S-SDD$:
- 将所有语义规则放到产生式的最后端
- 在移进规约算法中，符号栈多存储一列信息，符号的属性
- 当规约发生时，执行语义动作，算出产生式左侧符号的综合属性，在放入栈中

通过 $SDT$ ，可以分析$L-SDD$
- 相对产生式进行改造
	- 把计算某个非终结符号A的继承属性的动作插入到产生式中**紧靠在A的本次出现之前的位置**上。如果A的多个继承属性以无环的方式相互依赖，就需要对这些属性的求值动作进行排序，以便先计算需要的属性
	- 将计算一个产生式$LHS$的综合属性的动作放置在这个产生式体的最右端

## L属性的语义分析

### 核心示例说明
我们使用一个带有中间动作的 L-属性产生式作为通用模版：  
**A → B { C.i = f(B.s) } C { A.s = g(C.s) }**
- **{ C.i = f(B.s) }** 是一个嵌入在中间的动作，用于计算 C 的继承属性。
- **{ A.s = g(C.s) }** 是一个末尾动作，用于计算 A 的综合属性。

### 1. 递归的自上而下实现（Recursive Descent）

在这种方法中，产生式中的每一个位置（文法符号或动作）都按从左到右的顺序直接转化为代码。
- **实现机制：**
    1. **文法符号 B**：转化为对函数 B() 的调用，并接收其返回值 B_s。
    2. **中间动作 { C.i = f(B.s) }**：转化为一段赋值语句，利用刚刚拿到的 B_s 计算出 C_i。
    3. **文法符号 C**：转化为对函数 C(C_i) 的调用，将继承属性作为实参传入。
    4. **末尾动作 { A.s = g(C.s) }**：转化为最后的逻辑计算，并由 return 返回结果。
- **转换后的函数伪代码：**
    
    ```python
    def A(A_i): # A 可能也有自己的继承属性 A.i
        # 1. 匹配并执行 B
        B_s = B() 
        
        # 2. 执行中间动作：计算 C 的继承属性
        C_i = f(B_s) 
        
        # 3. 执行 C，传入继承属性
        C_s = C(C_i) 
        
        # 4. 执行末尾动作：计算并返回 A 的综合属性
        A_s = g(C_s)
        return A_s
    ```
    


### 2. 非递归的自上而下实现（Stack-based LL）
在 LL 分析中，符号栈不仅要存文法符号，还要存**“动作记录（Action Records）”**。
- **实现机制：**  
    当栈顶是非终结符 **A** 时，根据预测分析表将其展开。为了保证从左到右执行，必须将产生式右部**逆序压栈**。
- **压栈顺序（从栈底到栈顶）：**
    1. **{ Action 2: A.s = g(C.s) }**：计算 A 综合属性的动作记录。
    2. **C**：非终结符 C。
    3. **{ Action 1: C.i = f(B.s) }**：计算 C 继承属性的动作记录。
    4. **B**：非终结符 B。
        
- **执行流程：**
    
    1. 弹出 **B**，进行分析，计算出 B.s 并存入临时位置（属性栈）。
    2. 弹出 **Action 1**，立即执行 C.i = f(B.s)，将结果存入属性栈中为后续 C 准备的位置。
    3. 弹出 **C**，分析时从属性栈中读取 C.i。
    4. 弹出 **Action 2**，执行最终计算。
- **关键点：** 需要一个额外的**属性栈**来配合符号栈，确保动作记录能精准地找到它左侧符号产生的属性值。

### 3. 自下而上实现（LR Parsing / Marker Non-terminals）
LR 分析器只能在归约（Reduction）时执行动作。由于中间动作 { C.i = f(B.s) } 出现在产生式中间，LR 无法直接处理，必须进行**改写**。
- **步骤一：引入标记非终结符（Marker）**  
    将原产生式改写为：
    1. **A → B M C { A.s = g(C.s) }**
    2. **M → ε { M.s = f(B.s) }**  
        这里我们将中间动作变成了一个产生空串的非终结符 **M** 的归约动作。
- **步骤二：利用 LR 属性栈进行计算**  
    在 LR 分析器的值栈（Value Stack）中，每个状态都对应一个属性值。
    1. 当 **B** 归约时，其综合属性 B.s 处在栈顶某位置。
    2. 接着分析器会发现一个空的 **M**。此时执行 **M → ε** 的归约。
    3. **核心操作：** 在归约 **M** 的动作代码中，通过**相对偏移量**去拿 **B.s**。  
        M.s = f( val_stack[top] ) // 此时栈顶是 B 的值
    4. 归约完成后，**M.s** 被压入栈。
    5. 随后 **C** 被处理，其值 C.s 压入栈。此时栈的情况是：[... , B.s, M.s, C.s]。
    6. 最后归约 **A → B M C**，动作代码执行 A.s = g(val_stack[top])（即 C.s）。
- **总结技巧：**  
    在 LR 实现中，**M.s 实际上充当了 C.i 的存储介质**。由于 M 在栈中紧邻 C 的左侧，C 在归约时可以轻易地通过 val_stack[top-1] 拿到这个“继承”下来的值。
