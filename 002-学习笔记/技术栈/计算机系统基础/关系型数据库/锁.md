> 锁解决的是多事务并发下的数据一致性问题

## 什么是数据库的锁，有哪些类型


1. 全局锁
	- 锁住整个数据库，经典的应用场景是数据库的全库备份
	- 通过 `Flush tables with read lock` ，来对整个数据库实例加读锁，让数据库处于可读状态，防止所有对数据的修改操作，保证备份期间数据的一致性。
	- 通过 `unlock tables` 来解锁
2. 表级锁
	1. 表锁
		- 最简单的表级锁，锁的粒度较大，并发的性能较低，开销也比较小
		- 通过 `LOCK TABLES` 加锁：`LOCK TABLES table_name READ`(表读锁)、`LOCK TABLES table_name WRITE`(表写锁)
		- MyISAM 存储引擎默认使用表锁
	2. 元数据锁（MetaDate Lock / MDL）
		- 在访问数据库对象（表、视图、存储过程）时，给对象所属表加的锁
		- 当对表进行DDL操作的时候，如果表上有MDL锁，会阻塞事务。**目的就是保证对表结构的修改不影响DML操作**
	3. 意向锁(Intention Lock)
		- Inno DB 引擎自动添加的表级锁
		- 当事务读取表的数据行时，给表加IS锁；当事务写表的数据行，给表加IX锁
		- 目的是：**在需要控制锁的粒度，实现较高并发程度的情境下，用意向锁简化显式锁和隐式锁的判断逻辑，改为直接判断父节点的意向锁**
	4. 自增锁(AUTO—INC)
		- 在向 有自增列的表 中插入数据时，会自动使用，插入结束后释放
		- 目的时保证自增列数据能**正确自增**
3. 行级锁
	1. 记录锁
		- 也叫做行锁，锁的对象时数据表的行记录
		- 在事务修改或者删除或者写入表数据时，会加**排他记录锁**
		- 在事务读取表行数据，会加**共享记录锁**（在特定的隔离级别下）
	2. 间隙锁
		- InnoDB 在 RR 的隔离级别下，为了防止幻读引入的一种锁
		- 锁的对象不是记录，而是**索引记录的行间隙**，会阻塞对其他事务在间隙间插入数据
	3. 临界锁
		- Inno DB 默认的行锁类型
		- 锁的对象是**索引记录以及该记录之前的间隙**
		- 同样，在 RR 的隔离级别下使用该锁，可以额外防止幻读
	4. 插入意向锁
		- 一种特殊的间隙锁，在事务执行 Insert 时产生
		- 多个事务在同一个间隙之间插入记录，如果插入的位置不冲突，可以同时持有插入意向锁

## 悲观锁和乐观锁

1. **悲观锁**
	- 假设并发操作会引发较多的冲突。要求在读取数据的时候加锁，直到事务完成
	> 在现代化的DBMS中，设置隔离级别为RR，也只是通过使用MVCC，解决读写冲突，保证读操作读到的是一个有效的版本
	> 但是这样还是不能保证业务上的数据一致性
	- 保证读取数据的一致性

2. 乐观锁
	- 乐观锁假设并发冲突不常发生，因此在数据读取时不加锁，而是在数据更新时检查是否有其他事务修改过数据，如果有冲突则回滚或重试

## MVCC

MVCC是通过Undo Log 和 ReadView 两个核心组件，不使用锁，来实现数据库事务的 RC 或 RR 隔离级别。

Undo Log 是事务执行的记录，包括了事务的ID，执行的内容，可以推导出执行前的数据库状态。
ReadView会在读取事务执行时生成，通过记录一些事务ID，用于判断读取事务能够读取的有效数据库版本

事务中每一次读取操作生成一个 ReadView，可以实现RC的隔离级别
事务中的读取操作都是使用同一个 ReadView， 可以实现RR的隔离级别
