# 从 HTTP1.0 到 1.1

- 长连接：通过一个TCP连接可以发出多次请求和响应；不需要每一次请求都建立连接、释放连接
- 缓存方式：在1.0中，通过expire、If-Modified-Since字段作为缓存依据；HTTP/1.1 则引入了更多的缓存控制策略
- 带宽优化
	- 请求体压缩
	- 范围请求：请求文件中指定范围的数据，不需要从头传输（依赖于服务器端的文件读取的系统调用能力）
- 多主机名绑定IP：一个主机部署多个服务，通过反向代理访问服务，如果不加上请求域名，代理服务不知道分发给哪一个服务


# 从HTTP1.1 到 2.0

- 多路复用：一个连接可以同时处理多个请求和响应
	- 1.1 的长连接，会有应用层队头阻塞的问题
	- 2.0 的多路复用，只会在TCP顺序交付的机制中，在一个请求响应的窗口推进中因为头部数据丢失阻塞


# 从HTTP2.0 到HTTP3.0

- TCP vs QUIC
	3.0 使用了 QUIC 协议进行传输，在应用层实现可靠传输，具有较低的连接和传输延迟
- 支持对请求头进行压缩


# HTTPS 加密传输

HTTPS 不是一个全新的引用层协议，而是通过在 HTTP 和 TCP 之间加入一层 TLS/SSL ，补充了加密、校验的功能。


1. 端口号：HTTP使用80端口，HTTPS使用443端口
2. URL前缀：HTTP的URL前缀时`HTTP://`,HTTPS的前缀是`HTTPS://`
3. 安全性和性能消耗：HTTP工作在TCP协议上，传输明文内容，不进行加密，客户端和服务器端无法验证对方的身份；HTTPS工作在SSL/TLS协议，这两者工作在TCP协议，传输内容采用对称加密，对称加密的密钥用服务器的CA证书非对称加密。所以说，HTTP 安全性没有 HTTPS 高，但是 HTTPS 比 HTTP 耗费更多服务器资源。
4. SEO：搜索引擎优化，现在的搜索引擎，更青睐使用HTTPS的服务器，搜索结果会优先展示


加密过程简述：
### 1. 探路阶段

客户端向服务端发送探路消息，包含 **随机数1**、TLS版本、加密算法列表、SNI（域名）。

### 2. 服务端选定并给线索

服务端从列表里选好算法 A、B，生成 **随机数2**。
服务端是用私钥对算法 A 的“数学线索”进行了数字签名（加密了摘要）。
- 服务端把 CA 证书和这个签了名的线索一起发给客户端。
### 3. 客户端验身并隔空拼图
客户端收到大礼包，用 CA 证书里的**公钥**去解密那个数字签名。如果成功，说明服务端身份合法。

- 客户端拿到服务端的“数学线索”，结合自己**当场生成的一个秘密数字（临时密钥）**，隔空计算得到了 **预主密钥3**。
- 客户端把自己的“数学线索”也发给服务端。服务端收到后，用自己的秘密数字（临时密钥）一乘，也在内网算出了**一模一样的预主密钥3**。
- **各自搅拌：** 现在双方手里都有了：**随机数1 + 随机数2 + 预主密钥3**。两端各自在内网算出了后续对称加密用的 **终极主密钥**。

### 4. 试探对账
客户端试探性地用这个 **终极主密钥** 和服务器进行对称加密对账，服务端用相同的主密钥解密并成功回复，整个握手过程结束。
