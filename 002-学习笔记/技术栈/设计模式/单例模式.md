## 懒汉式的单例模式

如果想要实现细粒度的锁，可以锁住创建单例实例的部分，但是需要双重检查，同时防止指令重排

在Java单例模式的双重检查锁（Double-Checked Locking，DCL）实现中，`volatile`关键字是**必需的**，主要原因是防止**指令重排序**导致的线程安全问题。

## 问题场景

先看典型的DCL单例实现：

```java
public class Singleton {
    private static volatile Singleton instance;
    
    private Singleton() {}
    
    public static Singleton getInstance() {
        if (instance == null) {  // 第一次检查
            synchronized (Singleton.class) {
                if (instance == null) {  // 第二次检查
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
```

## 为什么需要volatile

关键在于这行代码：`instance = new Singleton()`

这条语句在JVM中并非原子操作，实际会分解为3个步骤：

1. **分配内存空间**
2. **初始化对象**（调用构造函数）
3. **将instance指向分配的内存地址**

### 指令重排序的危险

在没有`volatile`的情况下，JVM和CPU可能会进行指令重排序优化，执行顺序可能变成：1 → 3 → 2

这会导致以下问题：

- **线程A**执行到步骤3时，instance已经不为null（指向了内存地址），但对象还没初始化完成
- **线程B**此时执行==第一次检查==，发现`instance != null`，直接返回这个未完全初始化的对象
- **线程B**使用这个半成品对象时会出现不可预期的错误

## volatile的作用

`volatile`提供两个关键保障：

1. **禁止指令重排序**：确保对象创建的3个步骤按顺序执行（1 → 2 → 3）
2. **内存可见性**：一个线程对instance的修改立即对其他线程可见

这样就保证了其他线程看到instance不为null时，对象一定已经完全初始化完成。

## 总结

没有`volatile`的DCL是**线程不安全**的，可能返回未完全初始化的对象。添加`volatile`后，通过happens-before规则保证了对象的安全发布，这是DCL模式正确性的核心保障。
