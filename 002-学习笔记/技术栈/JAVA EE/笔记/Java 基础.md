## Java的数据类型分为哪两种，区别是什么

Java的数据类型可以分为基本数据类型和引用数据类型。

其中，基本数据类型变量在内存中存放变量的值；引用数据类型存放变量指向对象在内存中的地址。

Java中的基本数据类型有
- 整型
	- byte（1字节）
	- short(2字节)
	- int(4字节)
	- long(8字节)
- 浮点类型
	- float（32位）
	- double(64位)，默认都是double，如果需要用float，在数值后添加F或者f
- 字符
	- char（2字节）
- 布尔型
	- boolean（1位）

Java中的引用数据类型有
- 接口
- 类
- 数组

在用变量给变量赋值的时候，两者表现不同：
	基本数据类型变量赋值的时候，实现的是深拷贝，即内存中有两份数据
	引用数据类型变量赋值，变量存放相同的地址，即内存中实际上还是只有一份数据，实现的是浅拷贝

基本数据类型的一些特点
- 整型转换：小转大可以隐形转换，大转小需要显式转换
- `char、short、byte`在参与运算的时候，会隐式的`int`
- char使用Unicode编码，占2字节
- boolean不能和整型相互转化，和C++不一样

## Java中的\==和equals()方法有什么区别？


### Java中的\==运算符

== 是Java提供的比较运算符，其比较的内容是左右运算变量的内存地址。对于基本数据类型，等效于比较值。对于引用数据，比较内存的地址。

### equals()方法

equals方法是Object类中的方法，在Java中，所有的类都是Object类的子类。方法的作用是实现变量的值判等逻辑。Java提供的类中，一般通过重载equals方法，实现逻辑判等。在用户的自定义类中，也可以通过重写equals方法，自定义判等逻辑。
对于没有重写equals方法的类，equals实现的也是物理内存地址的判等。

## Java中的方法重载（Overloading）和方法覆盖（Overriding）（重写与方法覆盖是同一概念的不同表述））有什么区别？


重载和重写都是OOP类型语言中，多态的实现方式。二者的核心区别是运行时的多态和编译时的多态。

### 重载

**重载是指在同一个类中，有多个同名方法，但是参数列表不同**。Overloading是编译时多态，即编译器在编译时就能确定运行的具体的方法。区分的方式是通过传入方法的参数个数、类型、顺序的不同。对于重载的方法，其方法体内部不一定相同，返回的数据类型也不一定相同，但是方法名一定是相同的。

### 重写

**重写指的是子类重写父类方法，方法名和参数列表都相同**
Overriding是运行时多态，即编译器在编译的时候不能确定方法执行的具体内容，只有在实际运行行，根据上下文环境，来动态选择运行的具体方法。
重写往往发生在子类对父类的方法进行覆盖的场景中。在Java中，重写的方法要求方法名相同，方法的访问权限不能缩小，方法抛出的异常不能扩大，子类方法中的传入参数要求和父类完全一致，返回类型允许协变返回。
重写函数的方法名也一定是相同的，在Java中，可以通过@Override注解检查重写是否正确。

1. default：同一个包内的类可以访问
2. protected：同一个包内的类或者其他包的子类可以访问。

## Java中的静态变量（static）和非静态变量有什么区别？


static变量和非静态变量的主要区别在于访问方式、存储内存、生命周期

1. 访问方式
	1. static变量一般通过类名.方法名的方式进行访问，也可以通过对象名.方法名的形式进行访问
	2. 非静态变量只能通过对象名.方法名的方式进行访问
2. 存储内存
	1. 静态变量存储在JVM的元空间中，包括字段的信息和字段的值
	2. 非静态变量的字段信息存储在JVM的元空间中，但是实例化的值存在在堆区中。
3. 生命周期
	1. 静态变量在类首次被加载的时候就创建，当类在元空间被销毁时随带被销毁
	2. 非静态变量在对象实例化的时候被创建，当对象被销毁的时候随带被销毁
> 访问方式的不同也说明了他们所属的对象和生命周期的不同：静态变量是属于类的，生命周期和类一致，非静态变量是属于对象的，生命周期和对象一致。
> 类的静态变量被类的所有示例对象全局共享

## Java中的异常处理机制是怎样的


Java中的异常，如果按照继承关系来看
	1. Throwable类是所有异常类的根，继承Object类
	2. Error类和Exception类继承Throwable类
	3 .Exception类的子类有RunTimeException和其他子类

如果按照程序是否需要捕获异常的角度看
	1. Error异常是程序无法处理的错误，不需要捕获
	2. RunTimeException异常和其子类异常是运行时会出现的异常，不需要捕获
	3. 其他异常都需要被捕获

Java异常的捕获机制
	Java的异常捕获可以是链式处理的，即在方法的嵌套调用中，被调用方法可以选择不处理方法内部发生的异常，通过throws关键字抛出异常，交给调用者进行处理
	当方式内处理异常，需用try、catch、finally关键字
		try语句块内部是可能发生异常的代码块
		catch语句块内编写对异常的处理代码
		finally语句块内的代码编写try和catch运行完后的处理代码

Java异常处理的特殊情况
	1. catch和finally中都抛出异常，最终只会抛出finally中的异常-异常屏蔽现象，可以通过在方法体内设置一个Exception变量，在catch中赋值，在finally中，给原本要抛出的异常使用addSupressed方法
	2. 当catch语句需要处理异常种类可能性有很多，并且之间有继承关系时，需要先catch子类，在catch父类，保证子类额外信息不遗漏
	3. 当catch的多种异常没有继承关系，并且处理方式一样时，可以一次性catch多个异常

## Java中的final关键字可以修饰什么？被final修饰后有什么特点？


1. 修饰变量
	1. 如果是基本数据类型，那么变量值在初始化之后，不允许修改变量的值
	2. 如果是引用数据类型，那么变量指向的地址不能变，指向地址的内容可以修改
2. 修饰方法
	1. 该方法不能被子类重写
3. 修饰类
	1. 该类不能被继承

### 详细回答

- 修饰类
	- 特点
		使用final修饰的类不能被别的类继承，但是可以继承别的类。可以实例化对象，对于abstract类不使用final，因为抽象类需要子类重写抽象方法。
		一个类如果声明为final，那么方法一般不需要声明final，因为不会有子类重写的情况
	- 设计目的
		- 保证类的不可变性（比如String）
		- 确保安全性（比如防止System的某些核心API被篡改）
- 修饰方法
	- 特点
		- final不能被子类重写，但是可以被继承使用
		- final方法不能是abstract，因为抽象方法是需要子类重写的
	- 特点
		- 防止核心的业务逻辑被篡改
- 修饰变量
	`final`修饰变量，可以从以下两个维度进行考虑
	其一是修饰变量的类型：
	1. 如果修饰基本数据类型：意味着变量的值无法被改变
	2. 如果修饰引用数据类型：意味着变量无法指向一个新的对象，但是变量指向对象内部的状态可以变化
	
	其二是修饰变量的位置：
	1. 如果是在方法内修饰变量为`final`，那么只需要在使用前进行赋值即可，并且只能够赋值一次
	2. 如果是在类字段修饰变量为`final`，则需要继续细分
		- 静态常量：需要在类加载时，得到赋值。可以是 定义处初始化；静态代码块中初始化赋值；静态代码块中调用静态方法初始化
		- 成员常量：需要在类对象初始化中，得到赋值。可以是 定义出设置默认值；构造函数中赋值； 代码块中赋值

## 解释一下Java中的自动装箱和拆箱。

### 包装类概念引入

在Java，有包装类的概念。其作用是增强基本数据类型的功能。提供一系列便捷方法，以及安全保证措施。

### 自动装箱和自动拆箱
其中，包装类和对应的基本数据类型之间的转换就叫做自动装箱和拆箱。

1. 自动装箱：将基本数据类型的值直接赋值给对应的包装类。编译器会实例化一个包装类对象，内部的值字段就是赋值过来的变量。这个过程有编译器完成，不需要程序员手动实例化。
2. 自动拆箱：使用包装类对象赋值给基本数据类型。这个过程也有编译器完成。
   
### 编译器实现本质，以`Integer`和`int`为例子：
1. 自动装箱：调用`Integer`的`valueOf`方法
2. 自动拆箱：调用`Integer`的`intValue`方法

### 使用包装类优势

1. ==符合Java面向对象的特点==
2. ==可以配套Java的集合、泛型使用==

### 使用自动装箱和自动拆箱的缺点

1. 会频繁的创建对象和删除对象。降低代码的性能
2. 在拆箱中，如果包装类对象为null，会报出异常`NullPointerException`

## 解释Java中的泛型和作用

1. **泛型的概念**
	Java中的泛型是一种通过类型参数化提高代码的通用性和类型的安全性的机制。通过泛型，类、方法、接口可以针对泛化的参数类型进行操作，不依赖于具体的参数类型。总的来说，泛型就像是一套模板，在运行时确定具体的类型信息。
2. **泛型的作用**
	- ==类型安全==：使用泛型，编译器可以在代码编译的时候检查类型安全，不需要等到运行的时候在报错
	- ==提高代码的可重用性==：通过泛型代码，可以用同一套业务逻辑处理代码，处理多种不同的数据类型，避免了编写高度相似的代码，提高了代码的可重用性和通用性
	- ==强制类型转换==：使用泛型的地方，编译器会自动推断上下文信息，插入强制类型转换，不需要程序员手动写
3. **泛型的形式**
	使用`<>`包围使用的泛型类型，在代码中使用包围起来的类型变量，不使用实际的类型
	`class GenericClass<T> {}`
	`static <T> void GenericMethod(T arg) {}`
4. **泛型应用场景**
	泛型可以声明类，接口，方法。泛型类或者接口中，可以使用泛型的成员变量；泛型方法中，可以使用泛型参数
5. **泛型的边界**
	上边界：指定泛型变量的时候`<T> -> <T extends Numebr>`，就表示T只能是Number类以及他的子类
	下边界：指定泛型变量的时候`<T> -> <T super Numebr>`，就表示T只能是Number类以及他的父类
6. **通配符**
	通配符是使用泛型的一种方法，不是声明泛型类型的方法。
	`<? extends Number>`只读，生产者
	`<? super Number>`只写，消费者

## Java中的String类设计为不可变的优势，如何保证不可变

### 不可变优势
1. **线程安全**
	不可变类在多线程中天然具有线程安全的优势。当某一个线程试图修改String类对象时，会创建一个新的String对象，并且修改变量的引用地址。
2. **安全性**
	String对象存储一些敏感信息的时候，不用担心代码拿到String会修改信息
3. **性能优化**
	当用字符串值直接赋值字符串变量时（不是通过new），会引用字符串常量池中的已有对象，如果存在的话
### 实现保证
1. String类是final的：也就是说，无法通过子类继承String，改变String的Immutable
2. String类内部的字节数组是final的，也就是说，初始化之后无法进行引用修改
3. String类没有提供值修改的方法
4. 所有修改String对象操作，都会返回一个新的String对象

> 不可变的缺陷：当涉及到大量的对字符串修改，会大量的在堆区中创建对象，性能不佳
> 可以通过StringBuilder，StringBuffer构建字符串

## String、StringBuffer 和 StringBuilder 有什么区别？

| 特性       | String         | StringBuffer     | StringBuilder    |
| -------- | -------------- | ---------------- | ---------------- |
| **是否可变** | 不可变            | 可变               | 可变               |
| **线程安全** | 是              | 是 (通过同步实现线程安全)   | 否                |
| **性能**   | 性能较低（频繁修改时不推荐） | 性能较低（同步操作影响性能）   | 性能较高（适合单线程环境）    |
| **适用场景** | 字符串不变时，常量池等使用  | 多线程环境下需要修改字符串时使用 | 单线程环境下频繁修改字符串时使用 |
> 我们常说的使用 `+` 拼接 `String` 对象，性能较低，指的是在多行语句中拼接。
> `+` 的拼接中，只有有一个变量表示，就会创建堆区的 `String` 对象，否则会创建常量池中的对象，编译器还会采用字符串折叠的技术，减少常量池创建对象开销
> 在涉及到变量的 `+` 拼接，首先创建 `StringBuilder`对象，然后调用多次 `append`， 最后调用 `toString`， 这个和直接使用 `StringBuilder` 是完全一样的。区别就在于将 `+` 过程分到多行语句中。会反复创建 `StringBuilder`，说的性能低，也是这个性能低
> `concat` 函数，就是在堆区创建字符串，将拼接的两个字符串分别复制到新的缓冲内存中。哪怕有一个是空字符串，一定会创建一个新对象，这满足不变性的要求
## Java中的Optional类是什么？为什么需要它？ (考点：Java Optional类及其必要性) 【中等】

1. 定义：
	- 是一个包装类，内部封装的对象可能是 `null`
2. 特点：
	- 明确处理空值逻辑，`orElse`，`isPresent`，`ifPresent`方法的设计，都表达出需要处理空值的逻辑
3. 作用：
	- 代码更加的简洁，可以替换多层判空，对象内部有类对象成员变量 `if (obj != null)`
		`Optional<Person> o` `o.map(p - > p.address).map(a -> a.locationName).orElse("")`
	- 避免 `NullPointerException`，因为它的方法都要求处理空值
	- 接口的`API`更加明确，在函数的传入参数、或者返回参数中指定类型为 `Optional`=，说明有可能会是空值，语义更加的明确
