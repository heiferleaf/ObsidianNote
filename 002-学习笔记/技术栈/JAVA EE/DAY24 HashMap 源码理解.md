## 核心源码深度解析

### 1. 扰动函数：`hash(Object key)`

为了减少哈希碰撞，`HashMap` 没有直接使用 `hashCode()`，而是进行了高位参与运算。


```Java
static final int hash(Object key) {
    int h;
    // 让高16位与低16位进行异或，充分混合哈希值
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
```

- **目的**：当数组长度较小时，只有低位参与寻址运算。异或操作能让高位特征也影响索引结果，显著降低碰撞率。

### 2. 寻址算法：为什么容量必须是 2 的幂？

在源码中，索引的计算方式为：`i = (n - 1) & hash`。

- **逻辑**：当 `n`（长度）是 2 的幂时，`n-1` 的二进制全为 1（如 16-1 = 1111）。
    
- **优势**：`&` 运算效率极高；且能保证哈希值的每一位都均匀分布在数组索引中。


### 3. `putVal`：写入数据的逻辑决策

这是 `HashMap` 的核心方法，逻辑如下：

1. **检查 table**：如果数组为空，先调用 `resize()` 初始化。
    
2. **计算索引**：找到桶位。
    
    - **桶为空**：直接新建 `Node` 放入。
        
    - **桶不为空**：
        
        - **Key 相同**：直接覆盖 `value`。
            
        - **红黑树节点**：调用 `putTreeVal` 写入树。
            
        - **链表节点**：尾插法遍历。
            
            - 若链表长度达到 `TREEIFY_THRESHOLD (8)`，且数组长度 $\ge 64$，触发 `treeifyBin`（树化）。
                

---

### 4. `resize`：扩容与平移的艺术

扩容不仅仅是翻倍，更精妙的是数据平移：

- **扩容触发**：当前元素个数 `size > capacity * loadFactor`。
    
- **平移规律**：元素新位置要么在**原位置**，要么在**原位置 + 旧容量**。
    
    - **原理**：扩容后掩码 `(n-1)` 在高位多出一个 `1`。只需判断 `(hash & oldCap) == 0` 即可知道位置是否需要变动，无需重新计算哈希。
        

---

### 5. 红黑树的特殊性：`TreeNode`

`HashMap` 中的红黑树节点 `TreeNode` 不仅有左右子节点，还维护了 `prev` 和 `next` 指针。

- **双向链表特征**：即使变成了红黑树，它内部依然维持着一套双向链表的顺序。
    
- **作用**：
    
    1. 方便在 `untreeify`（反树化）时快速转回链表。
        
    2. 方便扩容迁移时保持数据的逻辑顺序。
