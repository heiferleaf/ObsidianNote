## 设计原则

> SOLID、最少知识原则

### S:单一职责原则

**定义**：一个类，只关注属于自己的部分。比如`user`类中，就不应该有用户注册方法、用户登录方法，只需要有和用户自己相关的方法，比如属性的getter和sertter。

**好处**：实现一个类的高内聚

### O:开闭原则

**定义**：对拓展开放，对修改关闭。需要实现一个新功能，应该是写新的代码，而不是修改已经存在的代码。
**实现**：依赖于抽象，使用接口和抽象类作为函数的参数，这样在实现新功能的时候，才不会修改函数内部的代码

**好处**：保持软件架构的稳定

### L:里氏替换原则

**定义**：凡是出现父类的地方，都可以用子类进行替换
**实现**：在子类写多态函数，保证能在父类该函数使用的地方，替换为子类而不出错
	如果正方形继承长方形，长方形有方法setWidth和setHeight，正方形的如果将这两个函数改为多态，必将需要在函数中，同时设置width和height。于是，在一些父类函数的使用场景中（增大width直到大于height），就不能使用子类的函数（正方形同时设置width和height，是一个死循环）
	子类尽量写新方法，少用多态，即便要使用，也要保证能替换父类出现

**好处**：保证多态的稳定性

### I:接口隔离

**定义**：一个类实现的接口中，不应有不需要的方法。
**实现**：对接口方法细拆分，做到方法场景语义的高内聚

好处：代码的简洁，同时符合语义

### D：依赖倒置

**定义**：高层模块和底层模块依赖于抽象，细节依赖于抽象，使用抽象编程
**实现**：方法参数中，使用抽象类或者接口作为参数；类关联的成员，使用抽象的类型

**好处**: 降低了使用者和方法内部的耦合程度（方法内部不需要在意使用者传入的具体类型，依赖于抽象编程）


### 最少知识原则

**定义**：类的朋友尽可能少
**实现**：类中，关联的成员，依赖的成员，产生的类对象，种类尽可能少
	明星和经纪人是朋友，和公司、粉丝没有直接接触，就不要在明星类中关联公司和粉丝
	经纪人和明星、粉丝、公司有接触，在类中可以关联成员，可以编写方法，将这几者联系起来。
	（这也是合成复用的体现，实现方法内部的低耦合、高内聚）
好处：类在符合现实语义的情况下，尽可能保持低耦合

### 合成复用

**定义**：实现一个新方法，尽量通过类的组合、聚合、关联、依赖来解决问题，而不是通过类的继承，来使用父类的方法
1. 通过类的继承
	**特点**：继承的耦合程度很高，父类的细节暴露给子类，属于"白箱复用"
	父类代码编译时确定，导致子类的复用情况也是确定的
2. 通过类方法的组合
	**特点**：关联类的方法不暴露，方法之间低耦合，属于"黑箱复用"
	在加上成员可以依赖于抽象，所以成员在运行时的行为可以通过多态动态改变

好处：整体的耦合度降低了，灵活性提高。


## 设计原则应用到编码

1. 对于接口：
	1. 接口中的方法在语义上保持高内聚，做到类在实现时都是需要的方法
2. 对于类：
	1. 习惯于定义抽象类或者接口，在编写具体实现
	2. 在关联类对象时，使用抽象的类型，保证在方法使用中，多态的灵活性
	3. 在实现多态函数时，保证可以替换父类函数出现的地方
	4. 关联或者依赖的类成员，保证是在语义上需要的，做到最少朋友
3. 对于方法：
	1. 参数依赖于抽象：内部低耦合，参数对象可以动态执行方法
	2. 实现新方法，尽量通过已有关联类的组合，而不是通过继承父类，使用父类的方法


## 创建型

### 单例模式

单例模式适用于以下情景：
1. 资源共享：创建系统中唯一的一份资源，使得资源共享变的简单，通信避免多余资源的创建
2. 日志管理器：管理性质的，全局要唯一
3. 线程池：保证线程池只有一份
4. 缓存：缓存也只能有一份，做到数据的一致性

总结的通用一些：
1. 会反复创建一个对象然后销毁，使用单例，内存中保有一份对象资源
2. 一些控制资源，只能有一个控制器，比如日志管理，比如线程池
3. 经常使用的对象，但是实例化时间长或者资源比较多，比如数据库的连接池

### 原型模式

当对象创建开销较大，并且创建出来的对象，大部分内容相同，可以通过原型方法，进行快速的克隆。

1. 浅拷贝：对象变量指向的堆区地址相同
2. 引用拷贝：对象变量指向的堆区地址不相同，但是对象内部引用数据类型指向同一片堆区地址
	需要把`String`排除在外，考虑到不可变性
3. 深拷贝：完全的克隆出一份独立的对象

### 建造者模式

![[QQ_1767514422444.png]]
**当创建对象的各个部分配置比较复杂，通过构造函数创建会使得代码不够简洁，通过建造者**

抽象建造者类（Builder）：这个接口规定要实现复杂对象的那些部分的创建，并不涉及具体的 部件对象的创建。 
具体建造者类（ConcreteBuilder）：实现 Builder 接口，完成复杂产品的各个部件的具体 创建方法。在构造过程完成后，提供产品的实例。 
产品类（Product）：要创建的复杂对象。 
指挥者类（Director）：调用具体建造者来创建复杂对象的各个部分，在指导者中不涉及具体产 品的信息，只负责保证对象各部分完整创建或按某种顺序创建。

优势：
	1. 对象创建和使用解耦
	2. 更加精细的配置对象

缺陷：
	不同的建造者继承抽象建造者，这就要求不同的建造者，造出的产品，部件是类似的，比如都有CPU、机箱、显示屏

### 创建型对比

| 工厂         | 建造者        |
| ---------- | ---------- |
| 注重于对象整体的创建 | 注重于对象细节的配置 |


## 结构型

关注如何组织类和对象，分为类结构性和对象结构性。
- 类结构：通过继承和实现来组织类间结构
- 对象结构：通过类的组合、关联

### 代理模式

### 装饰器

#### 装饰器和静态代理的区别

相同
1. 都实现或者继承目标类抽象
2. 内部持有对象
3. 不改变目标类的前提下，增强功能

不同
1. 内部持有对象来源不同：装饰器可以通过外部传入，静态代理通过内部直接实例化，实现保护
2. 目标不同：装饰器目标是强化功能，静态代理目标是保护封装

> IO流的FIlter模式，就是装饰器的体现，可以将流对象放到别的功能的流的构造函数中，得到增强的流

### 外观模式

但是不符合开闭原则，当系统发生变化，提供的接口很可能也变化
