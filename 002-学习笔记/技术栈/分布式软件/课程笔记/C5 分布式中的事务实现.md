> 事务，在原来的单机应用的场景中，一般是一个服务逻辑中，涉及到的对数据表的多次sql操作，所具备的 ACID 特征。
> 
> 在分布式的场景中，对数据库的sql操作，可能属于多个服务，比如订单和库存，订单确定之后，库存减少，这里sql操作可能是在多个物理机中，至少是在多个进程中执行的。传统的事务无法生效，原子性无法保障。


## 分布式事务需要考虑的问题

1. 原子性操作保证：
2. 隔离性的保证：分布式环境中，并发事务之间如何隔离，不相互影响
3. 容错问题：网络分区无法通信怎么办，协调者挂了怎么办，参与者挂了怎么办
4. 性能问题：在保证一致性的前提下，怎么减少锁的持有时间，和网络IO 次数

## 刚性事务（强一致性）

2PC 和 3PC 协议，主要都是解决了分布式事务的原子性问题，要么都做，要么都不做，当然有可能不能实现原子性。但是==没有完美解决==数据的一致性问题

### XA规范和2PC协议

2PC，两个阶段
- 第一个阶段，协调者向所有参与者发送 Prepare 请求，每一个参与者执行的时候，写undo log，但是现不提交，像协调者返回 yes / no
- 第二个阶段，协调者搜集所有的回答，如果有 no，发送rollback，所有的参与者回滚，反则提交

2PC存在的问题：
- 协调者挂了，就用不了
- 所有的参与者需要等待其他参与者的响应，是同步阻塞
- 数据不一致，可能部分参与者收到第二阶段的commit/rollback信息，导致数据的不一致
> 问题的本质：因为各个参与者，都是先执行操作，然后阻塞等待协调者的指令，是回滚，还是提交。
> 如果有一个结点无法执行操作，别的结点执行完操作之后，一直傻等，同时事务占用的锁资源也不会被释放

### 3PC

三个阶段
- 询问是否可以执行，如果有的结点无法回答，或者回答no，就可以在资源没有被占用的时候，提前返回事务失败的结果
- 剩下和2pc一样，协调者询问好了没，参与者执行，回答，不提交
- 协调者控制事务

生产者可以自己提交，如果等待时间太久了，这样可以防止阻塞，因为第一阶段大家都说可以执行操作，所以提交引起的数据不一性的概率比较小，同时也可以减少阻塞。

> [!NOTE] 对于一致性的补充做法
> 
> 2PC 3PC 都是完全有可能因为协调者宕机，或者网路分区故障，导致参与者结点数据的不一致性。
针对协调者的宕机，可以使用主从结点，通过主从复制协议比如 Raft 或者 Paxos，让单机协调者变为集群，同时把收集到的参与者的回答数据额外放到存储结点，存储节点也可以通过主从设计，保证数据不丢失。同时参与者实现幂等处理，可以重复收到commit不出错。

### 从数据库的事务 - 业务层的事务

考虑的原因
- 数据库层面的XA协议（2PC）锁的时间太长了，对数据库的压力大
- 数据的存储在不同的服务中，可能是 MySQL Redis ES 等不同的架构，所以用不了 2PC

### AT 模式

#### 1. 数据库准备：`undo_log` 表

必须在每个微服务的物理数据库中运行建表脚本。没有这张表，AT 模式会直接报错。

#### 2. Maven 依赖配置

在 Spring Boot 项目的 `pom.xml` 中引入 Seata 的 Starter 依赖。


```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-seata</artifactId>
</dependency>
```

### 3. 代码与核心配置

#### A. 配置代理数据源

在 Spring Boot 2.x 之后的版本中，Seata Starter 通常会自动代理你的数据源。但为了确保万无一失（特别是涉及多数据源或分库分表时），通常会显式配置：


```java
@Configuration
public class DataSourceConfig {

    @Bean
    @ConfigurationProperties(prefix = "spring.datasource")
    public DataSource druidDataSource() {
        return new DruidDataSource();
    }

    /**
     * 核心：将普通数据源包装成 Seata 的代理数据源
     */
    @Bean
    public DataSourceProxy dataSourceProxy(DataSource dataSource) {
        return new DataSourceProxy(dataSource);
    }
}
```

#### B. 配置文件 (`application.yml`)

你需要告诉服务：我的事务分组叫什么，以及 **TC (Seata Server)** 的地址在哪里。


```yaml
seata:
  enabled: true
  application-id: spike-order-service
  tx-service-group: my_spike_group  # 事务分组
  service:
    vgroup-mapping:
      my_spike_group: default      # 映射到 TC 的集群名
    grouplist:
      default: 127.0.0.1:8091      # TC 地址
  client:
    undo:
      log-serialization: jackson   # undo_log 的序列化方式
      log-table: undo_log          # 指定 undo_log 表名
```

---

#### 4. 业务使用：一键开启

配置完成后，使用起来极其简单。你不需要像 TCC 那样写三个方法，只需在**入口方法**上加一个注解：


```java
@Service
public class OrderService {

    @Autowired
    private StockFeignClient stockFeignClient; // 远程库存服务

    /**
     * @GlobalTransactional 开启全局事务
     * Seata 会自动处理所有的 SQL 拦截、快照抓取和 undo_log 写入
     */
    @GlobalTransactional(name = "create-order", rollbackFor = Exception.class)
    public void createOrder(OrderDTO orderDTO) {
        // 1. 本地扣减订单逻辑
        orderMapper.insert(orderDTO);

        // 2. 远程调用扣减库存逻辑
        // 此时库存服务的本地事务、undo_log 会在库存服务侧生成
        stockFeignClient.deduct(orderDTO.getProductId(), orderDTO.getCount());
        
        // 只要这里抛出异常，整个链路的所有 undo_log 都会被触发，自动回滚
    }
}
```

#### 5. 你的笔记深度补充：AT 模式的“锁”到底是怎么回事？

你在笔记中提到的“控制锁的释放时机”，在 AT 模式的代码层面体现为：

1. **本地事务提交（Local Commit）**：当 `orderMapper.insert` 执行完，本地数据库连接释放。此时物理锁已经没了，但 **`undo_log` 已经安全落盘**。
2. **全局提交（Global Commit）**：当 `createOrder` 方法彻底跑完且没报错，TM 会通知 TC。TC 再通知各服务**异步**删除 `undo_log`。
3. **冲突阻塞**：如果有另一个线程也想改同一行，虽然物理锁没了，但它在尝试获取 **Global Lock** 时会被 TC 挡住，从而实现了分布式环境下的隔离。




### TCC

一种业务层事务的实现方案

![[第五讲 分布式事务.pdf#page=24&rect=26,51,939,433|第五讲 分布式事务, p.24]]

1. 将原来所有服务完成的整个期间，占用锁的行为，改为在 try - commit - cancal 三个阶段分别占有锁的事件，并发程度更高
   
2. 对应1中的使用更短时间的锁的行为，以下订单的库存服务为例子 try冻结库存，commit实际减少库存，cancal取消冻结库存。try的行为需要先查询可用库存数量，然后冻结库存。当库用库存数量 > 申请数量时，服务可用。这个查询后冻结的行为，应该是原子性的。否则多并发下，可能会出现查询的时候库存足够，冻结的时候库存不够
	- 悲观锁，在查询语句加上 for update，实现正确的排他查询，然后更新冻结
	- 乐观锁，通过版本号，在其他并行进程try操作先执行的情况下，进行重试
	- 使用原子的update语句的where条件，单行语句解决查询库存+冻结的操作，通过返回的影响行数来判断是否执行成功
	  
3. 涉及到的对象
	1. TM，事务发起者
	2. TC，事务控制者
	3. RM，资源管理者，真正执行分布式事务的服务，try commit canal 执行的地方
	TM 首先先 TC 注册一个全局事务XID，TC中的服务被调用的时候，会向TC注册一个分支事务ID，并关联到XID。随后，RM执行try，把结果告诉TC，如果有失败，TC通知所有RMcancal，反之commit，把最后的结果告诉TM
	
4. 涉及到问题
	1. 空回滚：因为网络问题，TC发出的try命令比cancal命令后至，导致先处理cancal的操纵
	2. 多次提交，因为网络波动，TC发出的commit被多次接收，导致库存错误减少
	3. 悬挂：cancal先做，try后做，导致cancal什么都没有做，try之后库存永远冻结
	这三个问题都可以通过在数据库中多加一张表来解决，表字段 XID ID status，前两者对应一次事务的一个操作，status表示当前操作执行的状态，1-Tried, 2-Committed, 3-Rollbacked ，在对应的操作中，执行完之后，需要修改状态，执行之前，需要查看状态，保证不重复执行，不错误执行


> [!NOTE] 对于悲观锁或者乐观锁的在分布式场景下的补充
> 单机环境
> - 通过 @Transcational 注解，可以开启一个事务
> - 悲观锁在可重复读的隔离级别中的，锁释放的时机是事务结束之后
> - 结论：可以实现正确的逻辑，查询的时候，别的操作不能写
>   
> 数据库多结点，比如主从架构下
> - 悲观锁或者乐观锁的读操作，如果针对从库进行，会导致并发下的锁失效
>   
> 解决方法：
> 1. 在分布式事务的中的读写操作，都针对主库进行
> 2. 将悲观锁或者乐观锁这种针对数据库的读查询，改为使用分布式锁，比如Redission，既不会把读压力放到数据库，还可以通过Lua脚本实现原子性的读取后修改的操作
>    ==第二种方案更加的推荐，代替try的操作==


### SAGA 模式


如果说 **TCC** 是“先占位，再拿走”（预留资源），那么 **Saga** 就是“先拿走，不行再退回”（事后补偿）。它通常用于业务流程非常长、参与者非常多，或者涉及第三方系统（如银行转账、物流发货）无法提供 TCC 接口的场景。

 1. Saga 的核心理念：拆分与补偿

Saga 将一个长事务拆分为多个**本地事务** $T_1, T_2, ..., T_n$。

- 每个本地事务 $T_i$ 都有一个对应的**补偿事务** $C_i$（用于回滚）。
- **成功路径**：$T_1 \to T_2 \to T_3 \to \dots \to T_n$。
- **失败路径**：如果在执行 $T_i$ 时失败了，Saga 必须执行 $C_i, C_{i-1}, \dots, C_1$ 进行逆向操作。


 2. Saga 的两种实现方式：谁来带头？
#### 方式 A：协同式 (Choreography) - “多线单传”

没有中心协调者。每个服务在完成自己的任务后，发布一个消息/事件。

- **逻辑**：订单服务完成 $T_1$，发消息给库存服务；库存服务完成 $T_2$，发消息给支付服务。
    
- **缺点**：当流程变长，没人知道整个事务跑到了哪里，排查问题像迷宫。
    

#### 方式 B：编排式 (Orchestration) - “中心大脑”

这是目前主流（如 Seata Saga 模式）的做法。有一个 **Saga 状态机 (State Machine)** 负责决策。

- **逻辑**：状态机调用服务 A，成功后，状态机再调用服务 B。
    
- **优点**：流程可视化，状态可控。


3. Saga 的业务锁问题：ACID 中的 I（隔离性）

这是 Saga 与 TCC **最大的区别**
> **Saga 没有任何 Try 阶段，它直接修改数据，而不是通过预留字段。**

这意味着在整个分布式事务还没结束时，其他并发事务就能看到 $T_1$ 修改后的结果（产生了**脏写**或**不可重复读**）。

**解决隔离性问题的“业务手段”：**

1. **宁静模式 (Semantic Lock)**：在数据库增加一个 `status` 字段。例如订单状态为 `PENDING`，此时其他业务发现状态是 `PENDING`，则逻辑上拒绝修改。
2. **可交换策略**：如果业务是加减法（如加积分），由于加法满足交换律，可以不考虑隔离性。


3. 三大问题的 Saga 变体

|**异常情况**|**Saga 的应对方案**|
|---|---|
|**空补偿 (Empty Comp.)**|补偿事务 $C_i$ 触发时，发现 $T_i$ 压根没成功过。**对策**：$C_i$ 直接返回成功。|
|**幂等 (Idempotence)**|$T_i$ 或 $C_i$ 因为网络问题被多次调用。**对策**：必须在业务表或状态表记录执行记录。|
|**反向悬挂 (Hanging)**|补偿操作 $C_i$ 先于正向操作 $T_i$ 到达。**对策**：记录 $C_i$ 已执行，后续 $T_i$ 拒绝执行。|

5. Saga 的技术实现（以 Seata 为例）

在 Seata 中，Saga 是通过 **状态机图 (State Machine Graph)** 实现的。它不是靠注解，而是靠一个 **JSON 文件** 或可视化图形来定义：


```json
{
    "Name": "createOrderSaga",
    "States": {
        "DeductStock": {
            "Type": "ServiceTask",
            "ServiceName": "stockService",
            "ServiceMethod": "deduct",
            "CompensateState": "CompensateDeductStock", // 定义它的补偿节点
            "NextState": "CreateOrder"
        },
        "CompensateDeductStock": {
            "Type": "ServiceTask",
            "ServiceName": "stockService",
            "ServiceMethod": "addStockBack" // 补偿逻辑
        }
    }
}
```


6. Saga 与 TCC 的笔记对比总结

| **维度**   | **TCC**                | **Saga**                  |
| -------- | ---------------------- | ------------------------- |
| **资源锁定** | Try 阶段**预留**（锁定）。      | 没有预留，直接修改。                |
| **回滚机制** | Cancel 释放预留。           | **补偿**操作（逆向抵消）。           |
| **隔离性**  | 强（其他事务看不到中间态）。         | 弱（产生脏读）。                  |
| **开发成本** | 高（3个接口，逻辑复杂）。          | 中（2个接口，逻辑相对直观）。           |
| **适用场景** | 核心链路、高并发、一致性要求极高（如转账）。 | 流程长、调用第三方（如银行接口、12306订票）。 |

> [!NOTE] 补充：关于主从架构的延伸
> 
> 在 Saga 的补偿逻辑中，**主从延迟**依然是杀手。
> - 如果补偿事务 $C_1$ 试图恢复 $T_1$ 修改的数据，它必须去**主库**查找 $T_1$ 之前的镜像数据。
> - 如果 $C_1$ 根据从库过时的信息进行恢复，会导致“越补偿越乱”。
> - **最佳实践**：Saga 的状态记录必须持久化在主库中，所有补偿逻辑强制读主。
