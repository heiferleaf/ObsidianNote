## Redis

### Redis 的 epoll 多路复用机制
> 前提：linux 中，认为 anything is file，通过文件描述符 FD，来表示数据

在内核态，通过一个红黑树，监控所有的 fd和对应的数据，查找效率高。话有一个链表，存放所有准备好的数据的 FD。
当数据准备好后，通过中断，用户态-》内核态，在链表中获得所有准备好的fd，再根据fd找到数据，复制到内核态，进行处理。

Redis的Reactor模型，是说说线程不会等到一个IO请求准备好数据，去获取解析数据，而是继续处理其他请求的连接。对于数据的解析处理，是通过epoll触发中断，执行回调函数，执行的。Reactor模型，就是IO多路复用，使得CPU处理任务的特点表现。

**主从复制**只解决了数据冗余和读写分离，Master 挂了没人接管，是最基础的形态，生产环境基本不单独使用。

**哨兵模式**在主从基础上加了一个独立的监控集群（Sentinel），专门负责盯着 Redis 节点的心跳、投票决策、自动切换 Master。它解决了高可用，但数据仍然是全量存在每个节点，内存天花板没变。

**Cluster**把数据切成 16384 个 slot 分散到多个分片，每个分片是一组独立的主从，内置故障转移不再需要 Sentinel。核心收益是容量可以随分片数线性增长，代价是跨 slot 的操作（比如同一个事务里操作两个不同分片的 key）受到限制。

一个容易混淆的点：**哨兵和 Cluster 都有"故障转移"能力，但机制完全不同**。哨兵是外部裁判——一个独立进程集群投票后强制切换；Cluster 是内部自治——分片内的节点通过 Gossip 协议互相通报状态，自己选举新 Master。

### Redis 的读写模式

1. 旁路缓存模式
	1. 写操作：先更新数据库，然后删除缓存，缓存和数据库存在短暂的不一致性
2. 缓存代理
	1. 读操作：数据没有命中缓存，缓存会自行加载数据库数据
	2. 写操作：先更新缓存，缓存同步更新数据库。保证缓存和数据库强一致（作为一个事务执行）
3. 异步写回
	1. 写操作：先更新缓存，异步写回数据库
		- 最终一致性：因为是异步写回
		- 缓存宕机，数据可能丢失
		- 性能高，因为写操作是写缓存，面向内存，比较快
		- 写操作可以合并：多次更新可以合并为一次数据库写操作

> 后两种方案都是由redis来保证DB的数据更新，负责一致性

### 读写分离

> 这里说到读写分离，主要指数据库的主从架构
> 当然 Redis 缓存也可以设计成主从架构，也有读写分离，和一致性的问题

读写分离的使用原因：
- 有的业务场景读多写少，如果一个数据库，读的压力比较大，称为业务的响应时间瓶颈
- 避免了读写操作的锁争抢
- 数据有备份冗余，安全

架构方式：主从架构
- 一主多从
- 主从级联
- 多主多从
