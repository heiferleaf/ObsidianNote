> 分布式软件，是通过增大处理机器数量，解决大规模的问题。云技术可以很便捷的提供机器规模扩大的需求

## 什么是分布式软件

软件的各个组件分布在 ==可以通过网络通信互相连接的== 物理机器上。通过网络通信实现服务的调用，完成软件功能。
> 原来的部署，都是全部在一台服务器中，服务之间的调用，比如redis，mysql，都可以在后端中，通过localhost访问到，或者使用容器名的方式访问得到。
> 服务之间的协作，可以说是通过内存进行数据通信的

而当服务的规模增大，一台机子很难承担，于是可以横向的拓展，让多台机器运行同一种服务，一起承担业务规模。

这些也蕴含了分布式软件的设计目标：
1. 高并发：让更多的用户可以一起访问
2. 高性能：服务跑的快
3. 高可用：一台机器挂了，还有9台可以跑

分布式也会带来一些副作用：
- 可靠性：网络被切断，消息可能就会丢失、乱序
- 一致性：A节点上修改了余额，B结点看到的余额要马上一样，不然访问到B结点中，就是错误数据
> 分布式是为了解决单机无法承载的规模化问题，和单机无法提供的容错性问题
  不能为了分布式而分布式，否则系统将变得复杂而脆弱

## 概念区分

| 术语    | 范畴     | 定义                                                                                          | 解决的关键问题              |
| ----- | ------ | ------------------------------------------------------------------------------------------- | -------------------- |
| 分布式计算 | 计算范式   | 将一个巨大的计算任务拆分成多个小的子任务，分配给网络中的多台计算机同时处理，最后汇总结果。如MapReduce、Spark等分布式计算框架，DeepSpeed、Ray等分布式训推框架 | 任务切分、并行处理等计算优化问题     |
| 分布式系统 | 软硬件集合体 | 硬件或软件组件分布在不同的网络计算机上，彼此之间仅仅通过消息传递进行通信和协调                                                     | 节点、网络、局部失效、逻辑时钟等问题   |
| 分布式架构 | 系统设计   | 分布式系统内部各组件如何组织、如何交互的结构化设计方案。如主从架构、对等架构、微服务架构、无服务器架构 (Serverless)等                           | 拓扑结构、通信协议解耦、服务治理等问题  |
| 分布式软件 | 软件产品   | 运行在分布式系统之上的应用软件（如 Google 搜索、微信、淘宝），由多个协同工作的组件（如前端、后端服务、数据库集群）组成的软件集合                        | 高并发、高性能、高可用、高可靠（一致性） |

## **CAP 定理**

- C：一致性，所有结点看到的数据一摸一样
- A：可用性，即时有的结点坏了，系统还是能成功响应
- P：分区容错性，网络中断或者结点失联时，系统依然能正常工作

> **P 在分布式系统中是必须保留的**（网络故障不可避免），所以实际上是在 **CP 和 AP 之间二选一**。

**CP（保一致性，牺牲可用性）**

```
节点A 和 节点B 网络断了
→ 宁可拒绝请求，也不返回可能不一致的数据
→ 适合：金融、订单、库存等数据强一致场景
```

**AP（保可用性，牺牲一致性）**

```
节点A 和 节点B 网络断了
→ 继续响应请求，但两个节点数据可能短暂不一样
→ 适合：服务注册发现、点赞数、浏览量等允许短暂不一致的场景
```

## PACELC定理

> 这个定理是对 CAP 的补充，补充了在正常状态下，对系统特性的一个选择 

P-A-C | E-L-C
- `P-A-C` ： 在分区出现时，在 A C 之间选择
- `E-L-C` ： 否则(E else) 在正常状态，在 L（延迟）和 C（一致性） 之间权衡


| 策略类型  | 典型代表              | 核心特点                       |
| ----- | ----------------- | -------------------------- |
| PA/EL | Dynamo, Cassandra | 出现分区选可用；正常时选低延迟（牺牲强一致性）。   |
| PC/EC | BigTable, HBase   | 始终追求强一致性。分区时宁可不可用；正常时宁可慢点。 |
| PC/EL | MongoDB (默认)      | 分区时选一致性；但正常情况下为了性能选低延迟。    |
| PA/EC | VoltDB            | 分区时保可用；但正常情况下维持强一致（开销较大）。  |
> [!NOTE] 要点
> 1. 延迟是分布式系统的第一性能指标
> 	强一致性往往是高延迟，所以要进行取舍
> 2. 可调一致性强度，是分布式平台的标配

## BASE 理论

> BASE 理论是对 ACID （强一致性） 的一种"放松版"替代，专门为分布式/高可用系统设计的。

全称是：

- **B**asically Available（基本可用）
- **S**oft State（软状态）
- **E**ventually Consistent（最终一致性）

### 逐条解释

**Basically Available（基本可用）** 系统出现故障时，不是完全不可用，而是允许降级服务。比如电商大促时搜索结果可能不是最新的，购物车可能有延迟，但系统不会直接崩溃拒绝所有请求。"基本能用"比"要么全好要么全挂"更实际。

**Soft State（软状态）** 系统的状态不需要时刻保持一致，允许中间存在过渡状态。数据在不同节点之间同步是有时间差的，这段时间里各节点看到的数据可能不一样——这是被允许的，不被视为错误。

**Eventually Consistent（最终一致性）** 不保证任意时刻数据一致，但保证"只要没有新的写入，经过一段时间后，所有节点会收敛到同一个状态"。DNS 就是经典例子：你修改了域名解析，不会立刻全球生效，但几小时后所有节点都会同步到新值。

### 和 ACID 的对比

|      | ACID             | BASE                     |
| ---- | ---------------- | ------------------------ |
| 一致性  | 强一致，事务内保证        | 最终一致，允许中间态               |
| 可用性  | 为了一致性可以拒绝请求      | 优先保证可用，放宽一致性             |
| 适用场景 | 银行转账、订单支付        | 社交feed、购物车、点赞数           |
| 典型系统 | MySQL、PostgreSQL | Cassandra、DynamoDB、Redis |

### 和 CAP 的关系

BASE 本质上是选择了 CAP 中 **AP** 方向之后的具体实践指导。CAP 告诉你"三选二"，BASE 告诉你选了 AP 之后"怎么活"——接受软状态，设计最终一致性的同步机制，系统降级而不是崩溃。

### 实际中怎么用

并不是所有业务都适合 BASE。常见做法是**混用**：

- 支付、库存扣减 → 用 ACID（强一致）
- 用户评论、点赞、浏览历史 → 用 BASE（最终一致就够了）

判断标准就是问：**这个数据短暂不一致，业务上能接受吗？** 能接受就可以用 BASE，换来更高的可用性和性能。

## 分布式系统的性能指标

> 和之前的分布式系统 设计目标和存在问题挂钩

| 概念  | 核心定义           | 关键指标                                                    | 通俗理解  |
| --- | -------------- | ------------------------------------------------------- | ----- |
| 高并发 | 系统能同时处理大量请求的能力 | QPS、TPS                                                 | “扛得住” |
| 高性能 | 单个请求处理速度快      | 响应时间（RT）<br><br>延迟（p99）                                 | “跑得快” |
| 高可用 | 系统持续提供服务的能力    | SLA（如99.99% = 年停机52分钟）<br><br>MTBF（平均无故障时间）MTTR（平均恢复时间） | “不宕机” |
| 高可靠 | 数据不丢失、计算结果正确   | 持久性、错误率                                                 | “不出错” |
| 一致性 | 多个副本之间数据的关系    | 一致性模型（强、弱、最终等）                                          | “不矛盾” |

1. 高并发
	- 指标
		- QPS：每秒查询数
		- TPS：每秒数据数
	- 解决方案
		- 增加服务器的集群
		- 使用缓存，通过Redis缓存减轻数据库负担
		- 异步处理，通过消息队列MQ，实现削峰填谷
2. 高性能
	- 解决方案
		- 算法优化
		- 代码优化：并发编程，锁的粒度控制
		- 硬件优化：网络带宽，等等
3. 高可用
	- 解决方案
		- 多机房容灾：异地多活架构。
		- 故障自动切换：Keepalived + VIP 或 Zookeeper 选主。
		- 服务降级/熔断：在极端情况下关闭非核心业务，保住主流程
4. 高可靠
	- 解决方案
		- 故障隔离：故障范围最小化，防止雪崩效应；舱壁隔离、熔断、限流
		- 故障转移：故障发生时无需人工介入，系统自愈	健康检查 + 选主/切换
		- 容错与共识：部分节点故障不影响整体一致性，Raft、Paxos、2F+1模型
		- 可观测性：实时感知系统健康状态。监控、告警、链路追踪
5. 一致性
	```
	强一致性 ──────────────────────── 最终一致性
	（最严格）                        （最宽松）
	线性一致 > 顺序一致 > 因果一致 > 读写一致 > 会话一致性 > 单调读 > 单调写 > 最终一致
	```


> [!NOTE] 补充结语
> 对于高性能、高并发、高可用，不需要刻意地去做区分，高并发、高可用往往就是实现高性能的关键，三者的核心就是应用大规模的数据请求
