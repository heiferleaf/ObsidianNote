## 异步逻辑

客户端发送一个请求，网关服务器有两种做法

- ==同步阻塞==：接收到请求之后，接受请求的线程，等待响应结果，等到结果之后，进行处理，然后在处理其他线程。
- ==异步非阻塞==：接收请求的线程，在调用下游服务的时候，注册一个回调函数，然后接着处理其他请求，当原来的请求得到结果后，找一个空闲线程，执行注册的回调函数，对得到的结果进行处理

> 非阻塞的方式，本质上是通过编程的复杂度，换取了CPU的吞吐量


在考虑消息对列的异步处理，原本需要处理事件 A B C。
现在，只需要处理核心事件A，把消息发到对列，然后响应用户成功，B和C交给消息的消费者进行处理。

1. **提高了响应速度**
2. **代码解耦**
3. **削峰填谷**：原来大规模请求一下子带来，服务直接宕机，现在把请求都放到消息对列，然后响应正在处理，消息对列可以保证服务按照稳定的处理速度进行。==MQ的写磁盘速度很快，所以MQ不会崩==

==牺牲了一致性==，实现最终一致性。比如使用消息对列修改缓存或者删除缓存，可能会有读取任务在没有消费任务没有执行之前进行，导致读取到脏数据。**不过可以通过消息重试机制和死信队列保证消息被可靠的处理**

## 消息对列引入

消息对列的工作流
![[第四讲 高并发设计(3).pdf#page=6&rect=483,18,933,427|第四讲 高并发设计(3), p.6]]

1. 消息持久化存储到磁盘，防止宕机丢置
	- 消息写入磁盘文件，通过 append-only 顺序写的方式，写性能高
	- 内存中维护索引或者缓存，读取速度快
2. 消息确认机制
	消费者响应 ACK， 消息才从队列中删除
	生产者收到 ACK， 才认为消息发送成功
3. 消息可靠性
	- 持久化
	- 确认
	- 死信队列，消息失败重新，失败次数多了，放入死信队列，人工处理
4. 消息的顺序
	- Kafka：分区内有序
	- RabbitMQ：单一队列，所以有序
	- RocketMQ：两种顺序保障
		- 分区顺序：类似Kafka
		- 全局顺序：设置一个主题的对列数量为1，一种主题的消息在同一个队列中，消费者单线程处理，生产者单线程发送。牺牲吞吐量，保证顺序


## Kafka

### 基本概念

 - Broker：Kafka服务器实例 
 - Producer：消息生产者 
 - Consumer：消息消费者 
 - Topic：主题。生产者可以通过主题推送给多个消费者 
 - Partition：主题的分区。相当于银行的多个窗口，每个Partition维护一个队列。 
 - Replicas: 主题的副本数，支持高可用。 
 - Consumer Group：消费者的逻辑分组，实现消费者负载均衡和容错。 topic的一个消息由组内的任一个消费者处理

### 消息绑定逻辑

对于消费方：为了保证高并发情况下的消息来得及消费，消费者往往是多实例部署，相同的消费者组成一个消费者组。
对于Kafka服务器：为了让一个Topic的消息，可以被并发读取，将消息进行分区，每一个分区维护一个消息对列，不同的分组在不同的磁盘上，因此可以并发读取消息。
实际的读取中，一个消费者组绑定一个Topic内的分区，一个分区只能给消费者组内的一个消费者负责，如果消费者挂了，交给另外的来。

### 消息持久化存储逻辑

消息队列的一大特点是消息的持久化存储，防止MQ服务器宕机导致的消息丢失。所以也有必要了解消息在磁盘中的文件组织方式。
每一个Broker有一个log.dirs，这是一个文件夹，下面包含了所有的分区文件夹 `Topic-分区号`。在每一个分区文件夹中，都包含了很多段Segment，段体现为文件组：
- .log：包含消息的元信息和消息的数据
- .index：记录消息序号和对应物理偏移位置的序偶信息
- .timeindex：（可能不叫这个文件名，含义是时间戳文件）
==写的快==：.log 文件在生产者生产消息后，是顺序写入分区文件夹的，所以写磁盘的时候，磁针不需要一直移动，效率高；==找的快==：.index文件支持消费者消费时，快速定位消息，并且索引文件是稀疏的（不会每一个消息都有索引），可以全部放到内存中。==发的快==：通过零拷贝技术，将数据从磁盘直接写入内存缓冲区，然后写入Socket缓冲区。
为了防止磁盘出错，Topic会有副本。平时的读写都针对leader，会自动和副本follower同步。


### 当前处理消息的定位

Kafka内部有一个Topic，叫做 \_\_consumer_offsets，存放的数据如下

| **消费者组 ID** | **Topic 名称** | **分区号** | **读到哪了 (Offset)** |
| ----------- | ------------ | ------- | ----------------- |
| 短信通知组       | 医疗挂号         | P1      | 5                 |
当有消费者实例挂了，就需要让同一个消费者组的其他消费者来接替处理。
为了实现这个机制，消费者在消费的时候，需要
1. 拉取消息
2. 处理消息
3. 提交记录：消费组已经处理完Topic-x分组的第y个记录了
4. Kafka记录



## 消息对列的问题处理

### 问题一：消息重试导致失序

- 当消费者处理消息A失败，重试机制会将其重新入队，所以原本在A后的消息B，就有可能先执行。
- 解决方案：通过在业务设计的数据库中引入版本号/状态机，来让后执行的A失效。
  比如A（下单）B（取消订单），当B执行之后，订单数据行的status=2，到业务A，先检查status\==0，成立才执行
- 解决方案2：通过重试队列/死信队列的机制，阻塞后续消息

### 问题二：消费者并发导致乱序

保证一个业务的消息串行执行，将业务的ID号作为消息的Key，Kafka保证相同的Key，会进入相同的分区，分区会由一个线程执行

### 问题三：生产者因网络问题重复发送相同消息

生产者持有一个PID，发送的消息持有一个序列号，消息队列中的消息，需要进行幂等性校验（不接受重复的消息）
在消费端，进行去重逻辑

### 问题四：消息积压

- 一条消息处理慢，失败，导致消息积压
- 解决方案：通过非阻塞的重试队列，失败太多进入死信队列

# 分库分表

为什么需要：
- 一个库的数据量太大，即使有 SQL 优化和索引优化，性能还是不够好
- 一个库的SQL连接数有限，高并发写的时候，sql连接不够用

分库目的 
- 提升数据库的存储容量、支撑更多的数据库连接，大幅度提升数据读写性能 （磁盘 IO 指标） 
分表目的 
- 减少单表的存储容量，提升数据的读写效率。分表并没有将拆分后的表分散到不同物理机器上，仍然在竞争同一个物理机的 CPU、内存、网络 IO 等资源，
> 本质都是空间换时间

分库和分表都有水平分和垂直分的区别：
- 垂直分库：按照业务领域，进行库的划分
- 水平分库：把数据按照指定的规则，划分到不同的数据库中
- 垂直分表：在同一个数据库内，把大表拆为小表
- 水平分表：在同一个数据库中，把大数据量的表，按照制定规则，把数据映射到多个结构相同的表中

其中水平分库和水平分表，都需要进行数据的分片，把数据映射到不同的库或者表中。数据分片有几个原则
- 分片的值尽量离散，**防止数据倾斜**
- 业务相关性：**大部分的查询都需要能携带分片键**，不然需要扫描所有的库或者表
- 稳定性：**分片的值应该稳定**，不能经常变换

具体的分片策略
1. 哈希取模：hashCode(key) % 分片总数
2. 范围分片：按照分片键值所在的区间进行分片，比如按照日志的记录时间区间分片，按照物理地区分片。**会导致数据可能倾斜，以及热点数据可能都往一个表里写**
3. 一致性哈希：把数据库映射到一个环上的多个点（虚拟节点），数据也映射到环上。数据的存放位置，就是顺时针最近找到的结点。**适合结点变化频繁的场景，对数据的移动影响很小**
4. 基因算法：本质上就是根据分片键信息，加上事件戳、序列号等唯一性值，得到一个唯一的二进制串（基因），这样可以通过基因，得到数据在哪一个分片中

分库分表的代码使用：
1. 客户端模式：在代码中，直接连接所有分库分表，执行SQL，对得到结果进行聚合。代码和数据库耦合度高。==ShardingSphere-JDBC==
2. 代理模式：增加中间层，控制对数据库的分库分表的操作，对于业务逻辑来说，不需要直接连接数据库。==MyCat==

分库分表的技术挑战

| 挑战类别            | 具体问题描述                                     | 可采取的解决方案                       |
| --------------- | ------------------------------------------ | ------------------------------ |
| ==分布式事务==       | 一次业务操作涉及多个数据库，无法使用单库事务保证ACID，可能出现数据不一致。    | XA协议（2PC）、TCC模式、Seata AT、本地消息表 |
| ==跨库关联查询==      | 需要JOIN的数据分布在不同的数据库或分片中，无法直接执行数据库JOIN。      | 字段冗余、全局表、应用层组装、搜索引擎            |
| ==分页排序==        | 跨分片执行LIMIT M,N时，需要从各分片拉取大量数据再合并排序，深分页性能极差。 | 禁止深分页、游标分页、使用搜索引擎              |
| ==**分布式ID生成**== | 数据库自增ID在分库分表后无法保证全局唯一，且无法直接用于路由。           | 雪花算法、基因分片ID、Redis原子递增          |
| 数据倾斜            | 分片键选择不当导致某些分片数据量远大于其他分片，形成热点，影响整体性能。       | 重新选择分片键、复合分片键、虚拟节点、热点数据隔离      |
| 扩容与数据迁移         | 分片数不足以支撑未来数据量时需扩容，但取模分片下扩容会导致大量数据迁移。       | 预留分片、一致性哈希、双写迁移、停服迁移           |
| 跨库聚合统计          | 需要执行COUNT、SUM、AVG等聚合函数时，必须从全部分片拉取数据再计算。    | 应用层聚合、预计算、搜索引擎/OLAP库、业务降级      |


实际使用原则
1. 优先考虑优化数据库的索引、缓存、读写分离，能不拆分就不拆分
2. 真要拆分，选择适合分片用的键和映射到库表的策略
3. 使用分布式事务，解决分库分表带来的事务问题，通过ES中间件，解决分库之后的复杂查询问题
