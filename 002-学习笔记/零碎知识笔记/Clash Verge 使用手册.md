## 三大代理模式（规则）

|特性|规则模式 (Rule) ✅ (强烈推荐)|全局模式 (Global)|直连模式 (Direct)|
|---|---|---|---|
|**运作方式**|严格遵循你配置文件中的分流规则[](https://clash-verge.org/zh-CN/tutorial#proxy-recommend)。它会自动将国内网站和国外网站的流量，分别导向直连和代理节点。|**忽略所有规则**，将所有网络流量（无论国内还是国外）都强制发送到你所选择的代理节点上[](https://clash-verge.org/zh-CN/tutorial#proxy-recommend)。|**忽略所有规则和代理节点**，所有网络流量都不经过代理，直接通过你的本地网络连接[](https://www.clash-verge.com.cn/2187.html)。|
|**适用场景**|**日常使用的最优选择**。平衡了访问速度和规则可控性[](https://www.clash-verge.com.cn/2187.html)。|**临时性、全流量的代理需求**。比如访问某个严格限制IP的地区服务，或排查规则问题时临时使用[](https://www.clash-verge.com.cn/2187.html)。|**暂时完全绕过代理**。比如在某个国内网站遇到访问问题，需要排查是否是代理引起，或希望暂时使用本地网络时[](https://www.clash-verge.com.cn/2187.html)。|
|**优缺点**|**优点**：自动分流，国内访问快且省流量，体验最佳[](https://www.intoep.com/othen/68393.html#comments)。  <br>**缺点**：高度依赖规则文件的质量，规则不全时可能导致网站访问异常[](https://www.intoep.com/othen/68393.html#comments)。|**优点**：行为简单直接，可以快速确认某个网站能否通过代理访问[](https://www.intoep.com/othen/68393.html#comments)。  <br>**缺点**：浪费流量，拖慢国内网站访问速度，并加重节点压力[](https://www.intoep.com/othen/68393.html#comments)。|**优点**：简单直接地禁用代理功能。  <br>**缺点**：完全失去代理能力。|

|**启动门槛**|无需额外权限。|通常需要安装虚拟网卡驱动，有时需要管理员权限。|
1. 下面的监听端口协议，都是针对==系统代理==而言的。应用程序如果支持代理，就可以设置代理的协议和监听的端口，对应的 Clash Verge 内核设置监听的代理端口是 Mixed、Http、Socks等
2. 有的应用程序不支持代理，或者虽然走代理，然是对请求的域名查询，还是走的本地 DNS 服务器，留下了访问的记录，请求的数据最后由代理服务器发送。这种情况都叫做隐私泄露。因为原意是不想泄露 DNS 访问记录
3. 浏览器都支持被系统代理
4. TUN 拦截网络层的流量，所以不依赖请求的协议是否是内核设置的方式


## 使用总结

1. 对于日常访问浏览器中网址：系统代理 + 规则模式
2. 对于一些特定软件的使用
	1. 使用系统代理，需要额外配置软件中的代理内核监听的端口和协议
	2. 使用 TUN 模式
3. 对于一些国外网站：使用系统代理的前提下，还需要配置使用的代理的结点
4. 在流量的开销比较上，规则方式 优于 全局方式；系统代理优于TUN模式


## 故障排查

当使用 TUN 模式出错的时候，多想想是不是以下的问题

### 一、TUN 模式工作原理回顾

TUN 模式创建一个虚拟网卡（如 `clash0` 或 `wintun`），操作系统将所有符合条件的流量（默认是全部流量）路由到这张虚拟网卡。Clash 内核从虚拟网卡读取原始 IP 包，然后根据规则处理（直连、拒绝或发往代理节点），处理后再从物理网卡发出。

这个过程中，**DNS 解析通常由 Clash 的 DNS 模块接管**（推荐 `fake-ip` 模式），从而避免 DNS 泄露并实现域名分流。

---

### 二、为什么开启 TUN 模式后服务访问不了？四大原因详解

### 1️⃣ 防火墙拦截

**原理**：  
Windows Defender 防火墙或第三方杀毒软件会监控网络流量。当 Clash Verge 创建虚拟网卡后，`clash-meta.exe`（核心进程）需要从该虚拟网卡读取数据、向物理网卡写入数据。防火墙如果未信任该进程，可能会**阻止虚拟网卡与物理网卡之间的数据交换**，或者阻止 Clash 进程接收来自虚拟网卡的流量。

**表现**：  
- 所有流量都无法通过 TUN 模式，任何网站都打不开。
- 或者部分协议（如 UDP）被阻断，导致 DNS 查询失败。

**解决方法：**  
- **放行程序**：在 Windows Defender 防火墙中，添加 `clash-meta.exe` 和 `clash-verge.exe` 的入站/出站规则，允许所有通信。  
- **更彻底的方式**：临时关闭防火墙测试，如果恢复正常，则确定是防火墙问题，再精细配置规则。  
- **注意**：某些杀毒软件（如 McAfee、360）也有网络过滤功能，同样需要添加信任。

---

### 2️⃣ 路由表冲突

**原理**：  
操作系统通过路由表决定某个目标 IP 应该从哪张网卡发出。TUN 模式会修改路由表，添加一条默认路由（`0.0.0.0/0`）指向虚拟网卡，或者添加一组绕过规则。如果你同时运行了其他 VPN（如 OpenVPN、WireGuard、公司 VPN 客户端），它们也会修改路由表。  
冲突的表现是**路由环路**：  
- 数据包从应用发出 → 进入虚拟网卡 → Clash 处理后尝试从物理网卡发出 → 但物理网卡的默认路由又被指向了虚拟网卡（死循环）。  
- 或者，某些网段被多个路由规则覆盖，导致数据包被丢弃。

**表现**：  
- 访问任何 IP 都超时，或者只有本地局域网通，外网不通。  
- 开启 TUN 后，原有的 VPN 断开或工作异常。

**解决方法：**  
- **手动指定物理网卡名称**：在 Clash Verge 的 TUN 设置中，将 `interface-name` 设置为你真正的物理网卡名称（如 `以太网` 或 `WLAN`），而不是 `"auto"`。这样 Clash 在添加路由时会明确出口，减少冲突。  
- **关闭其他 VPN**：在开启 TUN 前，断开其他 VPN 连接。如果必须同时使用，需要研究路由表并手动调整（高级操作，不推荐普通用户）。  
- **重启 Clash 并观察**：以管理员身份运行 Clash Verge，确保它有权修改路由表。

---

### 3️⃣ DNS 污染 / 配置错误

**原理**：  
TUN 模式下，Clash 默认会接管 DNS 请求。如果你没有开启 `fake-ip` 模式，或者使用了不恰当的 `nameserver` 配置，可能导致：  
- **DNS 解析超时**：Clash 尝试通过代理节点查询 DNS，但代理节点本身不通或过慢。  
- **DNS 返回错误结果**：比如国内 DNS 服务器（如 114.114.114.114）对国外域名返回污染 IP，Clash 使用了这个结果，导致代理也无法正确连接。  
- **UDP 阻断**：某些网络环境下，UDP 53 包被运营商拦截，如果 Clash 尝试用 UDP 直连 DNS 服务器，会失败。

**表现**：  
- 浏览器提示“无法解析域名”。  
- 能 ping 通 IP，但用域名访问失败。  
- 部分国外网站打不开，国内正常。

**解决方法：**  
- **启用“增强模式 (fake-ip)”**：这是 Clash 最重要的 DNS 改进。  
  - 原理：Clash 为每个域名分配一个虚拟 IP（如 `198.18.x.x`），并返回给应用程序。应用程序访问这个虚拟 IP 时，Clash 根据虚拟 IP 反查域名，再根据规则处理。  
  - 好处：不依赖 UDP 53，完全避免 DNS 污染，且支持按域名分流。  
  - 操作：在 Clash Verge 的 DNS 设置中，勾选“启用增强模式”，选择 `fake-ip`。  
- **配置可靠的 nameserver**：使用加密 DNS，例如 `tls://1.1.1.1` 或 `https://dns.google/dns-query`，并确保这些 DNS 服务器可以通过代理访问（或者直连可用）。  
- **检查 fallback 设置**：避免 fallback 中的 DNS 服务器返回污染结果。

---

### 4️⃣ 规则配置问题

**原理**：  
TUN 模式接管所有流量后，**最终每个连接走代理还是直连，完全由规则决定**。如果你的规则文件存在以下问题，就会导致服务访问异常：  
- **缺少国内网站直连规则**：国内网站（如百度、知乎）被错误地匹配到了代理节点，但代理节点在国外且速度慢，甚至被国内 CDN 拒绝 → 国内网站打不开。  
- **国外网站被直连**：你想访问的国外网站（如 Google）匹配到了 `DIRECT` 规则，但你的本地网络无法直接连接该网站 → 国外网站打不开。  
- **规则顺序错误**：Clash 匹配规则是顺序匹配，若一条宽泛的规则（如 `DOMAIN-SUFFIX,com,DIRECT`）在前面，导致具体域名无法被后面的代理规则匹配。  
- **规则不完整**：某些应用使用非常规域名或 IP，没有对应规则，最终走了默认策略（可能 `DIRECT` 或 `Proxy`，取决于你的配置）。

**表现**：  
- 国内正常、国外打不开 → 可能是国外域名被直连了。  
- 国外正常、国内打不开 → 可能是国内域名被代理了。  
- 只有特定应用（如 VS Code 插件商店）打不开 → 缺少该服务的域名规则。

**解决方法：**  
- **切换到“规则模式 (Rule)”**：确保 Clash Verge 的代理模式是 `Rule`，而不是 `Global` 或 `Direct`。  
- **检查配置文件**：确认包含类似以下规则（通常是订阅自带）：  
  ```
  GEOIP,CN,DIRECT
  MATCH,Proxy
  ```
  并确保 `GEOIP,CN` 位于前面，这样国内 IP 直连。  
- **更新规则订阅**：使用知名提供商的规则集（如 Loyalsoldier/v2ray-rules-dat）。  
- **手动添加缺失规则**：通过日志查看某个域名走了错误策略，然后添加针对性规则。  
- **测试**：临时将所有流量设为 `Proxy` 或 `DIRECT`，看问题是否消失，从而定位是代理节点问题还是规则问题。

---

### 三、其他可能原因及通用排查步骤

除了上述四点，还可能存在：
- **虚拟网卡驱动损坏**：以管理员身份重新安装虚拟网卡驱动（Clash Verge 设置中通常有“安装 TUN 驱动”按钮）。
- **IPv6 干扰**：关闭 IPv6 或确保规则正确处理。
- **代理节点本身故障**：切换其他节点测试。
- **Clash 内核版本问题**：更新到最新版。

### 推荐排查顺序（新手友好）：

1. **先尝试简单修复**：重启 Clash Verge，以管理员身份运行。  
2. **切换代理模式**：将模式从 `Rule` 改为 `Global`（确认代理节点是否正常工作），再改回 `Rule`。  
3. **检查 DNS 设置**：开启 `fake-ip`，并设置可靠的 nameserver（如 `https://dns.google/dns-query`）。  
4. **关闭防火墙**：临时关闭 Windows Defender 防火墙，看是否恢复。  
5. **查看日志**：在 Clash Verge 的日志窗口中，观察访问失败时的日志输出（是 DNS 错误、连接超时、还是规则匹配到了 `REJECT`）。  
6. **重置 TUN**：卸载虚拟网卡驱动，重新以管理员身份安装。  
7. **检查路由表**：在命令行执行 `route print`，看是否有明显冲突（需要一定网络知识）。
