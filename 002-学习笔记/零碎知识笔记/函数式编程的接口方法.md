在 Java 的 `java.util.function` 包中，为了适配函数式编程，官方提供了 **43 个** 核心函数式接口。虽然数量多，但它们其实都是由 **4 个基本元接口** 演变而来的。

理解了这 4 个“元接口”，你就能秒懂剩下的 39 个。

---

### 1. 四大核心元接口

这是所有函数式编程的基石，你可以根据**“入参”**和**“出参”**的情况来记忆：

|**接口名称**|**核心方法**|**入参**|**出参**|**语义**|**常见场景**|
|---|---|---|---|---|---|
|**`Function<T, R>`**|`R apply(T t)`|有 (1个)|有|**转换**|`map()`, `Collectors.toMap()`|
|**`Consumer<T>`**|`void accept(T t)`|有 (1个)|无|**消费**|`forEach()`, `peek()`|
|**`Predicate<T>`**|`boolean test(T t)`|有 (1个)|有 (布尔)|**判断**|`filter()`|
|**`Supplier<T>`**|`T get()`|无|有|**生产**|`Optional.orElseGet()`, 创建对象|

---

### 2. 常用演变变体

为了应对不同的参数数量和性能需求，官方对上述 4 个接口做了扩展：

#### A. 增加参数数量 (Bi- 系列)

如果你需要处理两个参数，就找带 `Bi` 前缀的：

- **`BiFunction<T, U, R>`**: 接收两个参数，返回一个结果。
    
- **`BiConsumer<T, U>`**: 接收两个参数，不返回结果（如 `Map.forEach((k, v) -> ...)`）。
    
- **`BiPredicate<T, U>`**: 接收两个参数，返回布尔值。
    

#### B. 针对基本类型 (类型特化)

为了避免基本类型（如 `int`）频繁装箱/拆箱带来的性能损耗，Java 提供了特化版本：

- **`IntPredicate` / `LongPredicate`**: 专门处理 `int` 或 `long` 的判断，入参直接是基本类型。
    
- **`ToIntFunction<T>`**: 接收对象，返回 `int`（例如 `String::length`）。
    

#### C. 入参与出参类型一致 (Operator 系列)

这是 `Function` 的特殊情况，当你转换后的类型和输入类型完全一样时：

- **`UnaryOperator<T>`**: 一元算子（继承自 `Function<T, T>`）。例如：`s -> s.toUpperCase()`。
    
- **`BinaryOperator<T>`**: 二元算子（继承自 `BiFunction<T, T, T>`）。例如：`sum = (a, b) -> a + b`。
    

---

### 3. 如何快速选择接口？

当你自己写方法需要接收 Lambda 表达式时，按以下逻辑判断：

1. **它要返回东西吗？**
    
    - 不返回 ➔ **`Consumer`**
        
    - 返回布尔值 ➔ **`Predicate`**
        
    - 返回对象 ➔ **`Function`** (有入参) 或 **`Supplier`** (无入参)
        
2. **它需要几个入参？**
    
    - 1 个 ➔ 基础款
        
    - 2 个 ➔ **`Bi-`** 变体
        
3. **入参和出参类型一样吗？**
    
    - 一样 ➔ **`Operator`** 变体

### 4. 为什么会有 `Function.identity()`？

你刚才提到的 Collectors.toMap(Function.identity(), ...) 里的 identity() 实际上是 Function 接口的一个静态方法。

它的源码非常简单：

Java

```
static <T> Function<T, T> identity() {
    return t -> t;
}
```

它就是一个“原样返回”的转换器。在 `toMap` 中使用它，意思就是“直接把当前流里的对象当作 Map 的 Key”。
