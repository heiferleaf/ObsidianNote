# 什么是跨域问题

在前端开发和网络编程中，跨域问题一般指的是：浏览器为了安全，限制了脚本语言比如python、js对于跨源的http/https请求。

这种安全机制也叫做同源策略。

## 什么是同源的

两个请求的 url 只有
- 协议相同 : http / https
- 域名相同
- 端口相同
，才会被认为是同源的请求。

- 一个网址A，认为 url 为 【协议A】://【域名A】:【端口A】
- 一个网址B，认为 url 为 【协议B】://【域名B】:【端口B】
我们在网站A中，设置跳转连接，==脚本语言的行为是打开某一网站==，这个不算做跨域，是正常的用户行为。
我们在网站A中，发出 【协议B】://【域名B】:【端口B】/path。这个行为是通过 AJAX 访问别的网站中的后端API，才是真正放出 http 请求，希望数据通信的跨域行为。
> 正常的地址栏访问，会离开当前网页，到新的网页
> 恶意的跨域访问，是在一个网页里，去访问别的网站的接口，获取别的网站内的信息。
## 同源策略

同源策略，通常应对CSRF(跨站请求伪造)等安全攻击问题。
当我们在浏览器上浏览网站，留下了自己的 Cookie，如果恶意网站可以劫持获取 Cookie，那么就可以通过 Cookie 发出请求，让后端服务的身份验证、用户鉴权防护失效。

具体的防范手段：
1. 对 AJAX 的请求（Fetch、XMLHttpRequest）
2. 无法读取不同源的 Cookie、LocalStorage、IndexedDB
3. 无法操作不同源的 DOM

> 同源策略的方案，并不是说前端无法发出请求，而是指浏览器收到响应结果之后，根据同源策略判断能否给脚本语言返回结果

> 脚本语言可以通过 document.cookie("") 来读取当前域名下的 Cookie，如果检测到是读取不同域名的 Cookie 直接拒绝。
> 此外，浏览器会在发送一个请求的时候，自动附上同一个域下的Cookie
> 如果后端的Cookie设置了HttpOnly，那么就算是同一个域的也读不到，只能用来发送请求的时候附带消息，常用于保存绝对不能泄露的信息，比如双 Token 中的 RefreshToken。

## 跨域常见的解决方案

有时候我们确实需要允许跨域的访问，比如自己写的前后端分离架构，前端访问后端，浏览器运行的本地的代码，占用的端口和本地后端服务的端口肯定是不一样，这时候，前端发送的请求，就是跨源的。

1. CORS（跨源资源共享）：后端在响应体中手动添加 Access-Control-Allow-Origin: \*（或指定的域名）
2. 代理服务器：同源策略只存在于浏览器发出的请求的限制，可以设置代理服务器，前端发送给代理，代理实际发送请求，得到响应，最后把结果还给前端
3. WebSocket：ws协议不会限制跨域行为

以 SpringBoot 为例子
```java
import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

@Configuration
public class CorsConfig implements WebMvcConfigurer {

    @Override
    public void addCorsMappings(CorsRegistry registry) {
        // 添加映射路径，“/**”表示对所有的路径实行全局跨域验证
        registry.addMapping("/**")
                // 允许哪个域名访问。注意：Spring Boot 2.4 以后建议使用 allowedOriginPatterns
                .allowedOriginPatterns("*") 
                // 允许的HTTP请求方法
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")
                // 允许提交的请求头
                .allowedHeaders("*")
                // 是否允许携带 Cookie（非常重要，如果前端要带 Cookie，这里必须为 true）
                .allowCredentials(true)
                // 预检请求（OPTIONS）的有效期，单位为秒
                .maxAge(3600);
    }
}
```

## 其他的请求出错的现象

1. Mixed Content 混合内容

在 https 加密的网页中，尝试通过 XHR(XMLHttpRequest) | Fetch API来调用 http 接口。浏览器会直接禁止这种行为，因为 https 传输的数据是加密的，http传输数据是明文，会破坏安全性。

2. Secure Cookie

为了保持 HTTPS 的安全性，后端可以设置 Cookie 属性 Secure。浏览器接收到这个 Cookie ，在后续的请求行为中，只能在 https 协议的请求中附带这个 Cookie
