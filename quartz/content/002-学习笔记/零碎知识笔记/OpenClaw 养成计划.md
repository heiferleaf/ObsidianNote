# 会话 Session 管理

会话用于管理对话窗口。OpenClaw 的会话分为两种，**长期**会话和**子代理**会话。
- 长期会话（主会话）：相当于主进程，在会话中，可以查询`Memory.md` 文档，可以写入`Memory.md` 文档
- 子代理会话：交给子代理执行的任务，在会话中，无法查询外部的记忆信息，相当于从零开始的对话窗口，执行过程中，也无法写入记忆。整个环境和外部完全隔离，通常是用来测试，或者执行耗时任务。其最后得到的结果，会返回给主会话，主Agent判断是否写入记忆中。

一个会话中，运行的命令：
1. \new : 在当前会话中新开一个窗口，之前的窗口对话记录在 Sessions_history 中
2. \reset : 重置当前会话的窗口
3. \status : 当前会话的状态
4. sessions_list : 活跃会话列表，基本就是一个主会话，可能会有子代理会话
5. sessions_history 会话ID，找到当前会话的所有窗口的对话记录

从会话的记忆管理机制，可以发现 OpenClaw 是不会管理每一个会话窗口的上下文信息的（不像浏览器中对话页面可以直接点击历史对话）。当我们创建新的窗口，如果之前的窗口没有被保存到 Memory 文件中，那么新的窗口就感知不到之前窗口的信息。
虽然没有上次窗口的信息，但是会话的所有窗口，对话记录是被完整保存的。sessions_history 查询一个会话的所有窗口对话历史记录。

子代理会话，相关命令：
1. agents_list ：代理列表
2. sessions_send  \<to> \<message>：在主会话中，给代理子会话发送assistant消息，作用是补充上下文信息
3. sessions_spawn : 生成子代理会话
4. subagents list | kill | steer ： 列表，删除，生成新任务（以user身份发送消息）


# 网关管理

一个 OpenClaw 的Agent， 可以负责处理来自WebChat ，CLI 等多个不同的用户指令源的指令。借助于交换机的星形网（Hot-And-Spoke）架构，使用一个 Node.js 进程，作为一个网关 Gataway 使用，也就是架构中的交换机角色。

这样的好处是，把不同指令源发出指令，和指令被智能体实际执行的行为分离，同时网关来可以设计流量的限制，或者鉴权行为，防止非法访问。类似于动态代理模式。

- openclaw gateway run ：开启网关
- openclaw status -all  ： 可以查询网关以及通道连接状态

# 定时任务 Cron 管理

- Cron 定时任务，不是跑在 Agent 内部，而是在网关中的，有效的减少了 Token 的开销和模型的负担。
- 通过指定 Gateway 在指定的时间，或者时间间隔，向 Agent 发送执行内容，将得到的结果推送到配置的通道，比如微信。
- 通过指定 Cron 执行的会话，可以放在主会话中，能读取记忆，或者放在子代理会话中，可以隔离环境，防止污染主会话信息
