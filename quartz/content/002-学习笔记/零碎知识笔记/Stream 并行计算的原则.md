> 这是一个非常深入的问题。要理解哪些 API 支持并行，我们需要从 **“流的生命周期”** 出发。一个 Stream 操作通常分为三部分：**数据源**、**中间操作**、**终端操作**。

并不是所有的 API 在并行环境下都能获得加速，有些甚至会因为并行而变得更慢。

### 1. 数据源：哪些“源头”支持并行？

并行流的第一步是拆分（Splitting）。如果数据源很难拆开，后续 API 再怎么并行也没用。

- **高效并行（性能好）：** `ArrayList`、数组（`IntStream.range`）、`HashMap`。它们支持随机访问，可以轻易地被 `Spliterator` 对半拆分。
    
- **低效并行（性能差）：** `LinkedList`、`Stream.iterate`、`BufferedReader.lines()`。它们必须按顺序遍历才能到达中间位置，拆分代价极高。

> 数据源像是一个仓库，计算需要从中拿去数据。对于不支持随机访问的仓库，在并行计算拿数据时，需要前拿到前面n个数据，才能拿到 n + 1 个数据
> 对于顺序访问的仓库，可以分别拿去 n 个数据，这个数据拿去过程可以是并行的

---

### 2. 中间操作：哪些 API 是并行的“好搭档”？

中间操作分为**无状态（Stateless）** 和 **有状态（Stateful）** 两类。

#### (1) 无状态操作 —— 天生支持并行（强烈推荐）

这些操作处理每个元素时是独立的，互不影响，是并行流最理想的场景：

- **`map()` / `flatMap()`**：将 A 转 B，每个元素独立转换。
    
- **`filter()`**：判断元素是否保留。
    
- **`peek()`**：对每个元素执行某个动作。
    

#### (2) 有状态操作 —— 并行代价高（慎用）

这些操作需要知道“其他元素是什么”才能继续，在并行时需要复杂的同步或合并逻辑：

- **`distinct()`**：需要维护一个全局 Set 来去重。
    
- **`sorted()`**：必须等所有线程处理完，再在全局范围内进行排序。
    
- **`limit()` / `skip()`**：因为流是有序的，并行流必须确保拿到的正是前 N 个，这涉及到大量的线程间通信。
    

---

### 3. 终端操作：真正的并行“收官”

这是决定结果如何合并的地方。

#### (1) 归约操作（Reduce/Collect）—— 性能飞跃

这是并行流最擅长的地方。只要你的操作满足 **结合律**（即 `(a+b)+c == a+(b+c)`），性能就极好：

- **`reduce()`**：如求和、求最大值。
    
- **`collect()`**：将结果收集到容器中。使用并行流时，建议配合 `Collectors.toConcurrentMap()` 或 `groupingByConcurrent()`，它们能直接并发写入结果容器。
    
- **`count()` / `sum()` / `min()` / `max()`**：这些都是特化的 `reduce`，非常快。
    

#### (2) 匹配与查找（Short-circuiting）

- **`anyMatch()` / `allMatch()` / `noneMatch()`**：一旦某个线程找到了结果，其他线程可以立即停止，这种“短路”特性在并行下效率很高。
    
- **`findAny()`**：在并行流中比 `findFirst()` 快得多，因为它不需要关心元素的原始顺序。
    

#### (3) 副作用操作（不推荐）

- **`forEach()`**：在并行流中，它的执行顺序是随机的。如果你要求顺序，必须用 `forEachOrdered()`，但这会使并行失去意义。
    

---

### 4. 总结：什么样的任务适合并行？

你可以用这三条准则来判断你的任务是否该用并行 API：

1. **独立性**：每个元素的处理逻辑是否不依赖于其他元素？（如 `map`, `filter`）
    
2. **计算量**：每个元素的处理是否足够重？（如果只是简单的加法，并行的开销会超过收益）
    
3. **结果合并**：是否容易合并？（如 `sum`, `collect` 到 Map 比 `collect` 到顺序固定的 List 更适合并行）
    

**核心避坑指南：**

如果你在并行流中使用了 **`sorted()`**、**`limit()`** 或者 **`LinkedList`** 作为数据源，那么 `.parallel()` 很可能不仅不会提速，反而会因为线程竞争和排序合并导致性能下降。
