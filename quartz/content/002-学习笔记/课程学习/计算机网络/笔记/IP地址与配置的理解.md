# IPv6 与网络配置完整体系笔记

---

## 第一章：网络通信的本质与 IP 地址的诞生

### 1.1 网络通信的根本问题

**核心问题：如何找到对方？**

类比邮政系统，地址必须满足：

- **唯一性**：不能有两个相同地址
- **层次性**：便于投递（国家-省-市-街道-门牌号）
- **可路由性**：能找到投递路径

**MAC 地址的缺陷：**

- 平面结构，无层次性
- 无法有效路由
- 大规模网络中路由器无法记住所有设备位置

### 1.2 IP 地址的设计哲学

**核心设计：分层结构**

```
IP 地址 = 网络部分 + 主机部分
```

**优势：**

- 路由器只需关心网络部分
- 本地网络负责主机部分
- 网络可无限扩展
- 路由表规模可控

### 1.3 IPv4 的诞生（1981年）

**基本参数：**

- 地址长度：32 位
- 表示方法：四组十进制数（如 192.168.1.100）
- 地址总数：2³² ≈ 43 亿

**分类地址系统：**

|类别|地址范围|网络位|主机位|每网络主机数|用途|
|---|---|---|---|---|---|
|A 类|0.0.0.0 - 127.255.255.255|8|24|1600万|超大型网络|
|B 类|128.0.0.0 - 191.255.255.255|16|16|65,536|中型网络|
|C 类|192.0.0.0 - 223.255.255.255|24|8|254|小型网络|
|D 类|224.0.0.0 - 239.255.255.255|-|-|-|组播|
|E 类|240.0.0.0 - 255.255.255.255|-|-|-|保留|

### 1.4 IPv4 的第一次危机与 CIDR

**问题：分类地址太僵硬**

- 需要 300 台主机：C 类不够，B 类浪费 65,000+ 地址
- 地址空间迅速耗尽

**解决方案：CIDR（无类别域间路由）**

引入可变长度子网掩码，抛弃固定分类。

**子网掩码的本质：**

- 标记网络部分与主机部分的分界线
- 告诉系统前多少位是网络部分

**CIDR 表示法：**

```
192.168.1.0/24
- /24 表示前 24 位是网络部分
- 剩余 8 位是主机部分
- 可用主机数：2⁸ - 2 = 254

10.0.0.0/16
- /16 表示前 16 位是网络部分
- 剩余 16 位是主机部分
- 可用主机数：2¹⁶ - 2 = 65,534
```

**为什么要减 2：**

- 网络地址（主机位全 0）
- 广播地址（主机位全 1）

### 1.5 NAT：临时解决方案

**NAT（网络地址转换）工作原理：**

1. 内部使用私有地址（192.168.x.x, 10.x.x.x, 172.16.x.x）
2. 路由器将私有地址映射到公网地址
3. 多台设备共享一个公网 IP

**私有地址范围：**

- 10.0.0.0/8（10.0.0.0 - 10.255.255.255）
- 172.16.0.0/12（172.16.0.0 - 172.31.255.255）
- 192.168.0.0/16（192.168.0.0 - 192.168.255.255）

**NAT 带来的问题：**

- 打破端到端通信原则
- 外部无法直接访问内部设备
- 需要端口转发、STUN 等复杂机制
- P2P、VoIP 等应用受影响

### 1.6 IPv4 地址彻底耗尽

**时间线：**

- 2011 年：IANA 宣布 IPv4 地址完全分配完毕
- 各区域注册机构陆续耗尽

**迫使转向 IPv6**

## 第二章：IPv6 的设计哲学与核心架构

### 2.1 IPv6 的五大设计目标

**目标 1：足够的地址空间**

- 128 位地址
- 2¹²⁸ ≈ 3.4 × 10³⁸ 个地址
- 地球每粒沙子都能分配地址

**目标 2：恢复端到端通信**

- 每台设备拥有全球唯一公网地址
- 不再需要 NAT
- 简化网络架构

**目标 3：简化路由**

- 层次化地址分配
- 减少路由表大小
- 提高路由效率

**目标 4：内置安全性**

- IPsec 成为标准组件
- 默认支持加密和认证

**目标 5：更好的移动性**

- 设备在不同网络间漫游
- 保持连接不中断

### 2.2 IPv6 地址表示方法

**标准格式：**

```
2001:0db8:85a3:0000:0000:8a2e:0370:7334
```

- 8 组，每组 4 位十六进制数
- 用冒号分隔
- 共 128 位

**简化规则：**

**规则 1：省略前导零**

```
2001:db8:85a3:0:0:8a2e:370:7334
```

**规则 2：连续的零用 :: 替代（只能用一次）**

```
2001:db8:85a3::8a2e:370:7334
```

**特殊地址示例：**

```
::1          # 本地回环地址（相当于 127.0.0.1）
::           # 所有位为 0（未指定地址）
fe80::       # 链路本地地址前缀
```

**为什么 :: 只能用一次？**

- 如果用多次，无法确定每个 :: 代表多少个 0
- 例如：`2001::85a3::7334` 无法判断每个 :: 是几组 0

### 2.3 IPv6 地址的层次结构

**全球单播地址的标准结构（最常用）：**

```
|←─── 48 位 ───→|←─ 16 位 ─→|←────── 64 位 ──────→|
|  全球路由前缀   |   子网ID   |      接口ID        |
| 2001:0db8:1234 |    5678    | abcd:ef01:2345:6789 |
```

**三层含义：**

1. **全球路由前缀（48 位）**
    
    - 由 ISP 分配
    - 用于全球路由
    - 相当于 IPv4 的网络部分
2. **子网 ID（16 位）**
    
    - 组织内部管理
    - 可划分 65,536 个子网
    - 灵活的内部网络规划
3. **接口 ID（64 位）**
    
    - 标识具体设备接口
    - 生成方式：
        - 基于 MAC 地址（EUI-64）
        - 随机生成（隐私扩展）
        - 手动配置

**实例分析：**

```
地址：2001:0db8:1234:5678:abcd:ef01:2345:6789

分解：
2001:0db8:1234  → ISP 分配的全球路由前缀
5678            → 组织内部第 22136 号子网
abcd:ef01:2345:6789 → 设备唯一标识
```

**层次结构的优势：**

- 路由器只看前缀即可转发
- 组织灵活划分内部网络
- 设备可自动配置地址

## 第三章：IPv6 地址分类与类型

### 3.1 地址分类概览

IPv6 定义了三种主要地址类型：

|类型|用途|类比 IPv4|
|---|---|---|
|单播（Unicast）|一对一通信|普通 IP 地址|
|组播（Multicast）|一对多通信|组播地址|
|任播（Anycast）|一对最近一个|无直接对应|

**重要变化：IPv6 取消了广播**

- 用组播替代
- 更精确定位接收者
- 减少不必要流量

### 3.2 单播地址（Unicast）

#### 3.2.1 全球单播地址（Global Unicast Address）

**特征：**

- 前缀：`2000::/3`
- 全球可路由
- 相当于 IPv4 公网地址

**前缀 2000::/3 的含义：**

```
2000 (十六进制) = 0010 0000 0000 0000 (二进制)
/3 表示前 3 位固定：001

地址范围：
- 最小：2000:: (001后全0)
- 最大：3fff:ffff:ffff:ffff:ffff:ffff:ffff:ffff (001后全1)
```

**分配结构：**

```
用户获得的典型分配：/48 或 /56
- /48：可创建 65,536 个 /64 子网
- /56：可创建 256 个 /64 子网
- /64：标准子网大小，用于单个网络段
```

#### 3.2.2 链路本地地址（Link-Local Address）

**特征：**

- 前缀：`fe80::/10`
- 仅在本地链路有效
- 不被路由器转发
- 每个启用 IPv6 的接口自动生成

**前缀 fe80::/10 的含义：**

```
fe80 (十六进制) = 1111 1110 1000 0000 (二进制)
/10 表示前 10 位固定：1111111010

实际使用中通常是 fe80::/64
后 64 位是接口 ID
```

**用途：**

- 设备即插即用通信
- 邻居发现协议
- 路由协议本地通信
- 无需配置即可工作

**示例：**

```
fe80::1                    # 手动配置常用
fe80::21a:2bff:fe3c:4d5e  # 基于 MAC 自动生成
```

#### 3.2.3 唯一本地地址（Unique Local Address, ULA）

**特征：**

- 前缀：`fc00::/7`（实际使用 `fd00::/8`）
- 私有网络内部使用
- 不应在公网路由
- 相当于 IPv4 私有地址

**为什么有两个前缀？**

- `fc00::/8`：预留给未来（需中央分配）
- `fd00::/8`：当前使用（本地生成）

**生成方法：**

```
fd + 40位随机全球ID + 16位子网ID + 64位接口ID
例如：fd12:3456:789a:1::1
```

**用途：**

- 企业内部网络
- 与互联网隔离的网络
- 本地服务器通信

### 3.3 组播地址（Multicast）

**特征：**

- 前缀：`ff00::/8`
- 一对多通信
- 替代 IPv4 广播

**格式：**

```
ff<flags><scope>:<group-id>

flags（4位）：
- 0：永久分配的组播地址
- 1：临时组播地址

scope（4位）：作用域
- 1：接口本地
- 2：链路本地
- 5：站点本地
- 8：组织本地
- e：全球
```

**重要的组播地址：**

|地址|含义|用途|
|---|---|---|
|`ff02::1`|所有节点（链路本地）|类似广播|
|`ff02::2`|所有路由器（链路本地）|路由器发现|
|`ff02::5`|所有 OSPF 路由器|OSPF 协议|
|`ff02::1:2`|所有 DHCP 服务器|DHCPv6|
|`ff02::1:ff00:0/104`|请求节点组播|地址解析|

**请求节点组播地址详解：**

```
前缀：ff02::1:ff00:0/104
后缀：目标 IPv6 地址的最后 24 位

例如：
目标地址：2001:db8::1234:5678
请求节点组播：ff02::1:ff34:5678
               (取最后24位)

用途：
- 邻居发现
- 重复地址检测
- 只有匹配的设备处理，减少网络负担
```

### 3.4 任播地址（Anycast）

**特征：**

- 多个设备配置相同地址
- 数据包路由到"最近"的设备
- 基于路由度量（跳数、延迟等）

**格式：**

- 使用单播地址空间
- 通过配置标识为任播

**典型应用：**

1. **DNS 根服务器**
    
    - 全球多个节点使用相同 IP
    - 用户自动连接到最近的服务器
2. **CDN 节点**
    
    - 内容分发网络
    - 就近访问加速
3. **网关路由器**
    
    - 冗余和负载均衡
    - 自动故障切换

**子网路由器任播地址：**

```
前缀/128（接口ID全0）
例如：2001:db8:1234:5678::/64 的任播地址是
     2001:db8:1234:5678::
```

### 3.5 特殊地址汇总表

|地址|类型|前缀|用途|IPv4 对应|
|---|---|---|---|---|
|`::/128`|未指定地址|-|未获得地址时使用|0.0.0.0|
|`::1/128`|回环地址|-|本机通信|127.0.0.1|
|`fe80::/10`|链路本地|1111111010|本地链路通信|169.254.x.x|
|`fc00::/7`|唯一本地|1111110|私有网络|10.x, 192.168.x|
|`fd00::/8`|唯一本地（实际使用）|11111101|私有网络|10.x, 192.168.x|
|`2000::/3`|全球单播|001|公网地址|公网 IP|
|`ff00::/8`|组播|11111111|一对多通信|224.x 组播|
|`::ffff:0:0/96`|IPv4 映射|-|IPv4 兼容|-|

## 第四章：IPv6 地址自动配置机制

### 4.1 IPv4 配置方式回顾

**手动配置：**

- 管理员逐台设置：IP、子网掩码、网关、DNS
- 费时费力，易出错

**DHCP（动态主机配置协议）：**

- 客户端向服务器请求配置
- 服务器分配 IP、网关、DNS
- 依赖中心化服务器

**IPv4 的局限：**

- 必须有 DHCP 服务器或手动配置
- 无法真正"即插即用"

### 4.2 IPv6 的革命：SLAAC

**SLAAC（无状态地址自动配置）：**

- Stateless Address Autoconfiguration
- 真正的即插即用

**三大特点：**

1. **无状态**：路由器不记录地址分配
2. **分布式**：不依赖中心服务器
3. **自动化**：设备自己生成地址

### 4.3 SLAAC 完整工作流程

#### 步骤 1：生成链路本地地址

设备接入网络后立即生成链路本地地址：

```
前缀：fe80::/64
接口ID：基于 MAC 或随机生成
```

**EUI-64 算法（基于 MAC 生成接口 ID）：**

```
MAC 地址：00:1A:2B:3C:4D:5E

步骤：
1. 在中间插入 fffe：
   00:1A:2B:FF:FE:3C:4D:5E

2. 转换为 IPv6 格式：
   001A:2Bff:fe3C:4D5E

3. 翻转第7位（U/L位，通用/本地位）：
   00 = 0000 0000
   翻转第7位后 = 0000 0010 = 02
   
   最终：021A:2Bff:fe3C:4D5E

4. 完整链路本地地址：
   fe80::21a:2bff:fe3c:4d5e
```

**为什么要翻转第7位？**

- IEEE 规定：第7位为0表示全球唯一MAC
- 翻转后为1，表示本地管理的地址
- 区分真实MAC和衍生地址

#### 步骤 2：重复地址检测（DAD）

**目的：** 确保地址唯一性

**流程：**

```
1. 设备将地址标记为"暂定"（tentative）
2. 发送 NS（邻居请求）到请求节点组播地址：
   - 源地址：::（未指定）
   - 目标：ff02::1:ff3c:4d5e（请求节点组播）
   - 询问："有人在用 fe80::21a:2bff:fe3c:4d5e 吗？"

3. 等待 1 秒（标准等待时间）

4. 结果：
   - 收到 NA 响应 → 地址冲突，重新生成
   - 无响应 → 地址可用，状态变为 preferred
```

**DAD 的 ICMPv6 消息：**

```
类型：135（邻居请求 NS）
源地址：::
目标地址：ff02::1:ff3c:4d5e
目标 IPv6：fe80::21a:2bff:fe3c:4d5e（正在检测的地址）
```

**优化 DAD（RFC 4429）：**

- 标记为"乐观"（optimistic）
- 立即使用，但不确定
- 如果冲突，停止使用
- 加速连接建立

#### 步骤 3：发送路由器请求（RS）

设备需要获取网络前缀：

```
类型：133（Router Solicitation）
源地址：fe80::21a:2bff:fe3c:4d5e（刚配置的链路本地地址）
目标地址：ff02::2（所有路由器组播地址）
含义："有路由器吗？请告诉我网络配置"
```

#### 步骤 4：接收路由器通告（RA）

路由器响应或定期发送 RA：

```
类型：134（Router Advertisement）
源地址：fe80::1（路由器的链路本地地址）
目标地址：ff02::1（所有节点）或 RS 的源地址

包含信息：
1. 前缀信息：
   - 前缀：2001:db8:1234:5678::/64
   - 有效生存期：86400秒（24小时）
   - 首选生存期：14400秒（4小时）
   - 自动配置标志：1（允许自动配置）
   - 链路上标志：1（该前缀在此链路上）

2. 路由器信息：
   - 路由器生存期：1800秒
   - 可达时间：30000毫秒
   - 重传计时器：1000毫秒

3. MTU：1500字节

4. 配置标志：
   - M标志（Managed）：0（不需要DHCPv6获取地址）
   - O标志（Other）：0（不需要DHCPv6获取其他配置）

5. DNS信息（RFC 8106）：
   - DNS服务器：2001:db8::53
   - DNS搜索域：example.com
```

**RA 的发送频率：**

- 定期发送：每 200秒（MaxRtrAdvInterval）
- 收到 RS 后：立即或延迟少量时间
- 最小间隔：3秒（MinRtrAdvInterval）

#### 步骤 5：生成全球单播地址

设备组合前缀和接口 ID：

```
路由器提供的前缀：2001:db8:1234:5678::/64
设备的接口ID：21a:2bff:fe3c:4d5e

完整全球单播地址：
2001:db8:1234:5678:21a:2bff:fe3c:4d5e
```

#### 步骤 6：对全球地址进行 DAD

再次执行 DAD 流程，确保全球地址唯一。

#### 步骤 7：配置完成

设备现在拥有：

```
1. 链路本地地址：
   fe80::21a:2bff:fe3c:4d5e

2. 全球单播地址：
   2001:db8:1234:5678:21a:2bff:fe3c:4d5e

3. 默认网关：
   fe80::1

4. 前缀信息：
   2001:db8:1234:5678::/64

5. DNS服务器（如果RA包含）：
   2001:db8::53
```

### 4.4 隐私扩展（RFC 4941 / RFC 8981）

**问题：** 基于 MAC 生成的接口 ID 会暴露设备硬件信息：

- 同一设备在不同网络使用相同接口 ID
- 可被追踪
- 隐私泄露

**解决方案：隐私扩展**

**工作机制：**

```
1. 生成随机接口ID：
   - 使用加密安全的随机数生成器
   - 例如：3d2a:7b8f:4c9e:1a5d

2. 形成临时地址：
   2001:db8:1234:5678:3d2a:7b8f:4c9e:1a5d

3. 定期更换（默认24小时）

4. 废弃旧地址（但保留一段时间用于接收）
```

**设备同时拥有两种地址：**

|类型|示例|用途|特点|
|---|---|---|---|
|稳定地址|2001:db8::21a:2bff:fe3c:4d5e|入站连接|基于MAC，不变|
|临时地址|2001:db8::3d2a:7b8f:4c9e:1a5d|出站连接|随机，定期更换|

**地址生命周期：**

```
优先生存期（Preferred Lifetime）：24小时
- 在此期间，地址为"首选"
- 用于新建连接

有效生存期（Valid Lifetime）：7天
- 超过优先期后，地址变为"废弃"（deprecated）
- 不用于新连接，但保持现有连接
- 超过有效期后，地址删除
```

**现代操作系统行为：**

- Windows 10/11：默认启用
- Linux：可通过 `net.ipv6.conf.all.use_tempaddr` 配置
- macOS：默认启用
- Android/iOS：默认启用

### 4.5 DHCPv6：有状态配置

**为什么需要 DHCPv6？** 虽然 SLAAC 很方便，但某些场景需要更多控制：

- 需要固定 IP 分配
- 需要记录哪个设备获得哪个地址
- 需要提供更多配置参数
- 企业网络管理需求

**DHCPv6 的两种模式：**

#### 模式 1：有状态 DHCPv6（Stateful）

**触发条件：** 路由器 RA 中 M 标志 = 1

**工作流程：**

```
1. 客户端发送 Solicit：
   - 源：fe80::客户端链路本地地址
   - 目标：ff02::1:2（所有DHCPv6服务器）
   - 含义："我需要地址配置"

2. 服务器响应 Advertise：
   - 提供可用配置
   - 服务器优先级

3. 客户端发送 Request：
   - 选择一个服务器
   - 请求配置

4. 服务器响应 Reply：
   - IPv6 地址
   - 前缀长度
   - 有效和优先生存期
   - DNS服务器
   - NTP服务器
   - 其他配置
```

**服务器维护状态：**

```
客户端DUID    地址                      租期      状态
=====================================
01:02:03...   2001:db8::100/64        86400s    Active
04:05:06...   2001:db8::101/64        86400s    Active
```

#### 模式 2：无状态 DHCPv6（Stateless）

**触发条件：**

- M 标志 = 0
- O 标志 = 1

**特点：**

- 地址通过 SLAAC 获取
- 仅从 DHCPv6 获取其他配置（DNS、域名等）
- 服务器不维护地址分配状态

**流程：**

```
1. 设备通过 SLAAC 配置地址
2. 发送 Information-Request 到 DHCPv6 服务器
3. 服务器响应 Reply，包含：
   - DNS 服务器列表
   - 域名搜索列表
   - NTP 服务器
   - 其他选项
```

#### 模式 3：纯 SLAAC

**触发条件：**

- M 标志 = 0
- O 标志 = 0

**特点：**

- 完全不使用 DHCPv6
- 从 RA 获取所有配置
- RA 可包含 DNS 信息（RFC 8106）

**RA 中的 DNS 选项：**

```
RDNSS（递归 DNS 服务器）：
- 2001:db8::53
- 2001:db8::54

DNSSL（DNS 搜索列表）：
- example.com
- corp.example.com
```
