	三要素：
		1. 数据结构
			**定义数据的结构和数据之间的联系**

		3. 完整性约束

### 三级模式结构

外模式/子模式
	数据库用户能都看见和使用的局部数据 的逻辑结构和特征的表述
模式
	对数据库中数据的 逻辑结构和特征 的表述
内模式/物理模式
	对数据库中存储数据的物理结构表述

#### 两级映像

外模式/模式映像 支持逻辑独立性
模式/内模式 支持物理独立性

> [!NOTE] **数据模型和模式的区别**
> - 数据模型是理论层面的，对数据的结构和特征进行逻辑抽象，可以概括为数据结构、数据操纵、完整性约束。
> - 模式是是应用层面的，用于在具体的数据库中，对基于具体的数据模型存储的数据，进行结构和特征的描述。可以分为外模式、模式、内模式。
> 	- eg:在关系模式中，关系模式就描述了$R(U,D,DOM,F)$，即存储数据的数据表的结构和特征表述

## 第五章 数据库完整性

完整性包含了数据库的正确性、相容性、有效性。

而在关系型数据库中，关系模型的完整性包括了实体完整性、参照完整性、用户自定义完整性。

1. 实体完整性
	保证关系的元组是可以区分的，要求主键值非空唯一
	primary key 列级、表级
2. 参照完整性
	保证参考的值存在，要求外码值为空或者在对应的被参考码中存在
3. 用户自定义完整性
	not null、unique、check、default

## 第六章 关系数据理论

1. 从函数依赖集提取码
	1. L + N ：一定在任何候选码中
	2. R：一定不在任何的候选码中
	3. （L+N）的闭包不是候选码，就逐个组合LR部分
2. 判别范式的小结论 + 一般思路（23NF的非主属性很重要）
	1. 候选码都是由单属性构成的，至少满足2NF
	2. 所有属性都是主属性，至少满足3NF
	3. 只有两个属性的关系（二元关系），至少满足BCNF
	4. 候选码包含了所有的属性（全码），至少满足BCNF
	5. 一般思路
		1. 2NF：**非主属性**不存在**部分函数**依赖于码
		2. 3NF：**非主属性**不存在**传递函数**依赖于码
		3. BCNF：所有的函数依赖左部都包含码
3. 函数依赖集闭包 - 通过属性集在函数依赖集下的闭包
4. 最小函数依赖集（最小覆盖）三步
	1. 右部拆分为单属性
	2. 左右精简
		1. 对于$x\rightarrow A,x=B_1B_2\dots B_N$
		2. 要求$A \notin (x-B_i)_F^+$，这里最需要注意的就是，是在原来函数依赖集下的闭包，**不需要把$x \rightarrow A$排除掉**
	3. 去除冗余
		1. 令$G = F-\{x\rightarrow A\}$
		2. 要求$A \notin x_G^+$
5. 模式分解
	1. 保持函数依赖的模式分解3NF
		1. 极小化函数依赖，构造出第一个分解：不在$F_m$中的属性集，以及对应的函数依赖投影（其实为空集）
		2. 如果$F_m$中有函数依赖的左右部分并=剩余属性集，结束
		3. 函数依赖按照左部划分，划分后的组，将所涉及到的属性并集，去掉冗余的属性组，然后进行函数依赖的投影
	2. 保持函数依赖和无损连接的模式分解
		1. 在保持函数依赖的基础上
		2. 添加一组候选码的属性和对应的函数依赖投影，去除掉冗余属性集
	3. 保持无损连接的模式分解
		1. $t_0=\{ R \}$
		2. 对$t$中的所有子模式的函数依赖遍历，如果存在一个子模式，模式中有函数依赖的左部不包含子模式的候选码，进行拆分
		3. 直到无法进行拆分，否则重复2
	4. 判别是否保持函数依赖
		1. 子模式的函数依赖并集闭包 == 原函数依赖闭包
	5. 判别是否保持无损连接（对极小函数依赖操作）
		1. 分解成多个子模式
			1. 按照每一个子模式包含的属性情况填表
			   $a_j:表示子模式包含第j个属性$，$b_{ij}:表示第i个子模式不包含第j个属性$
			2. 循环遍历原函数依赖集
			   考虑左部相同的行，这些行中，右部属性列有$a$，都改为$a$，不然改为$b_{ij},i是最小的行标号$
			3. 循环后，如果有一行全为a，则保持无损连接，如果没有发生任何变化，不能保持无损连接，除此之外，继续循环
		2. 二元分解（分解两个子模式）
			$U_1-U_2 \in (U_1 \cap U_2)_F^+$
			是在原来的函数依赖集下的闭包判断
	6. 无损连接的特性
		1. $r$是$m_\rho(r)$的子集
		2. $m_\rho(r)$投影和$r$的投影相同（因为投影会去重）
		3. 幂等性
6. 触发器
	1. 触发器的概念
		1. 是数据库的一种特殊存储过程。由事件驱动，当某些特定操作发生时，由DBMS自动执行
	2. 触发器的组成
		1. 触发事件：数据库的操作事件
		2. 触发条件：当触发事件发生后，需要满足触发条件，才能执行动作
		3. 触发动作：触发器被触发的行为，可以定义为**事件发生**之前（BEFORE）和事件发生之后（AFTER）
	3. 和存储过程的区别
		1. 存储过程是用户自己编写的语句，可以在事务中执行，可以立即执行
		2. 触发器是由事件驱动并由DBMS自动执行，用于响应事件发生，对数据库进行监管
	4. 触发器的分类
		1. 行级触发器：针对数据行的变更，会触发触发器，需要判断每一行是否满足触发器的条件
		2. 语句级触发器：针对每一条SQL，触发一次触发器，用于对操作进行监控和限制
	5. 触发器的事件
		1. insert、delete、update
		2. 1中的组合

## 第七章 数据库设计

1. 数据库设计的概念
	1. 广义：设计完整的数据库系统和应用系统
	2. 狭义：设计数据库系统
	3. 一般定义：在给定的应用环境下，设计数据库的三级模式，建立数据库的应用系统，使之满足用户的需求，包括信息管理需求和数据操作需求。
2. 目标
	1. 较高的存储空间利用
	2. 较高的存储效率
	3. 较高的运维效率
3. 特点
	1. 重视基础数据
	2. 数据库设计和应用设计一起进行
4. 步骤
	1. 需求分析（DD）
	2. 概念模型设计
		1. EF图
			1. 基数的含义解释 - 怎么标 - 宾语解读
			2. 实体之间是组合关系的，part of 用弱联系和弱实体表示
			3. 联系可以涉及一个实体，二个实体，多个实体
		2. 混合方式设计：需求设计自顶而下，ER设计自底而上
		3. ER合并时候冲突
			1. 结构冲突
				1. 对象在不同应用中，有的实体，有的属性
				2. 实体的属性个数，顺序
				3. 实体之间的联系类型（1对1，1对多，多对多）
			2. 命名冲突
				1. 同名异义
				2. 异名同义
			3. 属性冲突
				1. 属性域冲突
				2. 属性取值单位冲突
			4. 通过函数依赖集和极小化依赖集的差集，识别冗余的属性
	3. 逻辑结构设计
		1. 实体转为关系表
		2. 联系看情况
			1. 一对一，任意一方插入外码，和联系的属性
			2. 一对多，多方插入外码
			3. 多对多，联系变成一张表，将连接的实体的主键合并作为主键
			4. 多个实体的联系：转为表，将连接的实体的主键合并作为主键
	4. 物理结构设计
		1. 索引
			1. 概念：快速的定位数据，不需要扫描整个表
			2. 不适用的情况
				1. 表比较小
				2. 表中不经常用于查询的列
				3. 表中常更新的列
				4. 列属性值太少
				5. 列属性值分布不均衡
				6. ==经常查询大部分数据的表==
		2. 聚簇索引
			1. 概念：数据行的顺序和索引的顺序一致，索引叶子节点存放数据行
			2. 特点
				1. 每张表只能有一个
				2. 查询速度快，插入数据速度慢（需要移动数据）
				3. 适用于范围查询
				4. 常常在主键设置索引
			3. 和非聚簇的对比
				1. 非聚簇索引的数据行的顺序和索引的顺序无关。索引叶子节点存放指向实际数据的指针。
				2. 可以有多个非聚簇索引
				3. 查询数据要先查指针，再找数据
				4. ==插入 / 更新的影响小==
			4. 适用的情况
				1. 主键
				2. 常用于范围查询的列
				3. 常用于分组的列
				4. 常用于排序的列
				5. 会返回连续大量数据的表
		3. 如何加快查询效率
			1. 索引
			2. 水平分裂
			3. 垂直分裂
	5. 数据库实施
	6. 数据库运行和维护

## 第八章 数据库恢复技术

1. 日志：记录事物对数据库的操作，写内存的数据部分之前要先写日志，才能实现恢复控制
2. 数据库的恢复全靠冗余
	1. 数据转存
		1. 静态转存：转存时，不能有操作
		2. 动态转存：可以由操作，所以不一定有效，用的最多，配合日志文件副本使用
		3. 海量转存：完整的数据库备份
		4. 增量转存：变化的数据的备份
	2. 日志：需要先写日志
		1. 登记点、重新开始文件：提高效率
	3. 数据库镜像
	4. 远程备份系统
3. 故障种类
	1. 事务故障：事务内部因为执行出错而故障，需要UNDO
	2. 系统故障：操作系统故障，导致部分事务没有执行完，有些执行完的事务，数据还在缓存中，需要UNDO和REDO（借助日志判断）
	3. 介质故障：通过数据库副本和日志文件副本，REDO已经完成的事情（没有完成的不会再数据副本中）
