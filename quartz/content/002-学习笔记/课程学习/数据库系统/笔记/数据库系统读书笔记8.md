> 数据库的恢复机制和并发控制技术是数据库管理系统的重要组成部分，本章介绍数据库恢复的基本概念。

## 第11章 数据库恢复技术

### 11.1 事务的基本概念

1. 事务
	事务就是一串操作序列，这串操作要么都做，要么都不做。
	事务的定于语法
	```sql
	BEGIN TRANSACTION;
	-- 一般事务都要用这条语句作为开始标志
	COMMIT;
	-- 表示提交，将事务对数据库的更新写回磁盘（写的是日志commit内容）
	ROLLBACK;
	-- 回滚，把事务的操作进行撤销
	
	执行到 COMMIT：
├─ 系统承诺：这个事务一定会持久化
├─ 即使崩溃，恢复时也会 REDO（重做）
└─ 保证：修改不会丢失 ✓

没执行到 COMMIT 就崩溃：
├─ 没有承诺 = 事务未完成
├─ 为了保持一致性，必须回滚
└─ 恢复时会 UNDO（撤销）所有修改 ✓
	```

2. 事务的ACID
	1. 原子性：操作不可分
	2. 一致性：从一个状态到另一个状态
	3. 隔离性：事物之间相互不能干扰
	4. 持续性：事务提交之后，对数据的修改的是永久的
	从特性可以观察到，事务收到破坏的原因可能有
		1. 多个事务并行运行，事务的代码交叉运行，破坏隔离
		2. 事务运行中被强制停止，破坏原子性

### 11.2 数据库恢复

虽然数据库系统采取了各种保护措施来防止数据库的安全性和完整性，保证并发事务的正确执行，但是计算机的硬件故障和软件出错、操作员的事务还是避免不了，这些都会导致数据的状态出错。==因此，数据库系统须有能力，在出错时恢复到正确的状态==

### 11.3 故障的种类

1. 事务内部的故障
	这里提到的事物内部的故障，更多强调非预期的事务出错，比如运算溢出，事务并发死锁。
	这类故障说明事务没有到达预期的运行终点（COMMIT或者显示的ROLLBACK）所以恢复程序需要强行的回滚事务，这种操作叫做==事务的撤销（UNDO）==
2. 系统故障（软故障）
	指造成系统停止运转的任何事件，使得系统要重新启动，这个是数据库的物理存储结构是没有被破坏的，影响的事务种类有两种：
	1. 事务还没有执行完，部分操作写入磁盘，部分还在内存缓冲区 --- 对这类操作UNDO
	2. 事务已经完成，commit日志写入磁盘，但是事务内容可能还有的没到磁盘， --- 对这类操作 REDO
3. 介质故障（硬故障）
	1. 外存故障、磁盘损坏。这种概率比较小，但是影响比较大，需要借助备份技术来进行数据的恢复

### 11.4 恢复的技术

> 恢复的技术概括来说就是两个字 -- 冗余
> 既然我需要恢复到正确的状态，我就肯定需要知道正确的状态长啥样，而知道长啥样有两种方法：数据转存和登记日志文件，通常这两种方法会一起使用

#### **11.4.1 数据转存**

把数据存储到其他介质上，留一个后备副本。不过这种恢复只能恢复到备份时的状态
具体可以细分
	静态：备份时，不能有对数据库的操作，要等用户活动结束
	动态：可以有，不一定有效
	海量：完整副本
	增量：新增的副本

| 海量转储 |      |      |
| 增量转储 |      |      |
#### **11.4.2 登记日志文件**

日志文件使用来记录事务对数据库的更新操作的文件，有两种格式
	1. 以记录为单位
		1. 日志登记内容包括
			1. 事务开始标志
			2. 事务结束标志
			3. 事务的更新操作
		2. 登记操作包括
			1. 事务标识，哪一个事务的
			2. 操作类型
			3. 操作对象
			4. 更新前旧值（对于插入来说，为空）
			5. 更新后数值（对删除来说，为空）
	2. 以磁盘块为单位
		1. 日志登记内容包括
			1. 事务标识
			2. 更新的数据块

日志文件有很重要的作用
	1. 事务故障恢复和系统故障恢复使用
	2. 动态转存时建立日志文件，后备副本和日志文件结合使用最好

为了保证日志文件的有效性，登记日志文件的时候必须遵循两条原则
1. 登记的次数严格按照并发事务的执行的时间次序
2. 先写日志文件，在写数据库（保证故障的时候，有日志的登记信息）

## 11.5 恢复策略

1. 事务故障的恢复
	事务在运行到终止点之前被终止，利用日志进行UNDO，==会由系统自己完成==，步骤如下
		1. 反向扫描日志文件，查找事务的更新操作
		2. 对更新操作进行逆操作
		3. 反复执行，直到读到事务的开始标记
2. 系统故障的恢复
	前面说过，分为两种，UNDO和REDO，UNDO和上面一样，REDO：
		1. 正向扫描日志文件，找到故障发生之前已经提交的事务（有begin transaction 和 commit），加入REDO队列，没有的加入undo队列
		2. 对redo队列重做日志记录操作
		3. 对undo队列做更新的逆操作
3. 介质故障的恢复
	1. 重装数据库，重做已经执行的事务
		1. 装入后备副本
		2. 装入对应的日志文件副本
		3. 对已经提交的事务进行重做
## 11.6 具有检查点的技术

根据上述的日志登记操作，会需要检查所有的日志文件，这种方式费力不讨好
	1. 需要查询所有的记录
	2. 有的重做实际上不需要，冗余操作了

一种解决方案：建立检查点和重新开始文件

检查点内容：
	1. 建立检查点的时候正在执行的事务清单
	2. 这些事务最近的一个日志记录的地址
重新开始文件：
	记录检查点记录在日志文件中的地址
动态维护日志文件
	周期性的建立检查点，保存数据库状态的操作，具体步骤
		1. 当前日志缓冲区的所有日志几率写入磁盘的日志文件
		2. 在日志文件中写入一个检查点记录
		3. 当前数据缓冲区数据写入磁盘
		4. 把检查点记录在日志文件中的地址写入重新开始文件
恢复操作：
	当事务的写入时间是在检查点之前的，一定已经执行了，不需要重做


## 11.7 数据库镜像

正如前述所说，介质故障时数据库系统故障中最严重的一种，所以需要周期性的转储数据库，这也加大了数据库管理员的负担。
那么随着技术的发展，现在许多数据库管理系统提供数据库镜像。镜像会自动把数据库系统关键数据复制到另一个磁盘上，这样子，当介质故障，可以用镜像内容，而不用关闭数据库去做副本的恢复
