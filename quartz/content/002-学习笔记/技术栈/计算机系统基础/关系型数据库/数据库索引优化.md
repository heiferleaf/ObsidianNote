前要：[[数据库中的索引使用]]

## 基本步骤

1. 开启慢查询日志，设置慢查询条件
2. 检查慢查询日志记录Sql的执行状况
3. 根据执行状况结果，判断是否进行优化，何种优化

## 优化思路

1. 通过索引，最后要实现的目标更快的把数据以需要的形式读取到内存中
	延伸出来的点包括
	- 减少磁盘IO次数---思路
		- 经常用作查询的列上设置索引
		- 索引覆盖查询字段，减少回表查询
		- 用于多表连接的列上设置索引，防止连接时全表扫描
	- 让读取到的数据少处理
		- 在需要排序的列上设置索引，进而读取到的数据就是有序的
		- 分组查询时，在分组列上设置索引
		- 使用联合索引时，覆盖where+order by涉及到的列

2. 如何发现全表扫描、回表查询、额外排序
	在 `EXPLAIN` 查询执行计划中，重点关注几个字段
	- type：本次查询中索引的使用情况
		- all：全表扫描
		- index：索引数扫描，扫描索引树中的所有结点，通常发生在索引覆盖查询字段的情况中
		- range：索引树中范围查询（b+树支持）
		- ref：非唯一性索引树，用于等值查询中，找到第一个索引键等于查询条件的结点，向后继续寻找
		- eq_ref：唯一性索引，依赖前表数据进行等值查询，找到第一个索引键等于查询条件的结点
		- const：在唯一性索引上，进行等值查询，能实现确定
	- rows：预计查询行数
	- keys：实际使用的索引
	- extra：关键字，表示发生什么事件
		- using index ： 索引覆盖
		- using filesort：额外排序，说明查询出来的数据在Order列上无序
		- using temporary：使用临时表，说明查询出进行分组的数据，在分组字段上无序

3. 根据查询计划判断
	1. 是否发生了多次回表查询--考虑联合索引覆盖？
	2. 是否Join连接操作中全表扫描--连接列上设置索引？
	3. 是否filesort：让查询出的数据order字段有序，联合索引覆盖
	4. 是否temporary：联合索引覆盖分组字段？
	**设计列索引时，也一定要先考虑字段在业务中用于查询的概率，是否会经常变化，取值种类，是否均衡**
