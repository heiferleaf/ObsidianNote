> 从基于服务器的开发到使用云原生技术：
> 从前，一个项目想要部署，需要精心准备合适内存大小，CPU算力的服务器，并且一旦服务器确定，这就是固定的开销。
> 而到现在的云原生，不需要关注所需资源的多少，一台机器的负载高了，云平台克隆相同的机器，用来负担负载，一台机器死机了，云平台直接开启一个新的服务。可以把云当作一个海量的资源平台，不需要担心资源的不足。
> 此外，云还具备弹性的重要特征，使用云的成本和实际业务需求量曲线一致，不会有买了100台服务器，有时候90台都是闲置状态。
> 最后说一点，云可以实现资源的接口，原先部署项目，前端、后端、数据库等，可能都部署在一台机子上面，而云原生中，会通过微服务的划分，将不同的领域服务放到不同的机器中，这样也能更适配云中资源的特点和服务的特点（如 数据库需要考虑备份冗余实现恢复技术）


==云计算的本质是 **资源池化（Resource Pooling）** 和 **自动化**，而其底层基石就是 **虚拟化技术**。==


# 对于云服务中涉及的IP地址理解

> 物理机：一般具备内部网IP，有的有全球IP
> Ingress：对用户通过DNS解析到的合法IP地址，会进入Ingress这个K8s管理的资源对象，通过域名，得到需要进行处理的Service，Service实现负载均衡，把请求给合适的Pod执行
> Pod内部暴露在外的IP地址：是内网IP地址，并且用户自己配置，所以物理层面的交换机、路由器不知道怎么转发，但是可以通过外面包一层物理机的IP，就知道怎么转发分组，并且，给Pod的IP加一个服务标识，就可以识别相同的内网地址的配置
> Service：实现负载均衡、滑动更新、副本替换的K8s对象，自己知道多个相同的Pod，选择合适的使用

### 一、 虚拟机（VM）内运行的容器，暴露的 IP 形式是怎么样的？

在 K8s 环境中，容器的 IP 形式和 VM 的 IP 形式本质上是**同一种东西**，它们都是**私有内网 IP**（Private IP），但在层级上多了一层分配逻辑。

#### 1. 容器 IP 的样子
如果你进入一个 Pod 内部执行 `ifconfig`，你会看到类似 `172.16.x.x` 或 `10.244.x.x` 的地址。
*   **特点：** 这个 IP 在整个 K8s 集群内是**唯一**且**互相可达**的。
*   **关系：** K8s 会给每个 Node（虚拟机或物理机）分配一个**大的网段**（比如 Node1 用 `10.244.1.0/24`），然后 Node1 里的每个 Pod 会从这个段里分到一个具体的 IP。

#### 2. “暴露”的真相
容器的 IP 默认**只在集群内部有效**。
*   **对于内部：** Pod A 可以直接通过 `10.244.1.5` 访问 Pod B。
*   **对于外部：** 容器的 IP 是**“不可见”**的。如果你想让家里的电脑访问这个容器，你直接 ping 这个 `10.244` 地址是 ping 不通的。
*   **结论：** 容器的 IP 形式就是**集群内网地址**。它需要通过 **Service** 或 **Ingress** 这两个“翻译官”才能被外界看到。

---

### 二、 Service：集群内部的“稳定调度员”

你已经知道 Pod 会死，IP 会变。**Service 的存在就是为了给这一组随时会变的 Pod 提供一个“永恒不变的身份”。**

#### 1. 详细流程（以访问数据库为例）
假设你有 3 个数据库 Pod，IP 分别是 `1.1, 1.2, 1.3`。

1.  **创建 Service：** 你定义一个 Service，K8s 会分配一个 **虚拟 IP（Cluster IP）**，比如 `10.96.0.10`。
2.  **绑定关系：** Service 通过 **标签（Selector）** 紧紧钩住这 3 个 Pod。
3.  **流量拦截（核心）：** 当你的程序访问 `10.96.0.10` 时，流量并没有发给任何真实的网卡。
4.  **内核魔术：** 每一台物理机上都有个组件叫 `kube-proxy`，它在 Linux 内核（iptables/IPVS）里写了条规则：*“凡是去找 `10.96.0.10` 的包，都给我**随机**转发给 `1.1, 1.2, 1.3` 中的一个。”*

#### 2. 作用
*   **负载均衡：** 自动把压力分给 3 个副本。
*   **服务发现：** 你的程序只需要在配置文件里写 `db-service`（K8s 自带 DNS 会解析到那个虚拟 IP），永远不用改。

---

### 三、 Ingress：集群大门的“金牌前台”

Service 虽然稳定，但它通常只在集群内网有效。如果你有 100 个服务，难道要开 100 个公网 IP 吗？**Ingress 解决了“一个公网 IP 管理成百上千个服务”的问题。**

#### 1. 详细流程（以访问网页为例）
假设你有两个域名：`shop.com`（商店）和 `mail.com`（邮箱）。

1.  **入口点：** 整个机房入口处有一个 **负载均衡器（LB）**，它拥有唯一的**公网 IP**（如 `8.8.8.8`）。
2.  **前台接机：** 所有的流量（不管找谁）都先发给 `8.8.8.8` 上的 **Ingress Controller**（本质上是一个超级强大的 Nginx）。
3.  **拆包查看（L7 应用层）：** Ingress 收到 HTTP 请求后，会撕开包看一眼里面的 **Host（域名）** 或 **Path（路径）**。
4.  **分发：**
    *   看到包头写着 `Host: shop.com` -> 转发给 `Service_Shop`。
    *   看到包头写着 `Host: mail.com` -> 转发给 `Service_Mail`。

#### 2. 作用
*   **路由管理：** 就像酒店前台，根据你找谁，把你带到不同的房间。
*   **节省资源：** 几十个域名可以共用同一个公网 IP。
*   **安全加固：** 可以在这一层统一做 SSL 证书（HTTPS）的卸载。

---

### 四、 完整逻辑链路大串联（从外到内）

当你在浏览器输入 `https://shop.com/order` 并回车时，发生了什么？

1.  **公网阶段：** DNS 把 `shop.com` 解析到公网 IP `8.8.8.8`。
2.  **进入大门（Ingress）：** 流量到达云网关，交给 **Ingress**。Ingress 看了一眼：*“哦，是找 shop 服务的 /order 路径，去叫 `Service_Shop` 接客。”*
3.  **内部调度（Service）：** `Service_Shop` 收到请求，它看了一眼手下的 Pod：*“Pod_1 好像比较闲，包给你了。”*
4.  **封包运输（SDN/VXLAN）：** 
    *   物理机发现 Pod_1 在另一台物理机上。
    *   **大信封：** 把请求套上物理机 IP 的大信封。
    *   **运输：** 通过机房交换机送到目的地。
5.  **最终处理：** 物理机撕开大信封，交给 **Pod_1**。Pod_1 里的 **容器** 处理完请求，原路返回。

### 总结笔记：
*   **容器 IP：** 它是“小包”上的门牌号，集群内可见。
*   **Service：** 它是集群内的“导医台”，确保 Pod 怎么换，内部访问都不乱。
*   **Ingress：** 它是集群的“总服务台”，根据域名把公网流量领进门。
*   **SDN：** 它是底层的“快递员”，负责在物理机之间搬运这些套娃包。
