# Kubernetes (K8s) 核心架构学习笔记

## 一、 为什么引入 K8s？（背景与初衷）

在没有 K8s 之前，我们虽然有了 **Docker（容器）**，解决了环境打包的问题，但在大规模生产环境下，我们面临以下“噩梦”：
1.  **手动挡难题：** 1000 个容器，哪台机器坏了？哪台机器空着？手动部署太累。
2.  **服务发现：** 容器 IP 经常变，A 服务怎么稳定地找到 B 服务？
3.  **自愈需求：** 半夜容器崩了，谁来把它重启？
4.  **动态扩容：** 流量突然翻倍，怎么瞬间变出 100 个副本？

**K8s 的本质：** 它是一个 **“容器编排平台”**。它像是一个**超级管家**，你只需要告诉它“我要跑 5 个登录服务”，剩下的找机器、分发代码、监控健康、自动修复，全部由它完成。

---

## 二、 核心概念逻辑图谱（按运行逻辑排序）

我们将图片中的 9 个概念分为四组，按照从“硬件”到“应用”的顺序引入。

### 第一组：算力底座（计算资源）

#### 1. Node（节点）
*   **定义：** K8s 集群中的一台机器（可以是物理机或云服务器 VM）。
*   **角色：** 它是“干活的工人”。一个集群由多个 Node 组成。
*   **逻辑：** 所有的容器最终都要跑在 Node 上。

#### 2. Pod（豆荚）
*   **定义：** K8s 中**最小的调度单位**。
*   **逻辑：** K8s 不直接操作容器，而是把一个或多个容器（比如你的 Java 程序 + 日志采集插件）打包进一个 Pod。
*   **特征：** 同一个 Pod 里的容器共享网络和存储。**Pod 是有生命周期的，坏了就丢弃，由 K8s 重新创建新的。**

---

### 第二组：控制器（应用管理）

既然 Pod 随时可能死掉，我们需要“监工”来管理它们。

#### 3. Deployment（无状态部署）
*   **定义：** 管理 Pod 的最常用工具。
*   **作用：** 负责 Pod 的**多副本、滚动更新、自愈**。
*   **逻辑：** 你定义“我要 3 个 Web 副本”，Deployment 就会盯着这 3 个 Pod。如果死了一个，它立刻在别的 Node 上补一个。它适合“无状态”应用（比如 Web 前端、API 接口）。

#### 4. StatefulSet（有状态部署）
*   **定义：** 专门管理**有状态**应用。
*   **逻辑：** 如果你要跑数据库（MySQL、Redis），每个副本都有自己的编号（mysql-0, mysql-1）和持久化数据。StatefulSet 保证这些副本的**顺序和唯一性**。

---

### 第三组：网络与访问（服务发现）

Pod 的 IP 地址是动态的，怎么访问它们？

#### 5. Service（服务）
*   **定义：** Pod 的**稳定入口（固定 IP）**。
*   **逻辑：** 它像一个**负载均衡器**。不管后面的 Pod 怎么死、怎么变，Service 的 IP 不变。请求发给 Service，它会自动转发给后端的某个健康 Pod。

#### 6. Ingress（入口）
*   **定义：** 集群的 **“大门”**。
*   **逻辑：** Service 只能在集群内部访问。如果你想让互联网用户通过域名（如 `www.myapp.com`）访问你的服务，就需要 Ingress 来做 **七层路由（域名/路径转发）**。

---

### 第四组：数据与配置（环境隔离）

程序不仅要跑起来，还需要配置和存储数据。

#### 7. Volumes（存储卷）
*   **定义：** Pod 的**外部硬盘**。
*   **逻辑：** Pod 死了数据会丢失。Volumes 允许你挂载外部存储（如我们之前讲的云盘、NAS），保证** Pod 换了机器，数据还能找回来**。

#### 8. ConfigMap（配置字典）
*   **定义：** 存放**非敏感配置**（如配置文件、环境变量）。
*   **逻辑：** 实现“代码与配置分离”。你修改 ConfigMap，不需要重新打包程序镜像，Pod 就能加载新配置。

#### 9. Secret（机密字典）
*   **定义：** 存放**敏感数据**（如数据库密码、密钥、证书）。
*   **逻辑：** 和 ConfigMap 类似，但数据会加密存储，安全性更高。

---

## 三、 总结：它们是如何协同工作的？

当你部署一个正式项目时，完整链路是这样的：

1.  **基础设施：** 准备好几台 **Node** 组成集群。
2.  **应用定义：** 创建一个 **Deployment**，描述你需要的 **Pod** 镜像和副本数。
3.  **注入配置：** 通过 **ConfigMap** 传入数据库地址，通过 **Secret** 传入访问密码。
4.  **持久化：** 给 Pod 挂载 **Volumes**，确保日志和上传的文件不丢失。
5.  **内部通信：** 创建 **Service**，让内部其他组件能找到这个应用。
6.  **对外暴露：** 配置 **Ingress**，绑定域名，让用户可以从浏览器访问。

# Kubernetes(K8s) 架构

Master-Worker 架构

Master 负责管理，指挥监控工作节点工作

## Worker-Node 工作结点

每一个Node，除了运行服务的Pod之外，还包含 Kubelet 和 Kube-proxy 和 容器运行时

容器运行时 ： 负责拉取镜像，运行容器、启动、停止容器 ， 所以必须要有这个组件
	常见的容器运行时：
	1. Docker-Engine
	2. Containerd
	3. CRI-O

Kubelet : 
1. 负责管理每一个Node上的结点
2. 从api-server组件中，获取pod的新的规范；
3. 监控结点的工作状态


Kube-proxy : 
1. 为 pod 结点提供网络代理和负载均衡服务
	1. 负载均衡：当集群之间想要通信，需要借助Servive，通过 service.yml 写入 API Server，表示后续请求某种集群的服务，Kube-proxy就会监听响应，将流量分发的规则写入每一个pod，实现负载均衡
		> 发给谁
	2. 网络代理：实现结点进行网络通信,可以实现服务优先路由到同一个node中的pod，直接通过容器服务名和端口实现通信，省去网络开销
		> 怎么发过去


## Master—Node 管理节点

对Worker—Node组成的集群进行管理，比如
- 怎么把pod调度到不同的node上
- 怎么监控结点状态
- 怎么删除、增加一个Node

四个基本组件：

kube-api-server(基础设施的管理中心) ectd controllerManager Scheduler

kube-apiserver:
- 提供集群需要的API接口，所有的组件都是通过API通信的；管理Node资源（增删改查）；提供认证鉴权（通过kubectl命令行工具，或者Dashboard工具，去操作apiserver的时候，都会先对身份进行认证）

Scheduler：
- 监控Node的工作状态，如果有的Node资源利用率太高，可以把pod调度到集群中的其他Node

Controller Manager 控制器管理器：
- 也是监控资源对象的工作状态，不过侧重结点是不是正常运行，如果出故障了，做处理，比如替换、重启等等

Etcd：
- 键值对的存储系统（和redis一样），存储集群中所有资源对象的状态信息，比如pod故障，pod新增

（ccm Cloud Controller Manager 云控制管理，是使用某些云服务商的集群时，提供的控制云平台的API管理器）


# MiniKube 环境搭建

> **目标**：通过 miniKube 和 kubetcl 搭建一个小型的 k8s 系统

### miniKube

是一个轻量级的 Kubernetes 的实现，可以在本地创建一个虚拟机，部署仅包含一个结点Node的集群。

> 正常的K8s系统的云服务，每一个集群都会有多个Master-Node和Woke-Node，以实现分布式软件的高并发、高可用。但是这就需要多台虚拟机或者物理机，以及对应的网络配置和存储配置，很复杂
> 
> 其实还有一些别的本地实现k8s系统的轻量级工具 k3s k3d kind


### Kubectl

命令行工具，用来和集群进行交互，具体是通过和Master-Node的apiServer交互
