# 可观测性

可观测性是指通过观察系统的外部输出（日志、指标、链路追踪）来推断其内部状态的能力。

为什么需要可观测性：
1. 服务数量多，导致故障定位困难
2. 传统的监控内存、CPU，无法解决实际的问题，比如请求为什么很慢

如何实现可观测性：
1. 日志：记录时间、事件、事件发出对象
2. 指标：统计数据，比如每秒有多少请求，请求的错误率是多少
3. 链路追踪：追踪一个请求的路径，每一个服务的耗时
> 通过指标发现服务发现问题，通过链路追踪，找到请求中每一个服务的耗时，通过日志找到这个服务发生了什么。

# 日志监控

记录时间、事件、事件发出对象。一般需要带上一个traceID，用来服务链路中，标识服务是同属于一次请求的。

对于结构化的数据，可以通过 JSON 结构化日志，更好检索信息。

一般日志中需要携带：时间戳timestamp、类别level（info、debug...）、服务service、追踪链路用的traceId、具体消息message

日志采集的方案

| 特性   | ELK 方案                            | Loki+Promtail方案           |
| ---- | --------------------------------- | ------------------------- |
| 组件   | Elasticsearch + Logstash + Kibana | Loki + Promtail + Grafana |
| 存储方式 | 全文索引（倒排索引）                        | 仅索引标签，日志内容压缩存储            |
| 查询语言 | Lucene 查询语法                       | LogQL（类似 PromQL）          |
| 资源占用 | 较高（ES 需要大量内存）                     | 较低（无全文索引）                 |
| 适用场景 | 复杂日志文本搜索                          | Prometheus 同一生态，关注标签而非全文  |
| 典型架构 | Filebeat → Logstash → ES → Kibana | Promtail → Loki → Grafana |


实现架构![[第八讲 服务治理(2).pdf#page=8&rect=91,15,924,464|第八讲 服务治理(2), p.8]]
> 1. logstash 用于接收非结构化、复杂的、原始的数据。进行数据的解析、过滤、结构化，之后可以通过MQ来把结构化的数据交给ES，在数据的接受上，也可以通过MQ来接受
> 2. ES 用来接收结构化、简单的数据。通过全文索引（倒排索引），加快通过关键字检索日志的过程
> 3. Kibana 用于检测 ES 中的数据，进行可视化的展示

# 指标监控

对系统的一些状态和行为，进行数据上的统计。

关键的统计数据指标：
1. 延迟，服务处理请求花费的时间，通常使用百分位数据（P95,P99）
	这是因为，在大规模的应用中，如果关注平均数或者中位数，就容易忽略较大规模请求的延迟，比如100万个用户，不能只保证一半的用户延迟低。
2. 流量：QPS TPS
3. 错误：请求的错误率
4. 资源饱和度：CPU、内存、连接数

> Prometheus 用来拉取指标数据
> Grafana 用来可视化指标


# 链路追踪

链路追求记录分布式系统中，一个请求的完整调用路径，路径中所有服务的耗时和状态。

> 可以看到系统具体是哪一个模块耗时多，同时传递一个 traceID，供日志记录一次请求的链路

# 服务网格

服务网格：将微服务治理下沉到基础设施层

传统的微服务的服务治理：每一个微服务都要集成SDK，比如重试、熔断、降级、服务发现，然后实现相同的治理逻辑，代码冗余，后续修改就要全修改，耦合程度高

服务网格的思想：把服务治理从业务代码抽离，下沉到基础设施层。通过 Sidecar边车 代理拦截进出的流量。进到服务内部的请求，和服务内部调用别的服务的请求，都被代理实例拦截，从而实现对于服务治理逻辑的解耦。

# 无服务器架构

无服务器架构是一种云计算执行模型，开发者将代码上传到平台，平台会自动管理服务器、伸缩，实现高可用，节省钱。
**无服务不是没有服务器，是开发者感知不到服务器**

## Faas

服务器运行无状态的、可以由事件驱动的函数。
Scale-to-zero：Faas平台的能力，当函数或者服务没有请求的时候，平台把实例数量缩减为0

开发流程：
1. 编写函数
2. 上传函数到云平台
3. 配置函数的触发器
4. 执行：当事件发生时
5. 伸缩：申请请求的并发数调整实例数量
6. 空闲回收：函数闲置一段时间，销毁实例。

> 1. Faas 并不能代替完整的后端服务，那么逻辑很重、需要长期在线的服务，还是在服务器中运行，或者以容器运行
> 2. Faas 函数冷启动：最开始的第一次调用，包含创建函数实例，时间比较长
## Baas

使用云托管的第三方服务。把传统的后端服务转为一个 SDK，提供调用的API，比如传统的数据库，用户登录等等。前端应用可以直接使用API，对数据库进行操作，不需要 controller-service...的分层
