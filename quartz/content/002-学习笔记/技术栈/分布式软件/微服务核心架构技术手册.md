## #1. Nacos 服务发现

### 技术定义
Nacos (Dynamic Naming and Configuration Service) 是一个更易于构建云原生应用的动态服务发现、配置管理和服务管理平台。其服务发现功能允许分布式系统中的实例在启动时==向中心注册信息，并允许消费方动态获取可用实例列表==。

### 使用原因与价值
1. **自动注册与注销**：解决容器化部署（如 Docker、K8s）下 IP 地址频繁变动的问题。
2. **健康检查**：自动剔除宕机实例，确保流量只分配给正常的节点。
3. **负载均衡支持**：配合客户端组件，实现流量在多个实例间的合理分配。

### 代码实现
**1. 依赖配置 (Maven)**
```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-discovery</artifactId>
</dependency>
```

**2. 核心配置 (application.yml)**
```yaml
spring:
  application:
    name: service-provider # 服务标识，用于被消费方查找
  cloud:
    nacos:
      discovery:
        server-addr: 127.0.0.1:8848 # Nacos 服务端地址
        ephemeral: true # 默认 true，表示临时实例（AP 模式）
```

### 原理深度解析
- **注册机制**：客户端启动时通过 REST 请求将自身元数据（IP、Port、服务名、权重等）发送至 Nacos 注册表。
- **心跳维持**：客户端默认每 5 秒发送一次心跳。若 Nacos 15 秒未收到心跳则将实例设为不健康状态，30 秒未收到则剔除实例。
- **服务发现**：消费方通过服务名拉取列表，并在本地缓存。Nacos 采用 **UDP 推送 + 定期拉取** 的双重机制保证数据一致性。

## #2. Nacos 配置中心

### 技术定义
配置中心是分布式系统中的统一参数管理组件，负责将分散在各个微服务中的静态配置文件提取到云端，实现配置的集中管理与实时生效。

### 使用原因与价值
1. **动态热更新**：无需重启 JVM 即可改变系统行为（如开关业务功能）。
2. **版本控制**：支持查看历史版本并快速回滚配置。
3. **多格式支持**：支持 YAML、Properties、JSON 等多种格式转换。

### 代码实现
**1. 依赖配置**
必须引入 Bootstrap 依赖，否则无法在 Spring 容器启动初期读取远程配置。
```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-nacos-config</artifactId>
</dependency>
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-bootstrap</artifactId>
</dependency>
```

**2. 预加载配置 (bootstrap.yml)**
```yaml
spring:
  application:
    name: order-service
  cloud:
    nacos:
      config:
        server-addr: 127.0.0.1:8848
        file-extension: yaml # 配置文件的 DataID 规则：${spring.application.name}.${file-extension}
```

**3. Java 动态刷新实现**
```java
@RestController
@RefreshScope // 核心：监听配置变更并重新实例化 Bean
public class ConfigController {
    @Value("${business.promotion.enabled:false}")
    private boolean promotionEnabled;

    @GetMapping("/status")
    public String getStatus() {
        return "当前活动开启状态: " + promotionEnabled;
    }
}
```

### 原理深度解析
- **长轮询机制**：客户端发起一个请求到服务端，服务端如果发现配置没变会挂起该请求（默认 30 秒），直到配置变动或请求超时再返回。这比定期拉取更省流量且更实时。
- **配置一致性**：配置数据持久化在 MySQL 中，Nacos 集群节点通过同步机制保证配置的一致。

## #3. Namespace 隔离

### 技术定义
Namespace（命名空间）是 Nacos 提供的逻辑层面的隔离机制。它通常被定义为最顶层的资源划分，用于区分不同的开发、测试和生产环境。

### 使用原因与价值
1. **环境冲突规避**：防止开发环境的服务无意间注册到测试环境，导致流量调用链路错乱。
2. **权限控制**：可以针对特定的 Namespace 分配不同的访问权限。

### 代码实现 (bootstrap.yml)
```yaml
spring:
  cloud:
    nacos:
      discovery:
        namespace: d8f4-4a21-9e8c-83b3 # 填入 Nacos 控制台生成的 Namespace ID
      config:
        namespace: d8f4-4a21-9e8c-83b3
```

## #4. Spring Cloud Gateway

### 技术定义
Spring Cloud Gateway 是基于 Project Reactor、Spring WebFlux 和 Spring Boot 构建的 API 网关。它旨在为微服务架构提供一种简单有效且统一的 API 路由管理方式。

### 核心要素说明
1. **Route (路由)**：网关的基本模块，包含 ID、目标 URI、断言和过滤器。
2. **Predicate (断言)**：Java 8 的 Predicate，用于匹配 HTTP 请求中的任何内容（如 Path, Method, Header）。
3. **Filter (过滤器)**：在发送下游请求之前或之后修改请求/响应。

### 代码实现
**1. 路由与权重配置 (application.yml)**
```yaml
spring:
  cloud:
    gateway:
      routes:
        - id: weight_v1
          uri: lb://service-v1
          predicates:
            - Path=/api/**
            - Weight=group1, 8
        - id: weight_v2
          uri: lb://service-v2
          predicates:
            - Path=/api/**
            - Weight=group1, 2
```

**2. 全局过滤器实现 (日志埋点)**
```java
@Component
public class TraceLogFilter implements GlobalFilter, Ordered {
    @Override
    public Mono<Void> filter(ServerWebExchange exchange, GatewayFilterChain chain) {
        long start = System.currentTimeMillis();
        return chain.filter(exchange).then(Mono.fromRunnable(() -> {
            long duration = System.currentTimeMillis() - start;
            System.out.println("请求路径: " + exchange.getRequest().getURI().getPath() + " 耗时: " + duration + "ms");
        }));
    }
    @Override
    public int getOrder() { return -1; }
}
```

**3. 限流配置 (基于 Redis)**
```yaml
filters:
  - name: RequestRateLimiter
    args:
      redis-rate-limiter.replenishRate: 10 # 每秒填充速率
      redis-rate-limiter.burstCapacity: 20 # 桶总容量
      key-resolver: "#{@ipKeyResolver}"
```

## #5. OpenFeign 声明式调用

### 技术定义
OpenFeign 是一款声明式的 HTTP 客户端，它封装了 Ribbon（负载均衡）和 RestTemplate 的功能。通过定义接口并添加注解，开发者可以像调用本地 Java 方法一样发起 HTTP 请求。

### 使用原因与价值
1. **代码解耦**：将网络请求细节（URL、参数处理）从业务代码中剥离。
2. **强类型支持**：利用 Java 接口定义请求，减少参数传递错误。
3. **负载均衡集成**：默认整合了 Spring Cloud LoadBalancer，支持在多个实例间轮询。

### 代码实现
**1. 依赖配置**
```xml
<dependency>
    <groupId>org.springframework.cloud</groupId>
    <artifactId>spring-cloud-starter-openfeign</artifactId>
</dependency>
```

**2. 定义调用接口**
```java
@FeignClient(name = "user-service") // 指定 Nacos 注册的服务名
public interface UserFeignClient {
    @GetMapping("/users/{id}")
    UserDTO getUserById(@PathVariable("id") Long id);
}
```

**3. 启用服务调用**
在启动类上增加注解。
```java
@SpringBootApplication
@EnableFeignClients
public class OrderApplication {
    public static void main(String[] args) {
        SpringApplication.run(OrderApplication.class, args);
    }
}
```

### 原理深度解析
- **动态代理**：Feign 在启动时会为接口创建一个 JDK 动态代理实例。
- **请求构建**：当调用接口方法时，代理对象会将方法名、参数解析为 HTTP 请求报文。
- **负载均衡器拦截**：请求发出前，拦截器会根据接口上的 `name`（服务名）去 Nacos 缓存的列表里选择一个真实的 IP 地址替换掉服务名，最终发起真实的 HTTP 通信。

## #6. Sentinel 流量治理

### 技术定义

Sentinel 是阿里开源的面向分布式服务架构的高可用防护组件。它以“流量”为切入点，从**流量控制、熔断降级、系统负载保护**等多个维度保护服务的稳定性。

### 使用原因与价值

1. **防止雪崩**：当下游服务响应过慢时，及时熔断，避免上游服务因等待超时而耗尽线程池资源。
    
2. **秒杀削峰**：在电商大促场景下，将超过系统承载能力的请求直接拒绝，保护数据库不被压垮。
    
3. **平滑限流**：支持匀速排队模式，处理突刺流量。
    
4. **与 Feign 深度集成**：可以为每一个远程调用接口配置兜底逻辑（Fallback）。
    

### 代码实现

**1. 依赖配置**


```xml
<dependency>
    <groupId>com.alibaba.cloud</groupId>
    <artifactId>spring-cloud-starter-alibaba-sentinel</artifactId>
</dependency>
```

**2. 整合 Feign 实现服务降级 (application.yml)**


```yaml
feign:
  sentinel:
    enabled: true # 开启 Sentinel 对 Feign 的支持
```

**3. 定义降级处理函数 (Fallback)**


```yaml
@FeignClient(name = "stock-service", fallback = StockFallback.class)
public interface StockFeignClient {
    @GetMapping("/stock/deduct")
    String deduct(@RequestParam("productId") Long productId);
}

@Component
class StockFallback implements StockFeignClient {
    @Override
    public String deduct(Long productId) {
        return "库存服务繁忙，请稍后再试（触发降级）";
    }
}
```

### 原理深度解析

- **资源与规则**：Sentinel 中一切皆“资源”。通过 Slot Chain（插槽链）模式，依次经过权限校验、流量监控、降级规则判断。
    
- **熔断策略**：支持按 **慢调用比例**、**异常比例** 或 **异常数** 进行熔断。当触发阈值时，熔断器进入 Open 状态，拒绝所有请求；经过探测恢复期后进入 Half-Open 状态尝试恢复。
    
- **规则持久化**：生产环境下，通常将限流规则存储在 **Nacos** 中。Sentinel Dashboard 修改规则后推送到 Nacos，客户端通过监听 Nacos 配置实现动态更新，解决重启丢失问题。

## #7. Seata 分布式事务

### 技术定义

Seata (Simple Extensible Autonomous Transaction Architecture) 是一款开源的分布式事务解决方案，致力于在微服务架构下提供高性能和简单易用的分布式事务服务。

### 使用原因与价值

1. **数据一致性**：解决“下单成功但扣库存失败”或“扣库存成功但订单创建失败”导致的业务逻辑错误。
    
2. **业务无侵入 (AT 模式)**：开发者只需关注业务代码，事务的提交与回滚由 Seata 框架自动完成。
    
3. **高性能**：相比于传统的 XA 强一致性协议，Seata AT 模式在性能上更契合互联网电商场景。
    

### 代码实现

**1. 核心注解使用 (业务逻辑层)**


```java
@Service
public class OrderService {
    @Autowired
    private StockFeignClient stockClient;
    @Autowired
    private OrderMapper orderMapper;

    @GlobalTransactional(name = "create-order-tx", rollbackFor = Exception.class)
    public void createOrder(Order order) {
        // 1. 本地保存订单
        orderMapper.insert(order);
        // 2. 远程调用库存服务 (另一个数据库)
        stockClient.deduct(order.getProductId());
        
        // 如果此处抛出异常，1和2都会回滚
    }
}
```

**2. 配置事务分组 (file.conf 或 Nacos)**


```yaml
seata:
  tx-service-group: my_test_tx_group # 事务分组名称
  service:
    vgroup-mapping:
      my_test_tx_group: default # 映射到真实的 Seata Server 集群
  registry:
    type: nacos # 通过 Nacos 发现 Seata TC 服务端
```

### 原理深度解析 (AT 模式)

- **一阶段 (Execution)**：Seata 代理数据源，拦截业务 SQL。在业务 SQL 执行前，保存“数据旧快照”；执行后，保存“数据新快照”并插入 undo_log 表。这些操作在同一个本地事务中提交。
    
- **二阶段 (Commit/Rollback)**：
    
    - **成功**：TC 通知各分支清理 undo_log，异步完成，性能极高。
        
    - **失败**：TC 通知各分支回滚。分支根据 undo_log 的旧快照进行补偿式还原。
        
- **写隔离**：通过全局锁（Global Lock）机制，防止多个分布式事务并发修改同一行记录导致的脏写。
