# 线程和进程

[[进程和线程#进程和线程的区别是什么|之前的笔记：进程和线程的区别是什么]]

上面的笔记讲解了在操作系统视角下的线程，包括所拥有的资源，存在的形式，通信的方
式。而在JVM中的线程，会稍微有一些区别。

在古早的版本，Java使用的线程，是一种用户级别线程，由用户程序来控制线程执行的生命周期和调用（也就是JVM管理），JDK1.2之后，Java的线程使用的是操作系统提供的内核级别的线程，由OS内核负责管理线程调度。

> 用户级别线程ULT：用户程序管理线程，无法执行异步IO操作，无法使用CPU的多核
> 内核级别线程KLT：操作系统管理线程，可以使用多核，创建和调度的开销比较高，因为涉及到系统调用，CPU切换状态
> 用户级别线程需要和内核级别线程关联映射：
> - 1对1：一个用户线程对应一个内核线程，本质上来说，线程都由OS进行管理
> - 多对1：多个用户线程对应一个内核线程，用户线程完全由用户程序进行管理调度，无法多核执行，并且进程的一个线程阻塞，会导致进程直接被阻塞
> - 多对多：用户进程和OS一起管理线程


# 线程具有的资源

**这里讨论的线程是JVM管理的用户级线程**

在JVM管理的内存空间中，线程可以共享JVM的堆区和方法区，拥有自己的本地方法栈（Native Method Stack）、虚拟栈、程序计数器

- 虚拟机栈：存放线程调用Java方法的栈帧，包括数据保护和程序流程控制
- 本地方法站：虚拟机使用到的本地方法（Native Method）的栈帧
- PC：保证线程调度之后，能从正确的位置继续执行

- 堆区：JVM中管理的内存空间，存放所有对象、字符串常量池
- 方法区：包含类定义信息、静态变量、JIT编译后的机器代码

- 至于线程运行的上下文环境的那些寄存器，由内核线程的TCB和内核栈保存。

# 并发编程中使用的锁

## Synchronized

同步锁锁的目标是一个实例对象，包括堆区中的普通 Java 对象、以及Class 类的对象。
具体的来说，锁应用的内容，是堆区中对象的对象头字段（2字，16字节，128位）其中的Mask Word（1字，8字节，64位）。

同步锁随着锁竞争的情况，会不断修改Mask Word中的bit位，这一过程看作是锁升级的过程。
1. 无锁状态（Initial State）
	这是对象的初始状态。
	- **Mark Word 布局：** 25 bit(空) + 31 bit(hashCode) + 1 bit(空) + 4 bit(分代年龄) + 1 bit(是否偏向:0) + 2 bit(锁标志位:01)。
	- **关键点：** 这里的 `hashCode` 是**延迟计算**的。只有你调用了 `Object.hashCode()` 或 `System.identityHashCode()` 后，这 31 位才会被填充。如果没调用，它就是全 0。
 2. 偏向锁（Biased Locking）
	**目标：** 消除无竞争情况下的同步原语，连 CAS 都不做。
	- **Mark Word 布局：** **54 bit(线程 ID)** + 2 bit(Epoch) + 4 bit(分代年龄) + 1 bit(是否偏向:1) + 2 bit(锁标志位:01)。
    - **关于 hashCode 的冲突：** 偏向锁位图里没有空间存 hashCode 了。所以，如果一个对象已经计算了 identityHashCode，它**永远无法**进入偏向锁状态；如果偏向锁过程中计算了 hashCode，偏向锁会立刻**撤销**并升级为重量级锁。
    - **重入逻辑：** 线程 A 再次进入时，只需看 Mark Word 前 54 位是不是自己的 ID。是，直接过；不是，说明有竞争，开始撤销。
 3. 轻量级锁（Lightweight Locking）
	**目标：** 竞争不激烈时，通过自旋避免线程在内核态和用户态之间切换。
	- **升级触发：** 当线程 B 尝试获取偏向 A 的锁时，偏向锁撤销，升级为轻量级锁。
	- **底层操作：**
    1. 线程在自己的栈（Stack Frame）中开辟一块空间叫 **Lock Record**。
    2. 把对象头里的 Mark Word 拷贝到 Lock Record 中（称为 Displaced Mark Word）。
    3. **核心 CAS：** 线程尝试用 CAS 操作将对象头里的 64 位内容修改为**指向自己 Lock Record 的指针**。
    
    - **自旋（Spinning）：** 如果 CAS 失败，说明锁被占用了。线程不会阻塞，而是执行一段空循环（自旋）。
    - **自适应自旋：** JVM 会根据上一次自旋是否成功来决定这次自旋多久。如果一直失败，就会发起“膨胀”。
        

4. 重量级锁（Heavyweight Locking）
	**目标：** 彻底解决高并发下的 CPU 空转问题。
	- **底层结构：** 对象头里的 62 位指向一个 C++ 编写的 `ObjectMonitor` 对象。
    
    - **为什么叫“重量级”？** 因为线程获取不到锁会进入 `_WaitSet` 或 `_EntryList` 队列，此时线程会被 **挂起（Park）**。挂起和唤醒需要操作系统介入，涉及用户态到内核态的转换，上下文切换非常耗时。
    - **hashCode 去了哪里？** 在重量级锁里，原来的 Mark Word 信息（含 hashCode）被保存在了 `ObjectMonitor` 对象内部。

 5. 锁升级的“单向性”
	- **核心规则：** 锁可以升级，但通常**不可以降级**（虽然在 GC 的安全点有时会发生降级，但在正常的业务逻辑执行中，你可以认为它是不可逆的）。

> [!NOTE] 总结
> 1. 锁竞争不激烈的情况下，通过线程ID或者用户态的CAS操作来获取锁。其中用户态CAS是自旋操作
> 2. 锁竞争激烈的情况下，自旋的CAS浪费CPU时间，通过指向C++写的一个对象，对象内部存放当前获取锁的线程ID，阻塞队列。来完成线程的挂起和后续的唤醒操作，涉及到用户态和内核态的切换
	
## ReentrantLock

不通过对象头中的 Mask Word标识锁的状态和获取方式。通过  `AbstractQueuedSynchronize` （AQS）来组织所有视图获取锁的线程。

核心特点是：
1. 锁的过程中，可以**响应中断**
2. 可以指定获取锁的**尝试时间** `tryLock(time)`
3. 可以在获取锁失败的逻辑下做别的事情 `boolean tryLock()`
4. 通过用户态CAS获取锁
5. 可以设置**公平**竞争锁还是**不公平**，同步锁都是不公平

```java
// 默认构造函数：创建的是【非公平锁】
ReentrantLock nonFairLock = new ReentrantLock(); 

// 传入 true：创建的是【公平锁】
ReentrantLock fairLock = new ReentrantLock(true); 

// 传入 false：创建的是【非公平锁】
ReentrantLock nonFairLockExplicit = new ReentrantLock(false);
```

具体锁的流程为：
-  一个进程尝试 `lock()`
	1. `reentrantLock` 对象中有一个线程安全队列，如果公平竞争，线程放到队列中，如果不公平，线程直接尝试获取锁
	2. 对于队列中的线程，大家都在尝试自旋，判断自己是不是头节点后面的结点（头节点中无信息，头节点后面的结点就是队列中的第一个线程），只有第一个线程有资格竞争锁（不公平竞争下，还有新来的线程），当竞争锁失败，或者自旋次数过多，就会挂起
	3. 竞争锁的过程，是通过用户态的CAS，尝试把`reentrantLock` 对象中的状态位`state`从0改为1。如果是一个线程的重入，直接状态位+1
	4. 状态位为0的时候，会唤醒队列的第一个线程
- 一个线程尝试 `lockInterruptibly()`
	- 在队列中被挂起的时候，队列可以唤醒这个线程；而线程被唤醒的时候也会检查是被中断信号唤醒的，还是被叫起来抢锁的
- 一个线程尝试 `tryLock()`
	- 线程直接CAS，不会进入队列，失败就返回false，线程直接其他逻辑
- 一个线程尝试 `tryLock(time)`
	- 线程会进入队列，在尝试获取锁失败，或者自旋一定次数后，会挂起，但是区别在于挂起的时候，指定一个剩余时间；时间到和通知抢锁都会唤醒线程，所以起来之后先检查，如果是时间到了，就返回false

> Reentrant 还支持定义多个条件变量，同步编程更适合
