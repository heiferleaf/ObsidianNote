### 2.2 简答题

**（1）** 请解释 Java 泛型中 `<? extends T>` 与 `<? super T>` 的区别，并说明各自适合用于"生产者"还是"消费者"场景（即 PECS 原则）。

- `<? extends T>`（上界通配符）：表示 T 或 T 的某个子类型。只能**读取**（读出来的元素保证是 T），不能写入（编译器不知道具体子类型）。适合**生产者（Producer）** 场景，即从集合中取出数据。
- `<? super T>`（下界通配符）：表示 T 或 T 的某个父类型。可以**写入** T 及其子类（写入安全），读出来只能当 Object 使用。适合**消费者（Consumer）** 场景，即向集合中放入数据。

**（2）** AOP 有哪五种核心通知类型（Advice）？请分别用一句话描述其触发时机。

| 通知类型              | 触发时机                                 |
| ----------------- | ------------------------------------ |
| `@Before`         | 目标方法**执行前**触发                        |
| `@AfterReturning` | 目标方法**正常返回后**触发（可获取返回值）              |
| `@AfterThrowing`  | 目标方法**抛出异常后**触发（可获取异常对象）             |
| `@After`（Finally） | 目标方法**执行完毕后**触发，无论正常还是异常             |
| `@Around`         | **环绕**目标方法执行，可在调用前后自定义逻辑，可控制是否执行目标方法 |

**（3）** 什么是函数式编程中的"闭包（Closure）"？在 Java Lambda 中，为什么捕获的外部局部变量必须是 effectively final 的？请从 JVM 实现层面简要解释。

**闭包**是指一个函数与其捕获的外部环境变量共同构成的整体，使函数可在其定义作用域之外访问那些变量。Java Lambda 是闭包的一种实现。

**为什么必须 effectively final？**

从 JVM 实现层面看：Lambda 表达式被编译为一个合成方法，捕获的局部变量是**按值复制**（值传递）到 Lambda 实例的字段中的，而非按引用共享。局部变量存储在栈帧（Stack Frame）上，Lambda 对象存储在堆上；当方法返回后，栈帧销毁，若允许修改变量，堆上的 Lambda 持有的副本与栈上原变量就会出现**数据不一致**的问题。强制 effectively final 保证了"复制的值"与"原始值"始终一致，从根本上避免了这种歧义。这也是满足了使用纯函数作为函数式编程的特征。

## 第三题：代码分析题

### 3.1 泛型分析

阅读以下代码，回答问题：

```java
public class TypeTokenUtil {
    public static <T> Class<T> getTypeToken(Class<T> clazz) {
        return clazz;
    }

    public static void main(String[] args) throws Exception {
        List<String> stringList = new ArrayList<>();
        List<Integer> intList = new ArrayList<>();

        // 语句A
        System.out.println(stringList.getClass() == intList.getClass());

        // 语句B
        Field field = MyContainer.class.getDeclaredField("data");
        Type genericType = field.getGenericType();
        if (genericType instanceof ParameterizedType) {
            ParameterizedType pt = (ParameterizedType) genericType;
            System.out.println(pt.getActualTypeArguments()[0]);
        }
    }
}

class MyContainer {
    private List<String> data;
}
```

**（1）** 语句 A 的输出结果是什么？从类型擦除角度解释原因。 
**（2）** 语句 B 的输出结果是什么？为什么字段上的泛型信息可以在运行时获取，而局部变量的不行？

**（1）语句 A 输出：`true`**

原因：Java 泛型采用**类型擦除**（Type Erasure），编译后 `List<String>` 和 `List<Integer>` 的类型参数均被擦除，运行时都是裸类型 `java.util.ArrayList`，两者 `getClass()` 返回完全相同的 `Class` 对象，故比较结果为 `true`。

**（2）语句 B 输出：`class java.lang.String`**

字段（Field）的泛型信息保存在**字节码的 Signature 属性**中，这是编译器专门为反射用途写入的元数据，与运行时类型擦除是两个不同层面。通过 `field.getGenericType()` 可读取此元数据，进而拿到 `ParameterizedType` 并获得实际类型参数 `String`。

局部变量的类型信息则完全不写入字节码（局部变量表只存原始类型），所以运行时无法获取。

阅读以下代码：

```java
@Target(ElementType.METHOD)
public @interface Validate {

public class ValidatorProcessor {

        // TODO: 利用反射读取 @Validate 注解并进行参数校验
        // 若第一个字符串参数不满足长度约束，抛出 IllegalArgumentException
        // 并附带注解中定义的 message
```

**（1）** 请补全 `ValidatorProcessor.process()` 方法的实现。  

```java
public class ValidatorProcessor {
    public static void process(Object target, String methodName, Object... args) throws Exception {
        // 1. 获取目标方法（根据参数类型精确匹配，这里简化为按名称查找第一个匹配）
        Method method = null;
        for (Method m : target.getClass().getDeclaredMethods()) {
            if (m.getName().equals(methodName)) {
                method = m;
                break;
            }
        }
        if (method == null) {
            throw new NoSuchMethodException(methodName);
        }

        // 2. 读取 @Validate 注解
        Validate validate = method.getAnnotation(Validate.class);
        if (validate != null && args != null && args.length > 0) {
            // 3. 取第一个字符串参数进行校验
            if (args[0] instanceof String value) {
                int len = value.length();
                if (len < validate.minLength() || len > validate.maxLength()) {
                    throw new IllegalArgumentException(validate.message());
                }
            }
        }

        // 4. 校验通过，执行目标方法
        method.setAccessible(true);
        method.invoke(target, args);
    }
}
```

**（2）** 若 `@Validate` 的 `@Retention` 改为 `RetentionPolicy.CLASS`，上述反射逻辑是否还能正常工作？为什么？

`CLASS` 策略表示注解信息保留在 `.class` 字节码中，但 **JVM 加载类时不会将其读入运行时元数据**，因此通过反射调用 `method.getAnnotation(Validate.class)` 只会返回 `null`，校验逻辑完全失效。必须使用 `RUNTIME` 才能让反射 API 访问注解。

### 3.3 动态代理分析

阅读以下代码：

```java
public interface PaymentService {
    String pay(String orderId, double amount);
}

public class PaymentServiceImpl implements PaymentService {
    @Override
    public String pay(String orderId, double amount) {
        return "Paid " + amount + " for order " + orderId;
    }
}

public class LoggingProxy implements InvocationHandler {
    private final Object target;

    public LoggingProxy(Object target) {
        this.target = target;
    }

    @Override
    public Object invoke(Object proxy, Method method, Object[] args) throws Throwable {
        System.out.println("[LOG] Before: " + method.getName());
        Object result = method.invoke(target, args);
        System.out.println("[LOG] After: " + method.getName() + " -> " + result);
        return result;
    }

    public static <T> T create(T target, Class<T> iface) {
        return iface.cast(Proxy.newProxyInstance(
            target.getClass().getClassLoader(),
            new Class[]{iface},
            new LoggingProxy(target)
        ));
    }
}
```

**（1）** 若执行以下代码，完整打印输出是什么？

```java
PaymentService service = LoggingProxy.create(new PaymentServiceImpl(), PaymentService.class);
String result = service.pay("ORD-001", 99.9);
System.out.println("Result: " + result);
```

```text

[LOG] After: pay -> Paid 99.9 for order ORD-001
Result: Paid 99.9 for order ORD-001
```

**（2）** 如果 `PaymentServiceImpl` 没有实现任何接口，能否使用 JDK 动态代理对其增强？应该怎么做？请简要说明方案并写出关键代码片段。

**`PaymentServiceImpl` 无接口时无法使用 JDK 动态代理**，因为 JDK Proxy 必须基于接口生成代理类。

应改用 **CGLIB**，它通过生成目标类的子类实现代理（需要目标类非 final）：
```java
// 引入 CGLIB 依赖（Spring 已内置）
import net.sf.cglib.proxy.Enhancer;
import net.sf.cglib.proxy.MethodInterceptor;
import net.sf.cglib.proxy.MethodProxy;

public class CglibLoggingProxy implements MethodInterceptor {

    @Override
    public Object intercept(Object obj, Method method,
                            Object[] args, MethodProxy proxy) throws Throwable {
        System.out.println("[LOG] Before: " + method.getName());
        Object result = proxy.invokeSuper(obj, args); // 调用父类（目标类）方法
        System.out.println("[LOG] After: " + method.getName() + " -> " + result);
        return result;
    }

    public static <T> T create(Class<T> targetClass) {
        Enhancer enhancer = new Enhancer();
        enhancer.setSuperclass(targetClass);          // 设置父类为目标类
        enhancer.setCallback(new CglibLoggingProxy());
        @SuppressWarnings("unchecked")
        T proxy = (T) enhancer.create();
        return proxy;
    }
}

// 使用
PaymentServiceImpl service = CglibLoggingProxy.create(PaymentServiceImpl.class);
service.pay("ORD-001", 99.9);
```

**背景：** 你正在开发一个微服务网关，需要对所有标注了 `@RateLimit` 注解的方法实现请求频率限制（每分钟最多调用 N 次），超过限制时抛出 `RateLimitException`。

**（1）** 定义 `@RateLimit` 注解，包含属性 `int limit()` 表示每分钟最大调用次数，`String key()` 用于标识限流资源。

```java
@Retention(RetentionPolicy.RUNTIME)
@Target(ElementType.METHOD)
public @interface RateLimit {
    int limit();          // 每分钟最大调用次数
    String key();         // 限流资源标识
}
```

**（2）** 使用 Spring AOP（AspectJ 注解风格）实现 `RateLimitAspect`，要求：

  - 切入点匹配所有带 `@RateLimit` 注解的方法；  
  - 使用 `ConcurrentHashMap<String, AtomicInteger>` 模拟计数器（无需实现计时器重置逻辑，但需在注释中说明生产环境应如何处理）；  
  - 在方法执行前检查并递增计数，超限则抛出 `RateLimitException`。  

```java
@Aspect
@Component
public class RateLimitAspect {

    // key -> 当前计数器
    // 生产环境应使用带 TTL 的结构，如 Redis（INCR + EXPIRE）或
    // Guava Cache（expireAfterWrite(1, TimeUnit.MINUTES)），
    // 并考虑分布式场景下的原子性，此处仅用本地内存模拟。
    private final ConcurrentHashMap<String, AtomicInteger> counters =
            new ConcurrentHashMap<>();

    // 切入点：匹配所有标注了 @RateLimit 的方法
    @Pointcut("@annotation(rateLimit)")
    public void rateLimitPointcut(RateLimit rateLimit) {}

    @Before("rateLimitPointcut(rateLimit)")
    public void checkRateLimit(JoinPoint joinPoint, RateLimit rateLimit) {
        String key = rateLimit.key();
        int limit  = rateLimit.limit();

        // 获取或初始化计数器
        AtomicInteger counter = counters.computeIfAbsent(key, k -> new AtomicInteger(0));

        // 先递增再判断，保证原子性
        int currentCount = counter.incrementAndGet();
        if (currentCount > limit) {
            throw new RateLimitException(
                String.format("资源 [%s] 已超出频率限制：%d 次/分钟", key, limit)
            );
        }
    }
}


public class RateLimitException extends RuntimeException {
    public RateLimitException(String message) {
        super(message);
    }
}
```

**（3）** 简述若将该限流逻辑改为使用**纯动态代理**（JDK Proxy）实现，与 Spring AOP 方式相比在**灵活性、耦合度、配置复杂度**三个维度上各有何优劣？

| 维度        | Spring AOP                         | 纯 JDK Proxy                                           |
| --------- | ---------------------------------- | ----------------------------------------------------- |
| **灵活性**   | 高：切入点表达式可匹配包、类、方法、注解等多种维度，无需改动被代理类 | 低：需为每个接口手动创建代理，切入逻辑与接口强绑定                             |

### 5.2 函数组合与柯里化

**（1）** 使用 `Function.compose()` 与 `Function.andThen()` 构建一个字符串处理管道，依次执行：去除首尾空格 → 转换为大写 → 在末尾追加感叹号 → 计算字符串长度。请写出代码，并解释 `compose` 与 `andThen` 的执行顺序差异。（4 分）


```java
Function<String, String>  trim      = String::trim;
Function<String, String>  toUpper   = String::toUpperCase;
Function<String, String>  addBang   = s -> s + "!";
Function<String, Integer> getLength = String::length;

// andThen：从左到右顺序执行（f.andThen(g) = g(f(x))）
Function<String, Integer> pipeline =
    trim.andThen(toUpper)
        .andThen(addBang)
        .andThen(getLength);

System.out.println(pipeline.apply("  hello world  ")); // 输出：13

// compose 与 andThen 的区别：
// f.compose(g) 等价于 f(g(x))，即先执行 g 再执行 f，顺序与 andThen 相反
// f.andThen(g) 等价于 g(f(x))，即先执行 f 再执行 g
// 示例：
Function<String, String> composed =
    addBang.compose(toUpper); // 等价于 addBang(toUpper(x))，先大写再加感叹号
```

**`compose` 对比 `andThen` 总结：**

- `f.andThen(g)`：先 f 后 g，数据从左向右流动，阅读顺序与执行顺序一致，更直观。
- `f.compose(g)`：先 g 后 f，数学上等同于 f∘g，适合习惯数学符号的场景。

**（2）** 若 `formatAddress()` 可能抛出受检异常 `AddressFormatException`，Lambda 内部无法直接抛出受检异常。请设计一个通用的 `CheckedFunction<T, R>` 函数式接口及其 `wrap()` 工具方法，将受检异常包装为运行时异常，使其可以在 Stream/Optional 链中使用。写出接口定义与 `wrap()` 实现，并演示如何在上题场景中使用。（4 分）

```java
// 1. 定义支持受检异常的函数式接口
@FunctionalInterface
public interface CheckedFunction<T, R> {
    R apply(T t) throws Exception;

    // 工具方法：将 CheckedFunction 包装为标准 Function（受检异常转为运行时异常）
    static <T, R> Function<T, R> wrap(CheckedFunction<T, R> f) {
        return t -> {
            try {
                return f.apply(t);
            } catch (RuntimeException e) {
                throw e;                                   // 运行时异常直接抛出
            } catch (Exception e) {
                throw new RuntimeException(e);             // 受检异常包装后抛出
            }
        };
    }
}
```

使用演示
```java

    .flatMap(this::getShippingAddress)
    .map(CheckedFunction.wrap(this::formatAddress))  // 用 wrap 包装受检异常
    .orElse("地址未填写");

// 若 formatAddress 内部抛出 AddressFormatException，
// 它会被包装成 RuntimeException 向上传播，
// 调用方可按需捕获 RuntimeException 或其 cause
```
